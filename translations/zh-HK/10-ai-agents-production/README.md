# AI Agents in Production: 觀察性與評估

[![AI Agents in Production](../../../translated_images/zh-HK/lesson-10-thumbnail.2b79a30773db093e.webp)](https://youtu.be/l4TP6IyJxmQ?si=reGOyeqjxFevyDq9)

隨著 AI 代理從實驗原型發展到現實應用，理解其行為、監控其效能並系統性地評估其輸出變得更為重要。

## 學習目標

完成本課程後，您將了解如何／理解：
- 代理可觀察性與評估的核心概念
- 改善代理效能、成本與有效性的技術
- 系統性地評估您的 AI 代理的內容與方式
- 如何在部署 AI 代理至生產環境時控制成本
- 如何為使用 Microsoft Agent Framework 構建的代理編寫儀器

目標是讓您具備將「黑盒子」代理轉化為透明、可管理且可靠系統的知識。

_**注意：** 部署安全且值得信賴的 AI 代理非常重要。請參閱[構建可信 AI 代理](./06-building-trustworthy-agents/README.md)課程。_

## 疑蹤與跨度

觀察性工具如 [Langfuse](https://langfuse.com/) 或 [Microsoft Foundry](https://learn.microsoft.com/en-us/azure/ai-foundry/what-is-azure-ai-foundry) 通常會將代理的執行表現表示為疑蹤（traces）與跨度（spans）。

- **疑蹤**表示從開始到結束的一個完整代理任務（例如處理用戶查詢）。
- **跨度**是在疑蹤中的個別步驟（例如調用語言模型或檢索資料）。

![Trace tree in Langfuse](https://langfuse.com/images/cookbook/example-autogen-evaluation/trace-tree.png)
<!-- Image URL retained for illustration purposes -->

如果沒有可觀察性，AI 代理會像一個「黑盒子」—其內部狀態和推理不透明，令診斷問題和優化效能變得困難。有了可觀察性，代理變成「玻璃盒子」，提供透明度，這對建立信任和確保代理按預期運行至關重要。

## 為何在生產環境中可觀察性如此重要

將 AI 代理過渡到生產環境會帶來一組新的挑戰和需求。可觀察性不再是「錦上添花」，而是關鍵能力：

*   **除錯與根因分析**：當代理失敗或產生意料之外的輸出時，可觀察性工具提供所需的疑蹤以鎖定錯誤源頭。這在涉及多次大型語言模型呼叫、工具互動與條件邏輯的複雜代理中尤為重要。
*   **延遲與成本管理**：AI 代理通常依賴按標記數或呼叫次數計費的語言模型和其他外部 API。可觀察性允許精準追踪這些呼叫，有助於辨識操作過慢或過貴的環節。團隊因此可優化提示、選擇更高效的模型或重新設計工作流程，以管控運營成本並確保良好的用戶體驗。
*   **信任、安全與合規**：許多應用中，確保代理安全且道德行事至關重要。可觀察性提供代理操作與決策的審計軌跡。這可用於檢測和緩解如提示注入、有害內容生成或個人識別資訊（PII）處理錯誤等問題。例如，您可查看疑蹤來了解代理為何給出某個回應或使用特定工具。
*   **持續改進循環**：可觀察性數據是迭代開發流程的基石。監控代理在實際環境中的表現，可協助團隊定位改進方向、收集微調模型的資料，並驗證改動影響。這形成一個反饋循環，線上評估的生產見解反哺離線實驗與精進，逐步提升代理表現。

## 需要追蹤的主要指標

為了監測並理解代理行為，應追蹤多種指標和訊號。雖然具體指標會依代理用途不同而異，但部分指標具有普遍的重要性。

以下是可觀察性工具多監控的一些常見指標：

**延遲：** 代理回應速度如何？長時間等待會負面影響用戶體驗。您應追蹤任務與各步驟延遲，透過疑蹤跟蹤代理運行。例如，所有模型呼叫耗時 20 秒的代理，可嘗試使用更快模型或平行執行呼叫以加速。

**成本：** 每次代理運行花費多少？AI 代理依賴按標記數計費的模型呼叫或外部 API。頻繁使用工具或多次提示會迅速提升成本。例如，若代理為小幅提升質量而呼叫語言模型五次，必須評估成本是否合理，或是否能減少呼叫次數或使用較便宜模型。即時監控還可識別意外飆升（如 bug 導致過度 API 呼叫迴圈）。

**請求錯誤：** 代理失敗的請求數量？這可能包括 API 錯誤或工具調用失敗。為提高代理在生產中的穩健性，您可設定備援或重試機制。例如，當語言模型供應商 A 故障時，可切換到供應商 B 作為備用。

**用戶反饋：** 實施直接用戶評價提供有價值洞見。包括明確評分（👍讚/👎倒、⭐1-5 星）或文字評論。連續負面反饋表示代理未達預期，應引起警覺。

**隱性用戶反饋：** 即使無明確評分，用戶行為也能提供間接反饋。例如立即重新表述問題、反覆提問或點擊重試按鈕。若發現用戶重複詢問同一問題，表示代理可能未達預期。

**準確率：** 代理產出正確或理想結果的頻率？準確率定義因場景而異（如問題解答正確、資訊擷取準確、用戶滿意度）。第一步是定義代理的成功標準。可透過自動檢查、評分或任務完成標籤追蹤準確率。例如，標記疑蹤為「成功」或「失敗」。

**自動評估指標：** 您也可設定自動評估。例如，使用大型語言模型評分代理輸出是否有幫助、準確等。另有多個開源庫可協助評分代理的不同面向，如用於 RAG 代理的 [RAGAS](https://docs.ragas.io/) 或檢測有害語言或提示注入的 [LLM Guard](https://llm-guard.com/)。

實務中，結合這些指標能最全面反映 AI 代理的健康狀況。本章[示例筆記本](./code_samples/10-expense_claim-demo.ipynb)將展示這些指標在真實範例中的呈現，首先我們將學習典型評估工作流程。

## 為您的代理編寫儀器

要收集追蹤資料，您需要為程式加上儀器。目標是對代理程式碼進行編輯，發出可被觀察性平台捕獲、處理與視覺化的疑蹤與指標。

**OpenTelemetry (OTel)：** [OpenTelemetry](https://opentelemetry.io/) 已成為大型語言模型監控的行業標準。它提供一套 API、SDK 和工具，用於生成、收集和導出遙測資料。

許多儀器庫包裝現有代理框架，方便將 OpenTelemetry 跨度導出至觀察性工具。Microsoft Agent Framework 原生整合 OpenTelemetry。以下是為 MAF 代理編寫儀器的範例：

```python
from agent_framework.observability import get_tracer, get_meter

tracer = get_tracer()
meter = get_meter()

with tracer.start_as_current_span("agent_run"):
    # 代理執行會自動被追蹤
    pass
```

本章的[示例筆記本](./code_samples/10-expense_claim-demo.ipynb)將演示如何為您的 MAF 代理編寫儀器。

**手動創建跨度：** 儘管儀器庫提供良好基線，常有更詳細或自訂資訊需求。您可手動創建跨度，加入自訂應用邏輯。更重要的是，可為自動或手動創建的跨度添加自訂屬性（又稱標籤或元資料）。這些屬性可包含業務相關數據、中間計算值、或有助於除錯和分析的上下文資訊，如 `user_id`、`session_id` 或 `model_version`。

使用 [Langfuse Python SDK](https://langfuse.com/docs/sdk/python/sdk-v3) 手動創建疑蹤與跨度範例：

```python
from langfuse import get_client
 
langfuse = get_client()
 
span = langfuse.start_span(name="my-span")
 
span.end()
```

## 代理評估

可觀察性提供我們指標，評估則是分析這些資料（並執行測試）以判定 AI 代理表現如何及如何改進的過程。換句話說，獲得疑蹤和指標後，如何利用它們來判斷代理並作決策？

定期評估重要，因為 AI 代理通常是非確定性的並會進化（透過更新或模型行為漂移）—沒有評估，您無法知曉「智能代理」是否真正執行得宜或是否退步。

AI 代理的評估分為兩類：**線上評估** 與 **離線評估**。兩者均有價值，且互為補充。我們通常從離線評估開始，這是部署代理前的最低必要步驟。

### 離線評估

![Dataset items in Langfuse](https://langfuse.com/images/cookbook/example-autogen-evaluation/example-dataset.png)

這涉及在受控環境中評估代理，通常使用測試數據集，而非實時用戶查詢。您使用精心策劃的資料集，知道預期輸出或正確行為，並執行代理。

例如，若您建構一個數學應用題代理，可能有一個擁有 100 道題目及標準答案的[測試數據集](https://huggingface.co/datasets/gsm8k)。離線評估常於開發期間執行（可成為 CI/CD 流程一部分），用以檢查改進或防止回退。好處是**可重複且因有標準答案能得出明確準確率指標**。您也可模擬用戶查詢，並將代理回答與理想答案比對，或使用先前提及的自動指標。

離線評估的主要挑戰是確保測試數據集的全面性與持續相關性—代理或許在固定測試集中表現良好，但生產環境查詢可能非常不同。故應保持測試集更新，包含新邊緣案例和反映真實場景的示例。混合小型「煙霧測試」和大型評估集頗為有用：小集用於快速檢查，大集用於評估整體表現。

### 線上評估

![Observability metrics overview](https://langfuse.com/images/cookbook/example-autogen-evaluation/dashboard.png)

指在實際生產環境中、即時評估代理表現，也就是實際使用期間的監控。線上評估涵蓋監視代理在真實用戶互動的表現並持續分析結果。

例如，您可能追蹤成功率、用戶滿意度分數或其他實時流量指標。線上評估的優點是**捕捉到實驗室環境可能無法預見的因素**—您能觀察模型漂移（如輸入模式變化使代理效能下降），並發現未納入測試數據的意外查詢或情況。它提供代理在現場運行的真實圖像。

線上評估通常包含收集隱性與顯性用戶反饋（前述），並可能執行影子測試或 A/B 測試（新版代理平行運行以與舊版比較）。挑戰在於現場交互很難取得可靠標籤或分數，可能需依賴用戶反饋或下游指標（如用戶是否點擊結果）。

### 結合兩者

線上與離線評估非互斥，而是高度互補。線上監控得到的洞見（例如代理在某類用戶查詢中表現不佳）可用以豐富和改善離線測試數據集。反過來，在離線測試中表現良好的代理，更能自信地部署與線上監控。

事實上，多數團隊採用以下循環：

_離線評估 -> 部署 -> 線上監控 -> 收集新失敗案例 -> 加入離線數據集 -> 精煉代理 -> 重複_。

## 常見問題

在將 AI 代理部署到生產時，您可能會遇到各種挑戰。以下列出一些常見問題及可能的解決方案：

| **問題**    | **可能解決方案**   |
| ------------- | ------------------ |
| AI 代理無法持續執行任務 | - 精煉對 AI 代理的提示，明確目標。<br>- 辨識是否可將任務拆分成子任務，由多個代理分別處理。 |
| AI 代理陷入無限迴圈  | - 確保設有明確的終止條件，讓代理知道何時停止。<br>- 針對需推理與規劃的複雜任務，使用更適合推理的大型模型。 |
| AI 代理工具調用表現不佳   | - 在代理系統外測試並驗證工具輸出。<br>- 精煉工具的參數、提示和名稱定義。  |
| 多代理系統表現不一致 | - 精煉給每個代理的提示，確保彼此具體且不同。<br>- 建立「路由」或控制代理的層級系統，以決定正確代理。 |

許多這些問題可透過導入可觀察性更有效識別。我們前述的疑蹤與指標有助精確定位問題出現的代理工作流程階段，大幅提升除錯和優化效率。

## 成本管理
以下是管理將 AI 代理部署到生產環境成本的一些策略：

**使用較小的模型：** 小型語言模型（SLMs）在某些具代理性的用例中表現良好，且能顯著降低成本。如前所述，建立一個評估系統來判定並比較與較大型模型的效能，是了解 SLM 在您的用例中表現如何的最佳方式。考慮在較簡單的任務如意圖分類或參數提取使用 SLM，而將較大型模型保留給複雜推理使用。

**使用路由模型：** 類似的策略是使用多樣的模型和大小。您可以使用 LLM/SLM 或無伺服器函式根據複雜度來路由請求至最合適的模型。這也有助於降低成本，同時確保在正確任務上的效能。例如，將簡單查詢路由至較小且較快的模型，只有在複雜推理任務時才使用昂貴的大型模型。

**快取回應：** 識別常見請求和任務，並在它們經過您的代理系統之前提供回應，是減少類似請求數量的好方法。您甚至可以實作一個流程，使用較基礎的 AI 模型來判斷請求與快取請求的相似程度。此策略可大幅降低常見問題或常用工作流程的成本。
