# नमुना पावत्या फिक्स्चर

नोटबुक चालवण्याशिवाय तपासणीसाठी तीन पूर्व-निर्मित पावती फाइल्स.

| फाइल | काय आहे |
|---|---|
| `01_valid_receipt.json` | `lookup_flights` साधन कॉलसाठी वैध सही केलेली पावती. प्रमाणीकरण True परत करते. |
| `02_tampered_receipt.json` | सही करण्यानंतर एका फील्डमध्ये बदल केल्या गेलेल्या त्याच पावती. प्रमाणीकरण False परत करते. |
| `03_chain_three_receipts.json` | तीन वैध पावत्यांची साखळी (search, hold, book) जिथे `previous_receipt_hash` प्रत्येकाला मागील पावतीशी लिंक करते. |

## नमुने तपासणे

नोटबुक प्रमाणीकरण चार विभागांमध्ये समजावून सांगते. या फिक्स्चरना
नोटबुक कथा न वाचता थेट सत्यापित करण्यासाठी:

```python
import json
from pathlib import Path

# समजले आहे की आपण आयात आणि सहाय्यक फंक्शन्स पूर्ण केले आहेत
# 18-signed-receipts.ipynb च्या 1 आणि 2 विभागांतून.

valid = json.loads(Path("01_valid_receipt.json").read_text())
print(f"Valid receipt: {verify_receipt(valid)}")        # खरे

tampered = json.loads(Path("02_tampered_receipt.json").read_text())
print(f"Tampered receipt: {verify_receipt(tampered)}")  # खोटे

chain = json.loads(Path("03_chain_three_receipts.json").read_text())
for r in verify_chain(chain):
    print(f"  Receipt {r['index']} ({r['tool']}): {'VALID' if r['overall_valid'] else 'INVALID'}")
```

## विद्यार्थी कच्च्या JSON तपासून काय शिकतात

कच्च्या पावती फॉरमॅटचा अभ्यास केल्याने नोटबुकमधील सेल्स कधी कधी देत नाहीत ते अंतर्ज्ञाना तयार होते.
JSON झिरपून पाहणारे विद्यार्थी अनेकदा लक्ष देतात:

1. सही ही अस्पष्ट base64url स्ट्रिंग आहे, पण इतर सर्व फील्ड सामान्य
   वाचण्यायोग्य JSON आहेत. सही सामग्रीला संकुचित करत नाही; ते त्याचे प्रमाणपत्र आहे.
2. `public_key` पावतीमध्ये समाविष्ट आहे. एक परीक्षकास सत्यापित करण्यासाठी काहीही
   वेगळे लागत नाही (फक्त की वास्तविकपणे दावा केलेल्या जारी करणाऱ्याची असल्याचा विश्वास ठेवणे आवश्यक; ओळख infra-ढाचाबाबत धडा README पहा).
3. कोणत्याही फील्डमधील एका अक्षराचा बदल करून, नंतर ही फाइल `02_tampered_receipt.json` शी पुन्हा तुलना केल्यास, बाइट-स्तरीय यंत्रणा ठसठशीत होते.
