# AI Agents in Production: Observability & Evaluation

[![AI Agents in Production](../../../translated_images/th/lesson-10-thumbnail.2b79a30773db093e.webp)](https://youtu.be/l4TP6IyJxmQ?si=reGOyeqjxFevyDq9)

เมื่อ AI agents ก้าวจากต้นแบบทดลองไปสู่แอปพลิเคชันในโลกจริง ความสามารถในการเข้าใจพฤติกรรมของพวกมัน ติดตามประสิทธิภาพ และประเมินผลลัพธ์อย่างเป็นระบบจึงมีความสำคัญ

## Learning Goals

หลังจากเรียนบทเรียนนีเสร็จ คุณจะรู้วิธี/เข้าใจ:
- แนวคิดหลักของการสังเกตการณ์และการประเมินผลของ agents
- เทคนิคเพื่อปรับปรุงประสิทธิภาพ ต้นทุน และประสิทธิผลของ agents
- สิ่งที่ควรประเมินและวิธีประเมิน AI agents ของคุณอย่างเป็นระบบ
- วิธีควบคุมต้นทุนเมื่อปรับใช้ AI agents สู่ระบบจริง
- วิธีติดตั้งเครื่องมือกับ agents ที่สร้างด้วย AutoGen

เป้าหมายคือการมอบความรู้ให้คุณสามารถเปลี่ยน agents ที่เป็น "กล่องดำ" ให้กลายเป็นระบบโปร่งใส จัดการได้ และเชื่อถือได้

_**หมายเหตุ:** การปรับใช้ AI Agents ที่ปลอดภัยและน่าเชื่อถือมีความสำคัญ ตรวจสอบบทเรียน [Building Trustworthy AI Agents](../06-building-trustworthy-agents/README.md) ด้วย_

## Traces and Spans

เครื่องมือสำหรับการสังเกตการณ์ เช่น [Langfuse](https://langfuse.com/) หรือ [Microsoft Foundry](https://learn.microsoft.com/en-us/azure/ai-foundry/what-is-azure-ai-foundry) ปกติจะแสดงการทำงานของ agent ในรูปแบบ traces และ spans

- **Trace** แทนงานของ agent ทั้งหมดตั้งแต่ต้นจนจบ (เช่น การจัดการคำถามของผู้ใช้)
- **Spans** คือขั้นตอนย่อยๆ ภายใน trace (เช่น การเรียกใช้โมเดลภาษา หรือการดึงข้อมูล)

![Trace tree in Langfuse](https://langfuse.com/images/cookbook/example-autogen-evaluation/trace-tree.png)

หากไม่มีการสังเกตการณ์ AI agent อาจรู้สึกเหมือนเป็น "กล่องดำ" — สถานะและตรรกะภายในไม่ชัดเจน ทำให้ยากต่อการวินิจฉัยปัญหาหรือเพิ่มประสิทธิภาพ เมื่อมีการสังเกตการณ์ agents จะกลายเป็น "กล่องแก้ว" มอบความโปร่งใสที่สำคัญสำหรับการสร้างความไว้วางใจและมั่นใจว่าพวกมันทำงานตามที่ตั้งใจไว้

## Why Observability Matters in Production Environments

การย้าย AI agents ไปสู่สภาพแวดล้อมผลิตจริงนำมาซึ่งความท้าทายและข้อกำหนดใหม่ๆ การสังเกตการณ์จึงกลายเป็นความสามารถที่จำเป็น ไม่ใช่แค่สิ่งเสริม:

*   **การดีบั๊กและวิเคราะห์สาเหตุหลัก:** เมื่อ agent ล้มเหลวหรือสร้างผลลัพธ์ที่ไม่คาดคิด เครื่องมือสังเกตการณ์จะให้ traces ที่จำเป็นเพื่อตรวจหาต้นตอของข้อผิดพลาด สิ่งนี้สำคัญโดยเฉพาะใน agents ที่ซับซ้อน อาจมีการเรียกใช้ LLM หลายครั้ง การโต้ตอบกับเครื่องมือ และตรรกะตามเงื่อนไข
*   **การจัดการความหน่วงเวลาและต้นทุน:** AI agents มักพึ่งพา LLM และ API ภายนอกที่คิดค่าบริการตามจำนวน token หรือจำนวนครั้งที่เรียกใช้ การสังเกตการณ์ช่วยติดตามการเรียกใช้เหล่านี้อย่างแม่นยำ ช่วยระบุขั้นตอนที่ช้าเกินหรือต้นทุนสูงเกินไป ทำให้ทีมสามารถปรับจูน prompt เลือกโมเดลที่มีประสิทธิภาพมากขึ้น หรือออกแบบเวิร์กโฟลว์ใหม่เพื่อควบคุมต้นทุนและรักษาประสบการณ์ผู้ใช้
*   **ความไว้วางใจ ความปลอดภัย และการปฏิบัติตามข้อกำหนด:** ในหลายแอปพลิเคชัน จำเป็นต้องมั่นใจว่า agents ทำงานอย่างปลอดภัยและมีจริยธรรม การสังเกตการณ์ให้แถบตรวจสอบการกระทำและการตัดสินใจของ agent ซึ่งช่วยตรวจจับและบรรเทาปัญหา เช่น การโจมตีด้วย prompt injection การสร้างเนื้อหาที่เป็นอันตราย หรือการจัดการข้อมูลส่วนบุคคล (PII) อย่างไม่เหมาะสม ตัวอย่างเช่น คุณสามารถตรวจสอบ traces เพื่อเข้าใจว่าทำไม agent ถึงตอบคำถามบางอย่างหรือใช้เครื่องมือเฉพาะ
*   **วงจรการปรับปรุงอย่างต่อเนื่อง:** ข้อมูลการสังเกตการณ์เป็นพื้นฐานของกระบวนการพัฒนาซ้ำ โดยติดตามการทำงานของ agents ในโลกจริง ทีมงานสามารถระบุจุดที่ต้องปรับปรุง เก็บข้อมูลสำหรับการปรับแต่งโมเดล และตรวจสอบผลกระทบจากการเปลี่ยนแปลง สร้างวงจรป้อนกลับที่ข้อมูลการประเมินออนไลน์ช่วยปรับปรุงการทดลองออฟไลน์และการปรับปรุง นำไปสู่ประสิทธิภาพของ agent ที่ดีขึ้นเรื่อยๆ

## Key Metrics to Track

เพื่อติดตามและเข้าใจพฤติกรรมของ agent ควรเก็บตัวชี้วัดและสัญญาณหลากหลายประเภท แม้จะมีตัวชี้วัดเฉพาะขึ้นกับจุดประสงค์ของ agent บ้าง แต่บางตัวชี้วัดก็สำคัญในทุกกรณี

นี่คือตัวชี้วัดที่เครื่องมือสังเกตการณ์มักติดตาม:

**Latency:** agent ตอบสนองรวดเร็วแค่ไหน? เวลารอนานส่งผลลบต่อประสบการณ์ผู้ใช้ ควรวัดความหน่วงสำหรับงานและขั้นตอนย่อยโดยการติดตามการทำงานของ agent เช่น agent ที่ใช้เวลาทั้งหมด 20 วินาทีสำหรับการเรียกใช้โมเดลทั้งหมด อาจเร่งความเร็วได้โดยใช้โมเดลที่เร็วกว่า หรือรันการเรียกโมเดลแบบขนาน

**Costs:** ค่าใช้จ่ายต่อการทำงานของ agent เท่าไร? AI agents พึ่งพาการเรียกใช้ LLM ที่คิดค่าบริการตาม token หรือ API ภายนอก การใช้เครื่องมือบ่อยหรือ prompt หลายรอบจะเพิ่มต้นทุนอย่างรวดเร็ว เช่น หาก agent เรียก LLM ห้าครั้งเพื่อปรับปรุงคุณภาพเล็กน้อย คุณต้องประเมินว่าต้นทุนเหมาะสมหรือไม่ หรือคุ้มค่าที่จะลดจำนวนครั้งเรียกใช้ หรือใช้โมเดลที่ถูกกว่า การติดตามแบบเรียลไทม์ยังช่วยตรวจจับจุดที่เกิดการเพิ่มขึ้นไม่คาดคิด (เช่น บั๊กที่ทำให้เกิดลูป API ซ้ำซ้อน)

**Request Errors:** agent ล้มเหลวกี่คำขอ? รวมถึงข้อผิดพลาด API หรือการเรียกเครื่องมือที่ล้มเหลว เพื่อให้ agent แข็งแรงขึ้นในระบบจริง คุณสามารถตั้ง fallback หรือ retry เช่น หากผู้ให้บริการ LLM A ล่ม คุณสลับไปใช้ผู้ให้บริการ LLM B เป็นแบ็คอัพ

**User Feedback:** การประเมินจากผู้ใช้อย่างตรงไปตรงมามอบข้อมูลมีค่า อาจเป็นการให้คะแนนชัดเจน (👍นิ้วโป้งขึ้น/👎ลง, ⭐1-5 ดาว) หรือความคิดเห็นเป็นข้อความ หากได้รับคำติชมเชิงลบอย่างสม่ำเสมอ ควรแจ้งเตือนว่านี่เป็นสัญญาณว่า agent ทำงานไม่ตรงตามคาดหวัง

**Implicit User Feedback:** พฤติกรรมผู้ใช้ให้ข้อมูลย้อนกลับโดยตรงแม้ไม่มีการให้คะแนนชัดเจน เช่น การตั้งคำถามใหม่ทันที การถามซ้ำ หรือการคลิกปุ่มลองใหม่ เช่น หากสังเกตเห็นว่าผู้ใช้ถามคำถามเดิมซ้ำ นั่นเป็นสัญญาณว่า agent ทำงานไม่ดีตามที่คาดหวัง

**Accuracy:** agent ให้ผลลัพธ์ที่ถูกต้องหรือเป็นที่ต้องการบ่อยแค่ไหน? นิยามของความแม่นยำแตกต่างกัน (เช่น ความถูกต้องในการแก้ปัญหา ความแม่นยำในการค้นคืนข้อมูล หรือความพึงพอใจของผู้ใช้) ขั้นตอนแรกคือกำหนดว่าอะไรคือความสำเร็จสำหรับ agent ของคุณ คุณสามารถติดตามความแม่นยำผ่านการตรวจสอบอัตโนมัติ คะแนนประเมิน หรือป้ายบอกความสำเร็จของงาน เช่น การทำเครื่องหมาย traces ว่า "สำเร็จ" หรือ "ล้มเหลว"

**Automated Evaluation Metrics:** คุณยังสามารถตั้งการประเมินผลอัตโนมัติ เช่น ใช้ LLM ให้คะแนนผลลัพธ์ของ agent ว่ามีประโยชน์ ถูกต้อง หรือไม่ หรือใช้ไลบรารีโอเพนซอร์สหลายตัวช่วยให้คะแนนด้านต่างๆ ของ agent เช่น [RAGAS](https://docs.ragas.io/) สำหรับ RAG agents หรือ [LLM Guard](https://llm-guard.com/) เพื่อตรวจจับภาษาที่เป็นอันตรายหรือ prompt injection

ในทางปฏิบัติ การผสมผสานตัวชี้วัดเหล่านี้ช่วยให้ครอบคลุมสุขภาพของ AI agent ได้ดีที่สุด ในบทนี้ [ตัวอย่างโน้ตบุ๊ก](./code_samples/10_autogen_evaluation.ipynb) จะแสดงให้เห็นว่าตัวชี้วัดเหล่านี้มีหน้าตาอย่างไรในตัวอย่างจริง แต่ก่อนอื่นเราจะเรียนรู้รูปแบบการประเมินทั่วไป

## Instrument your Agent

เพื่อเก็บข้อมูล tracing คุณต้องติดตั้งเครื่องมือกับโค้ด เป้าหมายคือการติดตั้งโค้ด agent เพื่อส่งออก traces และ metrics ที่สามารถเก็บ รวบรวม และแสดงผลโดยแพลตฟอร์มสังเกตการณ์

**OpenTelemetry (OTel):** [OpenTelemetry](https://opentelemetry.io/) กลายเป็นมาตรฐานอุตสาหกรรมสำหรับการสังเกตการณ์ LLM มีชุด API, SDK และเครื่องมือสำหรับสร้าง เก็บรวบรวม และส่งออกข้อมูล telemetry

มีไลบรารีการติดตั้งมากมายที่ห่อหุ้มเฟรมเวิร์ก agent ที่มีอยู่และทำให้ง่ายต่อการส่ง export OpenTelemetry spans ไปยังเครื่องมือสังเกตการณ์ ด้านล่างเป็นตัวอย่างการติดตั้ง AutoGen agent ด้วย [ไลบรารี OpenLit instrumentation](https://github.com/openlit/openlit):

```python
import openlit

openlit.init(tracer = langfuse._otel_tracer, disable_batch = True)
```
  
[ตัวอย่างโน้ตบุ๊ก](./code_samples/10_autogen_evaluation.ipynb) ในบทนี้จะแสดงวิธีการติดตั้งเครื่องมือให้กับ AutoGen agent ของคุณ

**Manual Span Creation:** แม้ไลบรารีติดตั้งจะมีฐานที่ดี แต่บางครั้งคุณอาจต้องการข้อมูลละเอียดหรือกำหนดเอง คุณสามารถสร้าง spans ด้วยตนเองเพื่อเพิ่มตรรกะแอปพลิเคชันเพิ่มเติม ที่สำคัญคือคุณสามารถเสริม spans ที่สร้างโดยอัตโนมัติหรือด้วยมือด้วยแอตทริบิวต์เฉพาะ (เรียกอีกอย่างว่าป้ายหรือเมตาดาต้า) ซึ่งรวมถึงข้อมูลเฉพาะธุรกิจ คำนวณกลางทาง หรือบริบทอื่นที่ช่วยดีบั๊กหรือวิเคราะห์ เช่น `user_id` , `session_id` , หรือ `model_version`

ตัวอย่างการสร้าง traces และ spans ด้วยตนเองด้วย [Langfuse Python SDK](https://langfuse.com/docs/sdk/python/sdk-v3): 

```python
from langfuse import get_client
 
langfuse = get_client()
 
span = langfuse.start_span(name="my-span")
 
span.end()
```

## Agent Evaluation

การสังเกตการณ์ให้ตัวชี้วัด แต่การประเมินคือกระบวนการวิเคราะห์ข้อมูลเหล่านั้น (และการทดสอบ) เพื่อตัดสินว่า AI agent ทำงานได้ดีแค่ไหน และจะปรับปรุงอย่างไร กล่าวคือ เมื่อคุณมี traces และ metrics เหล่านั้นแล้ว จะใช้ตัดสิน agent และตัดสินใจอย่างไร?

การประเมินอย่างสม่ำเสมอสำคัญเพราะ AI agents มักไม่เฉพาะเจาะจงแน่นอน และอาจเปลี่ยนแปลงได้ (ผ่านการอัปเดตหรือพฤติกรรมโมเดลที่เปลี่ยนไป) – หากไม่ประเมินคุณก็ไม่รู้ว่า "agent ฉลาด" ของคุณทำงานดีจริงหรือถดถอย

การประเมินมีสองประเภท: **online evaluation** และ **offline evaluation** ทั้งสองมีคุณค่าและเสริมกัน โดยปกติเราเริ่มจาก offline evaluation เนื่องจากเป็นขั้นตอนพื้นฐานก่อนปรับใช้ agent ใดๆ

### Offline Evaluation

![Dataset items in Langfuse](https://langfuse.com/images/cookbook/example-autogen-evaluation/example-dataset.png)

เป็นการประเมิน agent ในสภาพแวดล้อมที่ควบคุมได้ โดยทั่วไปใช้ชุดข้อมูลทดสอบ ไม่ใช่คำถามจากผู้ใช้จริง คุณใช้ชุดข้อมูลที่คัดสรรซึ่งรู้ว่าผลลัพธ์ที่คาดหวังคืออะไร หรือพฤติกรรมที่ถูกต้องคืออะไร จากนั้นรัน agent กับข้อมูลชุดนั้น

เช่น หากคุณสร้าง agent แก้โจทย์ปัญหาคณิตศาสตร์ คุณอาจมี [ชุดข้อมูลทดสอบ](https://huggingface.co/datasets/gsm8k) รวม 100 ปัญหาที่ทราบคำตอบ การประเมินออฟไลน์มักทำในระหว่างพัฒนา (และอาจเป็นส่วนหนึ่งของ pipeline CI/CD) เพื่อตรวจสอบการปรับปรุงหรือป้องกันการถดถอย ข้อดีคือ **สามารถทำซ้ำได้และได้รับตัวชี้วัดความแม่นยำที่ชัดเจนเพราะมีข้อมูลจริงที่เปรียบเทียบได้** คุณอาจจำลองคำถามผู้ใช้และวัดคำตอบของ agent เทียบกับคำตอบอุดมคติ หรือใช้ตัวชี้วัดอัตโนมัติเหมือนที่กล่าวไว้ข้างต้น

ความท้าทายหลักของ offline eval คือการทำให้ชุดข้อมูลทดสอบของคุณครอบคลุมและสอดคล้องกับความเปลี่ยนแปลง — agent อาจทำงานได้ดีบนชุดทดสอบเดียว แต่ต้องเจอคำถามที่แตกต่างมากในสภาพแวดล้อมจริง คุณจึงควรอัปเดตข้อมูลทดสอบอย่างสม่ำเสมอด้วยกรณีมุมและตัวอย่างใหม่ๆ ที่สะท้อนสถานการณ์จริง การผสมผสานระหว่างชุดขนาดเล็กสำหรับ "smoke test" และชุดขนาดใหญ่สำหรับวัดประสิทธิภาพทั่วไปก็มีประโยชน์: ชุดเล็กสำหรับตรวจสอบเร็วและชุดใหญ่สำหรับตัวชี้วัดภาพรวม

### Online Evaluation 

![Observability metrics overview](https://langfuse.com/images/cookbook/example-autogen-evaluation/dashboard.png)

คือการประเมิน agent ในสภาพแวดล้อมจริง ขณะใช้งานจริงในผลิตจริง การประเมินออนไลน์รวมถึงการติดตามประสิทธิภาพของ agent ในการโต้ตอบกับผู้ใช้จริงและวิเคราะห์ผลอย่างต่อเนื่อง

ตัวอย่างเช่น คุณอาจติดตามอัตราความสำเร็จ คะแนนความพึงพอใจของผู้ใช้ หรือประเมินตัวชี้วัดอื่นในทราฟฟิกจริง ข้อดีของการประเมินออนไลน์คือ **จับประเด็นที่คุณอาจไม่คาดคิดในห้องทดลอง** — คุณสามารถสังเกตการเปลี่ยนแปลงของโมเดลเมื่อเวลาผ่านไป (หากประสิทธิภาพลดลงเมื่อรูปแบบอินพุตเปลี่ยน) และจับคำถามหรือสถานการณ์ที่ไม่เคยมีในข้อมูลทดสอบที่มี​ มันแสดงภาพจริงว่าสถานะของ agent ในโลกภายนอก

การประเมินออนไลน์มักรวมถึงการเก็บฟีดแบ็กทั้งแบบชัดเจนและไม่ชัดเจนตามที่กล่าวไว้ข้างต้น และอาจมีการทดสอบเงา (shadow tests) หรือการทดสอบ A/B (ซึ่งเวอร์ชันใหม่ของ agent รันขนานเพื่อเปรียบเทียบกับเวอร์ชันเก่า) ความท้าทายคือการได้ป้ายกำกับหรือคะแนนที่เชื่อถือได้สำหรับการโต้ตอบจริง บางทีอาจต้องพึ่งพาความคิดเห็นของผู้ใช้หรือเมตริกปลายน้ำ (เช่น ผู้ใช้คลิกผลลัพธ์หรือไม่)

### Combining the two

การประเมินออนไลน์และออฟไลน์ไม่ใช่สิ่งแยกจากกัน แต่เสริมกันอย่างมาก ข้อมูลจากการติดตามออนไลน์ (เช่น รูปแบบคำถามใหม่ที่ agent ทำงานไม่ดี) สามารถนำมาขยายและปรับปรุงชุดทดสอบออฟไลน์ได้ ในทางกลับกัน agent ที่ทำงานดีในชุดทดสอบออฟไลน์จะปรับใช้และติดตามผลออนไลน์ได้มั่นใจยิ่งขึ้น

แท้จริงแล้วหลายทีมสร้างวงจร: 

_ประเมินออฟไลน์ -> ปรับใช้ -> ติดตามออนไลน์ -> รวบรวมกรณีล้มเหลวใหม่ -> เติมชุดข้อมูลออฟไลน์ -> ปรับปรุง agent -> ทำซ้ำ_

## Common Issues

เมื่อคุณปรับใช้ AI agents ในการผลิต คุณอาจพบความท้าทายต่างๆ นี่คือปัญหาทั่วไปและแนวทางแก้ไขที่อาจใช้ได้:

| **ปัญหา**    | **แนวทางแก้ไขที่เป็นไปได้**   |
| ------------- | ------------------ |
| AI Agent ทำงานไม่สม่ำเสมอ | - ปรับปรุง prompt ที่ให้กับ AI Agent ให้ชัดเจนเกี่ยวกับวัตถุประสงค์<br>- ระบุว่าการแบ่งงานออกเป็นงานย่อยและจัดการโดยหลาย agent อาจช่วยได้ |
| AI Agent ทำงานวนลูปไม่จบ  | - ตรวจสอบให้แน่ใจว่ามีเงื่อนไขการยุติที่ชัดเจนเพื่อให้ Agent รู้ว่าจะหยุดเมื่อไร<br>- สำหรับงานซับซ้อนที่ต้องใช้ตรรกะและการวางแผน ใช้โมเดลขนาดใหญ่ที่เชี่ยวชาญงานตรรกะ |
| การเรียกเครื่องมือของ AI Agent ทำงานไม่ดี   | - ทดสอบและตรวจสอบผลลัพธ์ของเครื่องมือภายนอกระบบ agent<br>- ปรับปรุงพารามิเตอร์ prompt และการตั้งชื่อเครื่องมือ |
| ระบบ Multi-Agent ทำงานไม่สม่ำเสมอ | - ปรับปรุง prompt ที่ให้แต่ละ agent ให้เฉพาะเจาะจงและแตกต่างกันอย่างชัดเจน<br>- สร้างระบบลำดับชั้นโดยใช้ "routing" หรือ agent ควบคุม เพื่อตัดสินใจว่า agent ไหนเหมาะสม |

หลายปัญหานี้จะถูกระบุได้ง่ายขึ้นเมื่อมีการสังเกตการณ์ Traces และ metrics ที่กล่าวข้างต้นช่วยระบุจุดที่เกิดปัญหาในเวิร์กโฟลว์ agent ทำให้การดีบั๊กและการเพิ่มประสิทธิภาพมีประสิทธิผลยิ่งขึ้น

## Managing Costs
นี่คือกลยุทธ์บางอย่างในการจัดการต้นทุนสำหรับการนำ AI agents ไปใช้งานจริง:

**การใช้โมเดลขนาดเล็ก:** Small Language Models (SLMs) สามารถทำงานได้ดีในกรณีการใช้งานแบบ agentic บางอย่าง และจะช่วยลดต้นทุนอย่างมาก อย่างที่ได้กล่าวไว้ก่อนหน้านี้ การสร้างระบบประเมินผลเพื่อกำหนดและเปรียบเทียบประสิทธิภาพกับโมเดลขนาดใหญ่เป็นวิธีที่ดีที่สุดในการทำความเข้าใจว่า SLM จะทำงานได้ดีเพียงใดในกรณีการใช้งานของคุณ พิจารณาใช้ SLM สำหรับงานที่ง่ายกว่าเช่น การจัดประเภทเจตนา หรือการสกัดพารามิเตอร์ ในขณะที่เก็บโมเดลขนาดใหญ่ไว้สำหรับการคิดวิเคราะห์ที่ซับซ้อน

**การใช้โมเดล Router:** กลยุทธ์ที่คล้ายกันคือการใช้ความหลากหลายของโมเดลและขนาด คุณสามารถใช้ LLM/SLM หรือฟังก์ชันแบบเซิร์ฟเวอร์เลสเพื่อเส้นทางคำขอตามความซับซ้อนไปยังโมเดลที่เหมาะสมที่สุด วิธีนี้จะช่วยลดต้นทุนในขณะที่ยังคงประสิทธิภาพสำหรับงานที่ถูกต้อง ตัวอย่างเช่น เส้นทางคำถามง่ายๆ ไปยังโมเดลขนาดเล็กและรวดเร็ว และใช้โมเดลขนาดใหญ่ที่มีราคาแพงเฉพาะสำหรับงานวิเคราะห์ที่ซับซ้อนเท่านั้น

**การเก็บแคชคำตอบ:** การระบุคำขอและงานที่พบบ่อย และจัดเตรียมคำตอบก่อนที่จะผ่านระบบ agentic ของคุณ เป็นวิธีที่ดีในการลดปริมาณของคำขอที่คล้ายกัน คุณสามารถสร้างโฟลว์ในการตรวจสอบว่าคำขอนั้นคล้ายกับคำขอที่ถูกเก็บไว้แคชมากเพียงใดโดยใช้โมเดล AI ที่พื้นฐานมากขึ้น กลยุทธ์นี้สามารถลดต้นทุนได้อย่างมากสำหรับคำถามที่ถูกถามบ่อยหรือเวิร์กโฟลว์ที่พบบ่อย
