# ตัวแทน AI ในการผลิต: การสังเกตการณ์และการประเมินผล

[![AI Agents in Production](../../../translated_images/th/lesson-10-thumbnail.2b79a30773db093e.webp)](https://youtu.be/l4TP6IyJxmQ?si=reGOyeqjxFevyDq9)

เมื่อเอเจนต์ AI ก้าวจากต้นแบบในห้องทดลองสู่การใช้งานในโลกจริง ความสามารถในการเข้าใจพฤติกรรมของพวกเขา ติดตามประสิทธิภาพ และประเมินผลลัพธ์อย่างเป็นระบบจึงกลายเป็นสิ่งสำคัญ

## เป้าหมายการเรียนรู้

หลังจากจบบทเรียนนี้ คุณจะรู้จักและเข้าใจ:
- แนวคิดหลักของการสังเกตการณ์และการประเมินผลเอเจนต์
- เทคนิคการปรับปรุงประสิทธิภาพ ต้นทุน และประสิทธิผลของเอเจนต์
- วิธีการและสิ่งที่ควรประเมินสำหรับเอเจนต์ AI ของคุณอย่างเป็นระบบ
- วิธีควบคุมต้นทุนเมื่อปรับใช้เอเจนต์ AI สู่สภาพแวดล้อมการผลิต
- วิธีติดตั้งเครื่องมือสำหรับเอเจนต์ที่สร้างด้วย Microsoft Agent Framework

เป้าหมายคือการให้ความรู้ที่ช่วยเปลี่ยนเอเจนต์ “กล่องดำ” ของคุณให้เป็นระบบที่โปร่งใส จัดการได้ และน่าเชื่อถือ

_**หมายเหตุ:** การปรับใช้เอเจนต์ AI ที่ปลอดภัยและน่าเชื่อถือเป็นสิ่งสำคัญ โปรดดูบทเรียน [การสร้างเอเจนต์ AI ที่น่าเชื่อถือ](./06-building-trustworthy-agents/README.md) ด้วย_

## แทรซและสแปน

เครื่องมือสังเกตการณ์ เช่น [Langfuse](https://langfuse.com/) หรือ [Microsoft Foundry](https://learn.microsoft.com/en-us/azure/ai-foundry/what-is-azure-ai-foundry) มักจะแสดงการทำงานของเอเจนต์ในรูปแบบของแทรซและสแปน

- **แทรซ** แสดงถึงงานของเอเจนต์ทั้งหมดตั้งแต่ต้นจนจบ (เหมือนการจัดการคำถามของผู้ใช้)
- **สแปน** คือขั้นตอนย่อยแต่ละขั้นตอนภายในแทรซ (เช่น การเรียกใช้โมเดลภาษา หรือ การดึงข้อมูล)

## ทำไมการสังเกตการณ์จึงสำคัญในสภาพแวดล้อมการผลิต

การย้ายเอเจนต์ AI สู่สภาพแวดล้อมการผลิตนำมาซึ่งความท้าทายและข้อกำหนดใหม่ ๆ การสังเกตการณ์จึงไม่ใช่แค่ "สิ่งที่ควรมี" อีกต่อไป แต่เป็นความสามารถที่จำเป็น:

*   **การดีบักและการวิเคราะห์สาเหตุหลัก**: เมื่อเอเจนต์ล้มเหลวหรือให้ผลลัพธ์ที่ไม่คาดคิด เครื่องมือสังเกตการณ์จะให้แทรซที่จำเป็นต่อการระบุสาเหตุของข้อผิดพลาด สิ่งนี้สำคัญโดยเฉพาะกับเอเจนต์ซับซ้อนที่อาจรวมการเรียก LLM หลายครั้ง การใช้เครื่องมือต่าง ๆ และตรรกะมีเงื่อนไข
*   **การจัดการความหน่วงและต้นทุน**: เอเจนต์ AI มักพึ่งพา LLM และ API ภายนอกซึ่งคิดค่าบริการต่อตัวอักษรหรือจำนวนครั้งการเรียกใช้ การสังเกตการณ์ช่วยติดตามการเรียกใช้เหล่านี้อย่างแม่นยำ ช่วยระบุการดำเนินการที่ช้าเกินไปหรือมีค่าใช้จ่ายสูง ทีมงานสามารถใช้ข้อมูลนี้เพื่อปรับแต่งคำสั่ง เลือกโมเดลที่มีประสิทธิภาพมากกว่า หรือออกแบบขั้นตอนการทำงานใหม่เพื่อลดต้นทุนและสร้างประสบการณ์ผู้ใช้ที่ดี
*   **ความไว้วางใจ ความปลอดภัย และการปฏิบัติตาม**: ในหลาย ๆ การประยุกต์ใช้งาน จำเป็นต้องรับรองว่าเอเจนต์ทำงานอย่างปลอดภัยและมีจริยธรรม การสังเกตการณ์ให้เส้นทางการตรวจสอบการกระทำและการตัดสินใจของเอเจนต์ ซึ่งช่วยตรวจจับและลดผลกระทบจากปัญหา เช่น การฉีดคำสั่ง (prompt injection) การสร้างเนื้อหาที่เป็นอันตราย หรือการจัดการข้อมูลส่วนบุคคล (PII) อย่างไม่เหมาะสม เช่น คุณสามารถตรวจสอบแทรซเพื่อเข้าใจว่าเพราะเหตุใดเอเจนต์จึงให้คำตอบแบบใดหรือใช้เครื่องมือใดเฉพาะเจาะจง
*   **วงจรปรับปรุงอย่างต่อเนื่อง**: ข้อมูลการสังเกตการณ์เป็นฐานรากของกระบวนการพัฒนาแบบวนซ้ำ โดยการติดตามความเป็นไปของเอเจนต์ในโลกจริง ทีมงานจะระบุจุดที่ต้องปรับปรุง เก็บข้อมูลสำหรับปรับแต่งโมเดล และยืนยันผลกระทบของการปรับเปลี่ยน ซึ่งสร้างวงจรป้อนกลับที่ข้อมูลเชิงลึกจากการประเมินออนไลน์นำไปสู่การทดลองและการปรับปรุงในสถานะออฟไลน์ ทำให้เอเจนต์พัฒนาและมีประสิทธิภาพดีขึ้นอย่างต่อเนื่อง

## ตัวชี้วัดที่สำคัญที่ต้องติดตาม

เพื่อมอนิเตอร์และทำความเข้าใจพฤติกรรมของเอเจนต์ ควรติดตามตัวชี้วัดและสัญญาณหลากหลาย แม้ว่าตัวชี้วัดเฉพาะอาจแตกต่างตามวัตถุประสงค์ของเอเจนต์ แต่บางตัวชี้วัดก็มีความสำคัญโดยทั่วไป

ตัวชี้วัดที่พบบ่อยที่สุดซึ่งเครื่องมือสังเกตการณ์มักติดตาม ได้แก่:

**ความหน่วง (Latency):** เอเจนต์ตอบสนองเร็วแค่ไหน? เวลารอนานส่งผลเสียต่อประสบการณ์ผู้ใช้ ควรวัดความหน่วงสำหรับงานและขั้นตอนย่อยโดยการติดตามการทำงานของเอเจนต์ เช่น เอเจนต์ที่ใช้เวลา 20 วินาทีสำหรับการเรียกใช้โมเดลทั้งหมด อาจเร่งความเร็วได้ด้วยการใช้โมเดลที่เร็วกว่า หรือเรียกใช้โมเดลพร้อมกันแบบขนาน

**ต้นทุน (Costs):** ค่าใช้จ่ายต่อการรันหนึ่งครั้งของเอเจนต์เท่าไร? เอเจนต์ AI อาศัยการเรียก LLM ที่คิดค่าบริการต่อตัวอักษรหรือ API ภายนอก การใช้เครื่องมือบ่อยครั้งหรือการส่งคำขอหลายครั้งสามารถเพิ่มต้นทุนอย่างรวดเร็ว เช่น ถ้าเอเจนต์เรียกใช้ LLM ห้าครั้งเพื่อปรับปรุงคุณภาพเล็กน้อย คุณควรประเมินว่าต้นทุนนี้คุ้มค่าหรือไม่ หรือคงลดจำนวนครั้งเรียกใช้หรือใช้โมเดลราคาถูกกว่าได้ การมอนิเตอร์แบบเวลาจริงยังช่วยระบุจุดที่เกิดค่าใช้จ่ายพุ่งสูงโดยไม่คาดคิด เช่น บักที่ทำให้เกิดการเรียก API ซ้ำ ๆ

**ความผิดพลาดของคำขอ (Request Errors):** มีคำขอล้มเหลวกี่ครั้ง? ซึ่งอาจรวมถึงข้อผิดพลาด API หรือล้มเหลวในการเรียกเครื่องมือ ในการทำให้เอเจนต์มีความทนทานมากขึ้นในการใช้งานจริง คุณอาจต้องตั้งกลไกการสำรองหรือการลองใหม่ เช่น ถ้าผู้ให้บริการ LLM A ไม่ทำงาน คุณสามารถสลับไปใช้ผู้ให้บริการ LLM B เป็นแบ็คอัพได้

**ฟีดแบ็กจากผู้ใช้ (User Feedback):** การประเมินโดยตรงจากผู้ใช้ให้ข้อมูลเชิงลึกที่มีค่า อาจเป็นการให้คะแนนชัดเจน (👍 ชอบ/👎 ไม่ชอบ, ⭐1-5 ดาว) หรือความคิดเห็นเป็นข้อความ ความคิดเห็นเชิงลบอย่างต่อเนื่องควรเตือนให้คุณทราบว่าเอเจนต์อาจทำงานไม่เป็นไปตามคาด

**ฟีดแบ็กซ่อนเร้นจากผู้ใช้ (Implicit User Feedback):** พฤติกรรมผู้ใช้ให้ฟีดแบ็กแบบอ้อมแม้ไม่มีการให้คะแนนตรง เช่น การเปลี่ยนคำถามทันที การถามซ้ำ หรือคลิกปุ่มลองใหม่ เช่น ถ้าคุณเห็นว่าผู้ใช้ถามคำถามเดิมซ้ำ ๆ นั่นคือสัญญาณว่าเอเจนต์ทำงานไม่ถูกต้อง

**ความถูกต้อง (Accuracy):** เอเจนต์สร้างผลลัพธ์ที่ถูกต้องหรือเป็นที่ต้องการบ่อยแค่ไหน? ความหมายของความถูกต้องแตกต่างกันไป (เช่น ความถูกต้องในการแก้ปัญหา, ความแม่นยำในการค้นข้อมูล, ความพึงพอใจของผู้ใช้) ขั้นตอนแรกคือกำหนดนิยามความสำเร็จสำหรับเอเจนต์ของคุณ คุณสามารถติดตามความถูกต้องผ่านการตรวจสอบอัตโนมัติ คะแนนการประเมิน หรือป้ายกำกับสถานะงาน ตัวอย่างเช่น การทำเครื่องหมายแทรซว่า "สำเร็จ" หรือ "ล้มเหลว"

**เมตริกการประเมินอัตโนมัติ (Automated Evaluation Metrics):** คุณยังสามารถตั้งค่าการประเมินอัตโนมัติได้ เช่น ใช้ LLM เพื่อให้คะแนนผลลัพธ์ของเอเจนต์ว่าเป็นประโยชน์ ถูกต้อง หรือไม่ นอกจากนี้ยังมีไลบรารีโอเพนซอร์สหลายตัวที่ช่วยให้คะแนนมิติต่าง ๆ ของเอเจนต์ เช่น [RAGAS](https://docs.ragas.io/) สำหรับเอเจนต์ RAG หรือ [LLM Guard](https://llm-guard.com/) เพื่อการตรวจจับภาษาอันตรายหรือการฉีดคำสั่ง

ในทางปฏิบัติ การผสมผสานตัวชี้วัดเหล่านี้จะให้ภาพรวมที่ดีที่สุดของสุขภาพของเอเจนต์ AI ในโน้ตบุ๊กตัวอย่างบทนี้ [example notebook](./code_samples/10-expense_claim-demo.ipynb) เราจะแสดงให้คุณเห็นว่าตัวชี้วัดเหล่านี้เป็นอย่างไรในตัวอย่างจริง แต่ก่อนอื่นเราจะเรียนรู้ว่ากระบวนการประเมินแบบปกติเป็นอย่างไร

## ติดตั้งเครื่องมือกับเอเจนต์ของคุณ

เพื่อรวบรวมข้อมูลแทรซ คุณจำเป็นต้องติดตั้งเครื่องมือในโค้ดของคุณ เป้าหมายคือการติดตั้งโค้ดของเอเจนต์เพื่อส่งออกแทรซและตัวชี้วัดที่สามารถถูกเก็บ ประมวลผล และแสดงผลโดยแพลตฟอร์มสังเกตการณ์

**OpenTelemetry (OTel):** [OpenTelemetry](https://opentelemetry.io/) ได้กลายเป็นมาตรฐานอุตสาหกรรมสำหรับการสังเกตการณ์ LLM มันให้ชุด API, SDK และเครื่องมือสำหรับการสร้าง, รวบรวม, และส่งออกข้อมูลเทเลเมทรี

มีไลบรารีการติดตั้งเครื่องมือหลายตัวที่ห่อหุ้มเอเจนต์เฟรมเวิร์กที่มีอยู่และทำให้ง่ายต่อการส่งออกสแปน OpenTelemetry ไปยังเครื่องมือสังเกตการณ์ Microsoft Agent Framework รวมการใช้งานกับ OpenTelemetry โดยตรง ด้านล่างคือตัวอย่างการติดตั้งเครื่องมือกับเอเจนต์ MAF:

```python
from agent_framework.observability import get_tracer, get_meter

tracer = get_tracer()
meter = get_meter()

with tracer.start_as_current_span("agent_run"):
    # การดำเนินการของเอเย่นต์ถูกติดตามโดยอัตโนมัติ
    pass
```

**การสร้างสแปนด้วยตนเอง:** ในขณะที่ไลบรารีเครื่องมือติดตั้งให้ฐานข้อมูลที่ดี แต่บางครั้งก็จำเป็นต้องมีข้อมูลละเอียดหรือกำหนดเอง คุณสามารถสร้างสแปนด้วยตนเองเพื่อเพิ่มตรรกะเฉพาะในแอปพลิเคชัน และสำคัญกว่านั้น คุณสามารถเสริมสแปนที่สร้างโดยอัตโนมัติหรือด้วยมือด้วยแอตทริบิวต์ปรับแต่ง (ซึ่งเรียกอีกอย่างว่าแท็กหรือเมตาดาต้า) แอตทริบิวต์เหล่านี้อาจรวมถึงข้อมูลเฉพาะธุรกิจ การคำนวณระหว่างทาง หรือบริบทที่ใช้สำหรับการดีบักหรือการวิเคราะห์ เช่น `user_id`, `session_id`, หรือ `model_version`

ตัวอย่างการสร้างแทรซและสแปนด้วยตนเองโดยใช้ [Langfuse Python SDK](https://langfuse.com/docs/sdk/python/sdk-v3):

## การประเมินเอเจนต์

การสังเกตการณ์ให้ตัวชี้วัดแก่เรา แต่การประเมินผลคือกระบวนการวิเคราะห์ข้อมูลนั้น (และทำการทดสอบ) เพื่อกำหนดว่าเอเจนต์ AI ทำงานดีแค่ไหนและจะปรับปรุงได้อย่างไร กล่าวคือ เมื่อคุณมีแทรซและตัวชี้วัดแล้ว จะใช้ข้อมูลเหล่านั้นตัดสินใจเกี่ยวกับเอเจนต์และการดำเนินการอย่างไร?

การประเมินผลเป็นประจำมีความสำคัญเพราะเอเจนต์ AI มักไม่เป็นแบบกำหนดตายตัวและสามารถพัฒนาได้ (ผ่านการอัปเดตหรือการเปลี่ยนแปลงพฤติกรรมของโมเดล) – หากไม่มีการประเมิน คุณจะไม่ทราบว่า “เอเจนต์อัจฉริยะ” ของคุณทำงานได้ดีจริงหรือถดถอยลง

การประเมินเอเจนต์ AI แบ่งออกเป็นสองประเภท: **การประเมินออนไลน์** และ **การประเมินออฟไลน์** ทั้งสองแบบมีคุณค่าและเสริมซึ่งกันและกัน โดยปกติเราเริ่มต้นด้วยการประเมินออฟไลน์ เพราะนี่คือขั้นตอนขั้นต่ำที่ต้องทำก่อนปรับใช้เอเจนต์ใด ๆ

### การประเมินออฟไลน์

![Dataset items in Langfuse](https://langfuse.com/images/cookbook/example-autogen-evaluation/example-dataset.png)

กระบวนการนี้คือการประเมินเอเจนต์ในสภาพแวดล้อมที่ควบคุมได้ โดยปกติใช้ชุดข้อมูลทดสอบ ไม่ใช่คำถามจากผู้ใช้จริง คุณใช้ชุดข้อมูลที่คัดเลือกมาเรียบร้อย ซึ่งคุณรู้ล่วงหน้าว่าผลลัพธ์ที่คาดหวังคืออะไร หรือพฤติกรรมที่ถูกต้องเป็นแบบไหน แล้วรันเอเจนต์ของคุณกับชุดข้อมูลเหล่านั้น

ตัวอย่างเช่น หากคุณสร้างเอเจนต์แก้โจทย์คำศัพท์คณิตศาสตร์ คุณอาจมีชุดข้อมูลทดสอบ [test dataset](https://huggingface.co/datasets/gsm8k) จำนวน 100 ปัญหาที่รู้คำตอบล่วงหน้า การประเมินออฟไลน์มักทำในช่วงพัฒนา (และอาจเป็นส่วนหนึ่งของ pipeline CI/CD) เพื่อตรวจสอบการปรับปรุงหรือป้องกันความถดถอย ข้อดีคือมัน **ทำซ้ำได้ และคุณจะได้ตัวชี้วัดความแม่นยำที่ชัดเจนเพราะมีความจริงพื้นฐาน (ground truth)** คุณอาจจำลองคำถามผู้ใช้และวัดคำตอบของเอเจนต์กับคำตอบในอุดมคติ หรือใช้เมตริกอัตโนมัติตามที่อธิบายไว้ข้างต้น

ความท้าทายหลักของการประเมินออฟไลน์ คือการรับประกันว่าชุดข้อมูลทดสอบมีความครอบคลุมและทันสมัย — เอเจนต์อาจทำงานได้ดีในชุดทดสอบที่ตายตัว แต่ในสภาพแวดล้อมจริงอาจเจอคำถามที่แตกต่างมาก ดังนั้นคุณควรปรับปรุงชุดทดสอบอย่างต่อเนื่องด้วยเคสขอบและตัวอย่างใหม่ ๆ ที่สะท้อนสถานการณ์จริง การผสมผสานชุดเล็กสำหรับ “smoke test” กับชุดใหญ่สำหรับวัดประสิทธิภาพกว้าง ๆ มีประโยชน์: ชุดเล็กสำหรับตรวจสอบอย่างรวดเร็วและชุดใหญ่สำหรับตัวชี้วัดประสิทธิภาพที่ครอบคลุม

### การประเมินออนไลน์

![Observability metrics overview](https://langfuse.com/images/cookbook/example-autogen-evaluation/dashboard.png)

คือการประเมินเอเจนต์ในสภาพแวดล้อมจริงแบบสด ๆ ระหว่างการใช้งานจริงในสภาพแวดล้อมการผลิต การประเมินออนไลน์เกี่ยวข้องกับการมอนิเตอร์ประสิทธิภาพของเอเจนต์บนปฏิสัมพันธ์ของผู้ใช้จริงและวิเคราะห์ผลลัพธ์อย่างต่อเนื่อง

เช่น คุณอาจติดตามอัตราความสำเร็จ คะแนนความพึงพอใจของผู้ใช้ หรือเมตริกอื่น ๆ บนผู้ใช้จริง ข้อดีของการประเมินออนไลน์คือมัน **สามารถจับสิ่งที่คุณคาดไม่ถึงในห้องแลปได้** — คุณสามารถสังเกตการเปลี่ยนแปลงของโมเดลตามเวลา (หากประสิทธิภาพเอเจนต์ลดลงเมื่อรูปแบบอินพุตเปลี่ยนไป) และจับคำถามหรือสถานการณ์ที่ไม่เคยอยู่ในชุดข้อมูลทดสอบมาก่อนได้ ให้ภาพจริงของพฤติกรรมเอเจนต์ในสภาพแวดล้อมป่า

การประเมินออนไลน์มักรวมการเก็บข้อมูลฟีดแบ็กโดยตรงและโดยอ้อมจากผู้ใช้ตามที่กล่าวมา และอาจมีการทดสอบแบบเงาหรือทดสอบ A/B (ซึ่งเอเจนต์เวอร์ชันใหม่รันคู่ขนานเพื่อเปรียบเทียบกับเวอร์ชันเก่า) ความท้าทายคือการหาคะแนนหรือป้ายกำกับที่เชื่อถือได้สำหรับปฏิสัมพันธ์จริง อาจต้องพึ่งพาฟีดแบ็กผู้ใช้หรือเมตริกปลายทาง (เช่นว่าผู้ใช้คลิกผลลัพธ์หรือไม่)

### การผสมผสานทั้งสอง

การประเมินออนไลน์และออฟไลน์ไม่ขัดแย้งกัน แต่เสริมซึ่งกันและกัน ข้อมูลเชิงลึกจากการมอนิเตอร์ออนไลน์ (เช่น ประเภทคำถามใหม่ที่เอเจนต์ทำงานไม่ดี) สามารถใช้เพิ่มชุดข้อมูลทดสอบออฟไลน์และปรับปรุงได้ ตรงกันข้าม เอเจนต์ที่ประสบความสำเร็จในชุดทดสอบออฟไลน์ก็จะมั่นใจขึ้นเมื่อนำไปใช้และมอนิเตอร์แบบออนไลน์

ในความเป็นจริง ทีมงานหลายแห่งใช้วงจร:

_ประเมินออฟไลน์ -> ปรับใช้ -> มอนิเตอร์ออนไลน์ -> รวบรวมกรณีล้มเหลวใหม่ -> เพิ่มชุดข้อมูลออฟไลน์ -> ปรับเอเจนต์ -> ทำซ้ำ_

## ปัญหาทั่วไป

เมื่อนำเอเจนต์ AI ไปใช้ในสภาพแวดล้อมการผลิต คุณอาจพบกับความท้าทายหลายอย่าง ต่อไปนี้คือปัญหาทั่วไปและวิธีแก้ไขที่เป็นไปได้:

| **ปัญหา**    | **วิธีแก้ไขที่เป็นไปได้**   |
| ------------- | ------------------ |
| เอเจนต์ AI ทำงานไม่สม่ำเสมอ | - ปรับปรุงคำสั่ง (prompt) ที่ให้กับเอเจนต์ AI ให้ชัดเจนในวัตถุประสงค์<br>- ระบุว่าการแบ่งงานออกเป็นงานย่อย ๆ และจัดการโดยเอเจนต์หลายตัวจะช่วยได้ |
| เอเจนต์ AI เข้าในลูปซ้ำไม่จบ | - กำหนดเงื่อนไขการหยุดงานให้ชัดเจนเพื่อให้เอเจนต์รู้ว่าเมื่อใดควรหยุด<br>- สำหรับงานซับซ้อนที่ต้องเหตุผลและการวางแผน ใช้โมเดลขนาดใหญ่ที่เชี่ยวชาญงานเหตุผล |
| การเรียกใช้เครื่องมือของเอเจนต์ล้มเหลวบ่อย | - ทดสอบและตรวจสอบผลลัพธ์ของเครื่องมือภายนอกระบบเอเจนต์<br>- ปรับแต่งพารามิเตอร์ คำสั่ง (prompt) และชื่อเครื่องมือให้เหมาะสม |
| ระบบมัลติเอเจนต์ทำงานไม่สม่ำเสมอ | - ปรับปรุงคำสั่งให้กับแต่ละเอเจนต์ให้เฉพาะเจาะจงและแตกต่างกัน<br>- สร้างระบบลำดับชั้นโดยใช้เอเจนต์ “routing” หรือคอนโทรลเลอร์เพื่อกำหนดว่าเอเจนต์ใดถูกต้อง |

ปัญหาเหล่านี้หลายอย่างสามารถตรวจพบได้อย่างมีประสิทธิภาพมากขึ้นด้วยการสังเกตการณ์ แทรซและตัวชี้วัดที่กล่าวถึงช่วยให้ระบุจุดในกระบวนการทำงานของเอเจนต์ที่เกิดปัญหาได้อย่างแม่นยำ ทำให้การดีบักและการปรับปรุงประสิทธิภาพมีประสิทธิผลมากขึ้น

## การจัดการต้นทุน
นี่คือกลยุทธ์บางประการในการจัดการต้นทุนของการนำ AI agents ไปใช้ในโปรดักชัน:

**การใช้โมเดลขนาดเล็ก:** Small Language Models (SLMs) สามารถทำงานได้ดีในกรณีการใช้งานเอเยนต์บางประเภทและจะช่วยลดต้นทุนอย่างมีนัยสำคัญ ดังที่กล่าวไว้ก่อนหน้านี้ การสร้างระบบประเมินผลเพื่อกำหนดและเปรียบเทียบประสิทธิภาพเมื่อเทียบกับโมเดลขนาดใหญ่เป็นวิธีที่ดีที่สุดในการทำความเข้าใจว่า SLM จะทำงานได้ดีเพียงใดในกรณีการใช้งานของคุณ พิจารณาใช้ SLM สำหรับงานที่ง่ายกว่ เช่น การจัดหมวดหมู่เจตนา หรือการดึงพารามิเตอร์ ในขณะที่เก็บโมเดลขนาดใหญ่ไว้สำหรับการคิดวิเคราะห์ที่ซับซ้อน

**การใช้โมเดลตัวกรอง (Router Model):** กลยุทธ์ที่คล้ายกันคือการใช้โมเดลและขนาดที่หลากหลาย คุณสามารถใช้ LLM/SLM หรือฟังก์ชันแบบ serverless เพื่อกำหนดเส้นทางคำขอตามความซับซ้อนไปยังโมเดลที่เหมาะสมที่สุด วิธีนี้จะช่วยลดต้นทุนในขณะเดียวกันก็รับประกันประสิทธิภาพในงานที่เหมาะสม ตัวอย่างเช่น กำหนดเส้นทางคำถามง่าย ๆ ไปยังโมเดลขนาดเล็กและรวดเร็ว และใช้โมเดลขนาดใหญ่ที่แพงเฉพาะสำหรับงานคิดวิเคราะห์ที่ซับซ้อนเท่านั้น

**การแคชคำตอบ:** การระบุคำขอและงานที่เกิดขึ้นบ่อยและจัดเตรียมคำตอบไว้ล่วงหน้าก่อนที่คำขอเหล่านั้นจะผ่านระบบเอเยนต์ของคุณเป็นวิธีที่ดีในการลดปริมาณคำขอที่คล้ายกัน คุณยังสามารถสร้างกระบวนการเพื่อระบุว่าคำขอนั้นมีความคล้ายคลึงกับคำขอที่แคชไว้เท่าใดโดยใช้โมเดล AI ที่พื้นฐานกว่า กลยุทธ์นี้จะช่วยลดต้นทุนได้มากสำหรับคำถามที่ถามบ่อยหรือเวิร์กโฟลว์ทั่วไป
