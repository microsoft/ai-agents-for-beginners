# تأمين وكلاء الذكاء الاصطناعي بالإيصالات التشفيرية

## مقدمة

سيغطي هذا الدرس:

- لماذا تعتبر سجلات التدقيق لوكلاء الذكاء الاصطناعي مهمة للامتثال، وتصحيح الأخطاء، والثقة.
- ما هي الإيصالات التشفيرية وكيف تختلف عن سطر سجل غير موقع.
- كيفية إنتاج إيصال موقع لمكالمة أداة الوكيل باستخدام بايثون العادي.
- كيفية التحقق من الإيصال بدون اتصال واكتشاف العبث.
- كيفية ربط الإيصالات بحيث يؤدي إزالة أو إعادة ترتيب واحدة إلى كسر السلسلة.
- ما الذي تثبته الإيصالات وما الذي لا تثبته صراحةً.

## أهداف التعلم

بعد إكمال هذا الدرس، ستعرف كيف:

- تحدد أوضاع الفشل التي تحفز إثبات الأصل التشفيري لأفعال الوكيل.
- تنتج إيصالاً موقعًا بـ Ed25519 على حمولة JSON معيارية.
- تتحقق من إيصال بشكل مستقل باستخدام المفتاح العام فقط.
- تكتشف العبث بإعادة تشغيل التحقق على إيصال معدل.
- تبني تسلسلًا مرتبطًا بالهاش للإيصالات وتشرح لماذا السلسلة مهمة.
- تميز الحدود بين ما تثبته الإيصالات (النسبة، السلامة، الترتيب) وما لا تثبته (صحة الفعل، سلامة السياسة).

## المشكلة: سجل تدقيق وكيلك

تخيل أنك نشرت وكيلاً ذكاءً اصطناعيًا لشركة كونتوسو للسفر. يقرأ الوكيل طلبات العملاء، وينادي على واجهة API للرحلات للبحث عن الخيارات، ويحجز المقاعد نيابة عن العميل. في الربع الأخير، عالج الوكيل 50,000 حجز.

اليوم يحضر مدقق. يطرح سؤالًا بسيطًا: "أرني ماذا فعل وكيلك."

تسلمه ملفات السجل الخاصة بك. ينظر المدقق إليها ويسأل السؤال الأصعب: "كيف أعلم أن هذه السجلات لم تُعدل؟"

هذه هي مشكلة سجل التدقيق. معظم نشرات الوكلاء اليوم تعتمد على:

- **سجلات التطبيق**: يكتبها الوكيل نفسه، يمكن تعديلها من أي شخص لديه وصول إلى نظام الملفات.
- **خدمات تسجيل السحابة**: تكشف عن العبث على مستوى المنصة لكن فقط إذا كان المدقق يثق في مشغل المنصة.
- **سجلات المعاملات في قواعد البيانات**: مناسبة جيدًا لتغييرات قواعد البيانات لكنها غير مناسبة لمكالمات الأدوات التعسفية.

لا يمكن لأي منها أن يجيب على سؤال المدقق دون أن يطلب من المدقق أن يثق بشخص ما (أنت، موفر السحابة، بائع قاعدة البيانات). للاستخدام الداخلي، غالبًا ما يكون هذا الثقة مقبولة. للأعباء المنظمة (المالية، الرعاية الصحية، أي شيء يخضع لقانون الذكاء الاصطناعي في الاتحاد الأوروبي)، لا تكون كذلك.

الإيصالات التشفيرية تحل هذه المشكلة بجعل كل إجراء للوكيل قابلًا للتحقق بشكل مستقل. لا يحتاج المدقق إلى الوثوق بك. يحتاج فقط إلى مفتاحك العام والإيصال نفسه.

## ما هو الإيصال التشفيري؟

الإيصال هو كائن JSON يسجل ما فعله الوكيل، موقع بتوقيع رقمي.

```mermaid
flowchart LR
    A[الوكيل يستدعي أداة] --> B[بناء حمولة الإيصال]
    B --> C[تطبيع JSON وفق RFC 8785]
    C --> D[تجزئة SHA-256]
    D --> E[توقيع Ed25519]
    E --> F[الإيصال مع التوقيع]
    F --> G[المراجع يتحقق دون اتصال]
    G --> H{هل التوقيع صالح؟}
    H -- نعم --> I[دليل مقاوم للتلاعب]
    H -- لا --> J[تم رفض الإيصال]
```

الإيصال الأدنى يبدو هكذا:

```json
{
  "type": "agent.tool_call.v1",
  "agent_id": "contoso-travel-bot",
  "tool_name": "lookup_flights",
  "tool_args_hash": "sha256:a3f9c1...",
  "result_hash": "sha256:7b2e1d...",
  "policy_id": "contoso-travel-policy-v3",
  "timestamp": "2026-04-25T14:30:00Z",
  "sequence": 47,
  "previous_receipt_hash": "sha256:9d4e6a...",
  "signature": {
    "alg": "EdDSA",
    "sig": "c5af83...",
    "public_key": "8f3b2c..."
  }
}
```

ثلاث خصائص تقوم بالعمل:

1. **التوقيع**. الإيصال يوقعه بوابة الوكيل باستخدام مفتاح خاص Ed25519. يمكن لأي شخص لديه المفتاح العام المقابل التحقق من التوقيع بدون اتصال. العبث بأي حقل يبطل التوقيع.

2. **الترميز المعياري**. قبل التوقيع، يتم تسلسل الإيصال باستخدام مخطط التوحيد القياسي لـ JSON (JCS، RFC 8785). هذا يضمن أن اثنين من التطبيقات التي تنتج نفس الإيصال المنطقي تنتج خروجًا مطابقًا بالبايت. بدون التوحيد، سينتج عن محولات JSON المختلفة توقيعات مختلفة لنفس المحتوى.

3. **ربط الهاش**. حقل `previous_receipt_hash` يربط كل إيصال بالإيصال السابق له. إزالة أو إعادة ترتيب إيصال يكسر كل الإيصالات التي تليه. يصبح العبث مرئيًا على مستوى السلسلة حتى لو تم تجاوز التوقيعات الفردية.

معًا، توفر هذه الخصائص ثلاث ضمانات:

- **النسبة**: هذا المفتاح وقع هذا المحتوى.
- **السلامة**: المحتوى لم يتغير منذ التوقيع.
- **الترتيب**: جاء هذا الإيصال بعد ذلك الإيصال في السلسلة.

## إنتاج إيصال في بايثون

لا تحتاج إلى مكتبة خاصة لإنتاج إيصال. البنى التشفيرية متوفرة على نطاق واسع والمنطق عدة عشرات من أسطر بايثون.

تمارين العمل اليدوي في `code_samples/18-signed-receipts.ipynb` تشرح السلسلة كاملة. النسخة الملخصة:

```python
import json
import hashlib
import base64
from nacl import signing
from jcs import canonicalize  # JSON قانوني بموجب RFC 8785

def b64url_nopad(data: bytes) -> str:
    return base64.urlsafe_b64encode(data).decode("ascii").rstrip("=")

def sha256_canonical(obj) -> str:
    """SHA-256 of a Python object's JCS-canonical JSON form."""
    return f"sha256:{hashlib.sha256(canonicalize(obj)).hexdigest()}"

# توليد أو تحميل مفتاح التوقيع (في الإنتاج، يخزن في مخزن المفاتيح)
signing_key = signing.SigningKey.generate()
verify_key = signing_key.verify_key

# بناء حمولة الإيصال (بدون توقيع بعد)
tool_args = {"origin": "SYD", "destination": "LAX"}
tool_result = [{"flight": "QF11", "price": 1850, "stops": 0}]

payload = {
    "type": "agent.tool_call.v1",
    "agent_id": "contoso-travel-bot",
    "tool_name": "lookup_flights",
    "tool_args_hash": sha256_canonical(tool_args),
    "result_hash": sha256_canonical(tool_result),
    "policy_id": "contoso-travel-policy-v3",
    "timestamp": "2026-04-25T14:30:00Z",
    "sequence": 0,
    "previous_receipt_hash": None,
}

# تحويل إلى صيغة قانونية، تجزئة، توقيع.
canonical_bytes = canonicalize(payload)
message_hash = hashlib.sha256(canonical_bytes).digest()
signature_bytes = signing_key.sign(message_hash).signature

# إرفاق كائن توقيع منظم.
receipt = {
    **payload,
    "signature": {
        "alg": "EdDSA",
        "sig": b64url_nopad(signature_bytes),
        "public_key": b64url_nopad(bytes(verify_key)),
    },
}
```

هذه هي كامل سلسلة التوقيع. التمارين في دفتر الملاحظات تشرح كل خطوة.

## التحقق من إيصال واكتشاف العبث

التحقق هو العملية العكسية:

```python
import base64
import hashlib
from nacl import signing
from nacl.exceptions import BadSignatureError
from jcs import canonicalize

def b64url_decode(s: str) -> bytes:
    padding = "=" * ((4 - len(s) % 4) % 4)
    return base64.urlsafe_b64decode(s + padding)

def verify_receipt(receipt: dict) -> bool:
    # التوقيع هو كائن منظم: {"alg"، "sig"، "public_key"}.
    sig_obj = receipt.get("signature")
    if not sig_obj or sig_obj.get("alg") != "EdDSA":
        return False

    # أعد بناء الحمولة التي تم توقيعها فعليًا (كل شيء باستثناء التوقيع).
    payload = {k: v for k, v in receipt.items() if k != "signature"}

    canonical_bytes = canonicalize(payload)
    message_hash = hashlib.sha256(canonical_bytes).digest()

    try:
        verify_key = signing.VerifyKey(b64url_decode(sig_obj["public_key"]))
        verify_key.verify(message_hash, b64url_decode(sig_obj["sig"]))
        return True
    except BadSignatureError:
        return False
```

تأخذ هذه الدالة إيصالًا وتعيد `True` إذا كان التوقيع صالحًا، و`False` إذا لم يكن كذلك. لا اتصال بشبكة، لا اعتماد على خدمة، لا حاجة لثقة بأي طرف ثالث.

لرؤية كشف العبث في العمل، يشرح الدفتر:

1. إنتاج إيصال صالح والتأكد من صحته.
2. تعديل بايت واحد من حقل `tool_args_hash`.
3. إعادة التحقق ورؤية الفشل.

هذا هو الإثبات العملي على أن الإيصالات تكشف العبث: أي تعديل مهما كان صغيرًا يكسر التوقيع.

## ربط الإيصالات للوكلاء متعدد الخطوات

يحمي إيصال موقع واحد إجراءً واحدًا. تسلسل من الإيصالات يحمي سلسلة.

```mermaid
flowchart LR
    R0[الإيصال 0<br/>التكوين] --> R1[الإيصال 1]
    R1 --> R2[الإيصال 2]
    R2 --> R3[الإيصال 3]
    R1 -. previous_receipt_hash .-> R0
    R2 -. previous_receipt_hash .-> R1
    R3 -. previous_receipt_hash .-> R2
```

يسجل كل إيصال هاش الإيصال السابق له. لإزالة الإيصال 2 بصمت، يحتاج المهاجم إلى:

- تعديل حقل `previous_receipt_hash` في الإيصال 3 (يكسر توقيع الإيصال 3)، أو
- تزوير توقيع جديد على الإيصال 3 المعدل (يتطلب المفتاح الخاص للوكيل).

إذا كان المفتاح الخاص في خزنة مفاتيح مادية ونشرت المفتاح العام مع كل إيصال، لا يمكن تنفيذ أي هجوم دون اكتشاف.

يشرح الدفتر:

1. بناء سلسلة من ثلاثة إيصالات.
2. التحقق من أن كل `previous_receipt_hash` للإيصال يطابق الهاش الفعلي للإيصال السابق.
3. العبث بإيصال في الوسط ورؤية السلسلة تكسر تمامًا عند تلك النقطة.

هذه هي طريقة إنتاج سجل تدقيق يمكن لمدقق خارجي التحقق منه بدون الوثوق بك.

## ما تبرزه الإيصالات (وما لا تبرزه)

هذه أهم فقرة في الدرس. الإيصالات قوية لكن قوتها محدودة.

**تثبت الإيصالات ثلاثة أشياء:**

1. **النسبة**: مفتاح معين وقع حمولة معينة.
2. **السلامة**: الحمولة لم تتغير منذ التوقيع.
3. **الترتيب**: جاء هذا الإيصال بعد ذاك في سلسلة الهاش.

**الإيصالات لا تثبت:**

1. **الصحة**: أن فعل الوكيل كان الفعل الصحيح. يمكن توقيع إيصال على إجابة خاطئة كما على الإجابة الصحيحة.
2. **الامتثال للسياسة**: أن السياسة المشار إليها في `policy_id` قد تم تقييمها فعليًا، أو أنها كانت ستسمح بهذا الفعل إذا فُحصت. الإيصال يسجل ما تم الادعاء به، وليس ما تم تطبيقه.
3. **الهوية خارج المفتاح**: يقول الإيصال "هذا المفتاح وقع هذا المحتوى." لا يقول "هذا الإنسان فوَّض هذا." الربط بين مفتاح وشخص أو منظمة يتطلب بنية تحتية هوية منفصلة (دليل، سجل مفاتيح عامة، الخ).
4. **صدق المدخلات**: إذا تلقى الوكيل موجهًا معدلًا وتصرف بناءً عليه، يسجل الإيصال الفعل بأمانة. الإيصالات تأتي بعد تحقق المدخلات، وليست بديلاً عنه.

هذه الحدود مهمة لسببين:

- تخبرك لما تكون الإيصالات مفيدة: لجعل سلوك الوكيل قابلًا للتدقيق وكشف العبث، حتى عبر حدود منظمات مختلفة.
- تخبرك ما هي الطبقات الإضافية التي لا تزال تحتاجها: التحقق من المدخلات (الدرس 6)، تطبيق السياسات (مذكور إجمالًا أدناه)، وبنية الهوية (خارج نطاق هذا الدرس).

خطأ شائع هو افتراض أن "لدينا إيصالات" تعني "نحن محكومون." هذا غير صحيح. الإيصالات هي الأساس. الحوكمة هي النظام الذي تبنيه فوقه.

## مراجع الإنتاج

كود بايثون في هذا الدرس متعمد أن يكون بسيطًا لكي تقرأ كل سطر وتفهم بالضبط ما يحدث. في الإنتاج، لديك خياران:

1. **البناء مباشرة على البُنى التشفيرية.** الـ 50 سطرًا التي رأيتها أعلاه تكفي لكثير من حالات الاستخدام. مكتبة PyNaCl (Ed25519) وحزمة `jcs` (JSON المعياري) هما مكتبتان مدعومتان وموثوقتان.

2. **استخدام مكتبة إيصال إنتاجية.** هناك عدة مشاريع مفتوحة المصدر تطبق نفس النمط مع ميزات إضافية (تدوير المفاتيح، التحقق بالجملة، توزيع مجموعة JWK، التكامل مع محركات السياسات):
   - صيغة الإيصال المستخدمة في هذا الدرس تتبع مسودة IETF (`draft-farley-acta-signed-receipts`) في عملية المعايير حاليًا.
   - مجموعة أدوات حوكمة الوكيل من مايكروسوفت تدمج الإيصالات مع قرارات السياسات القائمة على Cedar؛ انظر درس 33 في ذلك المستودع لمثال شامل.
   - حزم `protect-mcp` (npm) و `@veritasacta/verify` (npm) توفر تنفيذ Node لتوقيع الإيصال والتحقق بدون اتصال، مخصصة لتغليف أي خادم MCP مع سجل تدقيق يكشف العبث.
   - مكتبة **[nobulex](https://github.com/arian-gogani/nobulex)** لبايثون (`pip install nobulex`) توفر نفس نمط توقيع Ed25519 + JCS مع تكاملات LangChain و CrewAI، بما في ذلك نشر متجهات اختبار التحقق المتقاطع وخريطة الامتثال المساهمة عبر [OWASP PR #2210](https://github.com/OWASP/CheatSheetSeries/pull/2210).

القرار بين بناء المكتبة بنفسك أو استخدام مكتبة مماثل لقرار بين كتابة مكتبة JWT بنفسك أو استخدام مكتبة مختبرة: كلاهما معقول؛ المكتبة توفر الوقت وتقلل السطح المدقق؛ النهج من الصفر يجبرك على فهم كل بُنية. هذا الدرس يعلم طريق الصفر لكي تملك الأساس لأي خيار.

</details>

</details>

<summary>الإجابة</summary>

لا. الإيصال الصالح يثبت ثلاثة أشياء: النسبة (هذا المفتاح وقع هذا المحتوى)، السلامة (المحتوى لم يتغير)، والترتيب (هذا الإيصال جاء بعد ذاك). لا يثبت أن الفعل كان صحيحًا، أو أن السياسة المذكورة في `policy_id` قد قُيمت فعليًا، أو أن الوكيل اتبع كل القواعد. تجعل الإيصالات سلوك الوكيل قابلاً للتدقيق، وليس بالضرورة صحيحًا. هذه هي الحدود الأهم في الدرس.

## تمرين عملي

افتح `code_samples/18-signed-receipts.ipynb` وأكمل الأقسام الأربعة كلها:

1. **القسم 1**: وقع أول إيصال لك وتأكد من صحته.
2. **القسم 2**: عبث بالإيصال وراقب فشل التحقق.
3. **القسم 3**: أنشئ سلسلة من ثلاثة إيصالات وتحقق من سلامة السلسلة.
4. **القسم 4**: طبق النمط على وكيل مبني بإطار عمل وكيل مايكروسوفت: غلف مكالمة أداة بتوقيع الإيصال، ثم تحقق من الإيصال بشكل مستقل.
**التحدي الإضافي 1:** قم بتمديد مخطط الإيصال بحقل إضافي تختاره بنفسك (على سبيل المثال، معرف طلب للتتبع)، حدّث منطق التوقيع الرسمي ليشمله، وتأكد من أن الإيصال لا يزال يمر بعملية التحقق بشكل دائري. ثم عدّل الحقل بعد التوقيع وتأكد من فشل التحقق. هذا يجبرك على فهم كيف يساهم كل بايت من الترميز الرسمي في التوقيع.

**التحدي الإضافي 2:** قم بعمل تجزئة SHA-256 لاثنين من إيصالاتك معًا (قم بدمج بايتاتهم الرسمية بترتيب حتمي) وقم بتضمين الخلاصة الناتجة كحقل جديد في إيصال ثالث قبل توقيعه. تحقق من أن جميع الإيصالات الثلاثة لا تزال تمر بعملية التحقق بشكل دائري. لقد قمت ببناء إثبات شمول خطوة واحدة: أي شخص يحمل الإيصال الثالث يمكنه إثبات أن الإيصالين الأول والثاني كانا موجودين وقت توقيعه، دون الحاجة إلى الكشف عن محتوياتهم. هذه هي الطريقة التي تستخدمها إيصالات الكشف الانتقائي على نطاق واسع (التزامات ميركل، RFC 6962).

## الخلاصة

توفر الإيصالات المشفرة لوكلاء الذكاء الاصطناعي مسار تدقيق يتميز بـ:

- **قابلية التحقق المستقلة:** يمكن لأي طرف يمتلك المفتاح العام التحقق، دون الاعتماد على خدمة.
- **مقاومة التلاعب:** أي تعديل يبطل التوقيع.
- **قابلية النقل:** الإيصال هو ملف JSON صغير؛ يمكن أرشفته، نقله، والتحقق منه في أي مكان.
- **متوافقة مع المعايير:** مبنية على Ed25519 (RFC 8032)، JCS (RFC 8785)، وSHA-256، وهي جميعًا أدوات منتشرة على نطاق واسع.

هي ليست بديلاً عن التحقق من المدخلات، أو فرض السياسات، أو بنية التوثيق. هي أساس لهذه الطبقات. عندما تقوم بنشر وكلاء في بيئات منظمة بأنظمة مراقبة متعددة المنظمات، أو في أي حالة حيث لا يمكن الافتراض بأن المراجع المستقبلي سيثق بك، تكون الإيصالات هي كيفية جعل مسار التدقيق صادقًا.

أهم نقطة يجب تذكرها: الإيصالات تثبت من قال ماذا، ومتى. لا تثبت أن ما قيل كان صحيحًا أو دقيقًا. احتفظ بهذا التمييز جيدًا. هو الفرق بين نظام نسب صادق ونظام مضلل.

## قائمة التحقق للإنتاج

عندما تكون مستعدًا للانتقال من هذا الدرس إلى نشر وكلاء موقّعين بالإيصالات في بيئة حقيقية:

- [ ] **انقل مفتاح التوقيع من حاسوب المطور.** استخدم Azure Key Vault، AWS KMS، أو وحدة أمان الأجهزة. يجب ألا يعيش المفتاح الخاص الذي يوقع إيصالاتك في التحكم بالمصدر أو كنص صريح على أجهزة التطبيق.
- [ ] **انشر المفتاح العام للتحقق.** يحتاج المراجعون إليه للتحقق دون اتصال. النمط القياسي هو مجموعة JWK في عنوان URL معروف (RFC 7517)، مثل `https://your-org.example.com/.well-known/agent-keys.json`.
- [ ] **ارسا سلسلة الإيصالات خارجيًا.** قم دوريًا بكتابة تجزئة رأس السلسلة الأخيرة إلى سجل الشفافية (Sigstore Rekor، RFC 3161 سلطة الطابع الزمني، أو نظام داخلي ثاني) لكي يؤكد طرف خارجي "أن هذه السلسلة كانت موجودة في هذا الوقت."
- [ ] **خزن الإيصالات بشكل لا يمكن تغييره.** التخزين ذو الإضافة فقط (Azure Storage بسياسات عدم التغيير، AWS S3 Object Lock) يمنع شخصًا داخليًا من إعادة كتابة التاريخ على طبقة التخزين.
- [ ] **قرر سياسة الاحتفاظ.** كثير من أنظمة الامتثال تتطلب احتفاظاً لعدة سنوات. خطط لنمو الإيصالات (كل إيصال حوالي 500 بايت؛ إذا انتج الوكيل 10 آلاف بيان يوميًا فإنه ينتج نحو 1.8 جيجابايت سنوياً).
- [ ] **وثّق ما لا تغطيه الإيصالات.** الإيصالات تثبت النسبة والأمان والترتيب. يجب أن تدرج دليلك التشغيلي صراحةً ما هي الضوابط الإضافية (التحقق من المدخلات، فرض السياسات، تحديد المعدل، بنية الهوية) التي تجلس جنبًا إلى جنب مع الإيصالات في وضع الحوكمة لديك.

### هل لديك المزيد من الأسئلة حول تأمين وكلاء الذكاء الاصطناعي؟

انضم إلى [خادم ديسكورد Microsoft Foundry](https://aka.ms/ai-agents/discord) للقاء المتعلمين الآخرين، وحضور ساعات المكتب، والحصول على إجابات لأسئلتك حول وكلاء الذكاء الاصطناعي.

## ما بعد هذا الدرس

يغطي هذا الدرس توقيع إيصال واحد وتسلسلات متسلسلة باستخدام التجزئة. نفس الأدوات تشكل عدة أنماط متقدمة قد تواجهها مع تطور وضع الحوكمة لديك:

- **الكشف الانتقائي.** عندما ترتبط حقول الإيصال بشكل مستقل (شجرة ميركل على نمط RFC 6962)، يمكنك الكشف عن حقول معينة لمراجعين محددين وإثبات أن الباقي لم يتغير دون الكشف عنها. مفيد عندما يحتاج الإيصال نفسه إلى تلبية تدقيق شامل (يطلب اكتمال البيانات) ولائحة تقليل البيانات مثل GDPR (ترغب في أن يرى المراجع أقل ما يمكن).
- **إلغاء صلاحية الإيصالات.** إذا تعرّض مفتاح التوقيع للخطر، تحتاج طريقة لوضع علامة على كل الإيصالات الموقعة بذلك المفتاح كغير موثوق بها من نقطة زمنية فصاعدًا. الأنماط القياسية: مفاتيح توقيع قصيرة العمر مع قائمة إلغاء منشورة، أو سجل شفافية مع مدخلات إلغاء.
- **الإيصالات الثنائية / توقيع مقسّم.** بعض التنفيذات تقسم الحمولة الموقعة إلى نصفي قبل التنفيذ (`authorization_*`) وبعد التنفيذ (`result_*`) بتوقيعات مستقلة، مفيدة عندما يقرر شخص ما التفويض وتنتج جهة أخرى النتيجة المراقبة في أوقات أو جهات مختلفة. هذا يضاف تكميليًا على تنسيق الإيصال الذي تعلمته في هذا الدرس.
- **تكوين الحمولة.** الإيصال يختم أي بايتات تضعها في `result_hash`. الحمولة الحقيقية غالبًا ما تكون أغنى من ناتج أداة واحدة: يمكن أن تعيش داخل الحمولة منطق ما قبل القرار (التنبؤ بالنموذج، الخيارات المدروسة، الأدلة واكتمالها، مخاطر الوضع، سلسلة المساءلة، نتيجة البوابة)، كلها مختومة بإيصال واحد. هذا يحافظ على تنسيق الإيصال بسيطًا مع السماح لتطور مخططات الحمولة حسب النطاق.
- **التوافق بين التطبيقات المختلفة.** عدة تنفيذات مستقلة لنفس تنسيق الإيصال (بايثون، تايب سكريبت، راست، جو) تتحقق عبر متجهات اختبار مشتركة. إذا قمت ببناء تنفيذ خاص بك، فإن التحقق مقابل المتجهات المنشورة يؤكد التوافق السلكي.
- **الانتقال لما بعد الكم.** Ed25519 منتشر اليوم لكنه ليس مقاومًا للحوسبة الكمومية. تنسيق الإيصال مرن الخوارزميات: يمكن لحقل `signature.alg` حمل `ML-DSA-65` (معيار التوقيع بعد الكم لـNIST) عند الحاجة للانتقال. خطط لفترة انتقال حيث تكون الإيصالات موقعة بشكل مزدوج.
