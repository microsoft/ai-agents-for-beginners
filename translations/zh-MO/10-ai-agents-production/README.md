# 在生產環境中的 AI 代理人：可觀察性與評估

[![在生產環境中的 AI 代理人](../../../translated_images/zh-MO/lesson-10-thumbnail.2b79a30773db093e.webp)](https://youtu.be/l4TP6IyJxmQ?si=reGOyeqjxFevyDq9)

當 AI 代理人從實驗性原型轉為真實世界的應用時，理解其行為、監控其效能，以及系統性地評估其輸出變得非常重要。

## 學習目標

完成本課程後，您將知道/理解：
- 代理人可觀察性與評估的核心概念
- 提升代理人效能、成本與效果的技術
- 系統性評估 AI 代理人時應評估的內容與方法
- 在將 AI 代理人部署到生產環境時如何控制成本
- 如何為使用 Microsoft Agent Framework 建立的代理人加入監測與儀表化

目標是讓您具備將「黑盒」代理人轉變為透明、可管理且可靠系統的知識。

_**注意：** 部署安全且值得信賴的 AI 代理人十分重要。也請參閱 [建立值得信賴的 AI 代理人](./06-building-trustworthy-agents/README.md) 這一課程。_

## 追蹤 (Trace) 與 區段 (Span)

像 [Langfuse](https://langfuse.com/) 或 [Microsoft Foundry](https://learn.microsoft.com/en-us/azure/ai-foundry/what-is-azure-ai-foundry) 等可觀察性工具通常會將代理人執行表示為追蹤與區段。

- **Trace（追蹤）** 表示從開始到結束的完整代理人任務（例如處理使用者查詢）。
- **Spans（區段）** 是追蹤中的個別步驟（例如呼叫語言模型或檢索資料）。

![Langfuse 的追蹤樹](https://langfuse.com/images/cookbook/example-autogen-evaluation/trace-tree.png)
<!-- Image URL retained for illustration purposes -->

沒有可觀察性時，AI 代理人會像「黑盒」一樣——其內部狀態與推理過程不透明，使得診斷問題或優化效能變得困難。有了可觀察性，代理人變成「玻璃盒」，提供的透明度對建立信任並確保其按預期運作至關重要。

## 為何在生產環境中可觀察性很重要

將 AI 代理人轉入生產環境會帶來一套新的挑戰與需求。可觀察性不再是「可有可無」，而是一項關鍵能力：

*   **除錯與根因分析**：當代理人失敗或產生意外輸出時，可觀察性工具提供追蹤資料以精確找出錯誤來源。這在可能涉及多次 LLM 呼叫、工具互動與條件邏輯的複雜代理人中特別重要。
*   **延遲與成本管理**：AI 代理人常依賴按 token 或按呼叫計費的 LLM 與其他外部 API。可觀察性可使這些呼叫被精確追蹤，幫助識別過於緩慢或昂貴的操作。這讓團隊能優化 prompts、選擇更高效率的模型，或重新設計工作流程以管理營運成本並確保良好的使用者體驗。
*   **信任、安全與合規**：在許多應用中，確保代理人行為安全且符合倫理十分重要。可觀察性提供代理人行動與決策的稽核軌跡。可用來偵測並緩解像是 prompt 注入、有害內容生成或個人識別資訊（PII）處理不當等問題。例如，您可以檢視追蹤以理解代理人為何提供某個回應或使用特定工具。
*   **持續改進循環**：可觀察性資料是反覆開發流程的基礎。透過監控代理人在真實世界中的表現，團隊能識別改進空間、收集用於微調模型的資料，並驗證變更的影響。這會建立一個反饋環（由線上評估的生產洞見去指導離線實驗與精煉），逐步提升代理人的表現。

## 需要追蹤的關鍵指標

為了監控並理解代理人的行為，應追蹤各種指標與訊號。雖然具體指標會依代理人的目的不同而異，但有些是普遍重要的。

以下是可觀察性工具常監控的一些常見指標：

**延遲（Latency）：** 代理人回應速度如何？過長的等待時間會負面影響使用者體驗。您應透過追蹤代理人執行，針對任務與個別步驟衡量延遲。例如，一個在所有模型呼叫上需 20 秒的代理人，可透過使用更快的模型或並行執行模型呼叫來加速。

**成本（Costs）：** 每次代理人執行的費用是多少？AI 代理人仰賴按 token 計費的 LLM 呼叫或外部 API。頻繁使用工具或多次 prompt 會快速提高成本。例如，如果代理人為了微幅改善品質而呼叫 LLM 五次，就必須評估該成本是否合理，或是否能減少呼叫次數或使用較便宜的模型。即時監控也能幫助識別意外的尖峰（例如造成過度 API 迴圈的錯誤）。

**請求錯誤（Request Errors）：** 代理人失敗的請求數量是多少？這可以包含 API 錯誤或工具呼叫失敗。為了讓代理人在生產環境中更有韌性，您可以設定備援或重試機制。例如，如果 LLM 供應商 A 故障，您可以切換到 LLM 供應商 B 作為備援。

**使用者回饋（User Feedback）：** 實作直接的使用者評價會提供寶貴洞見。這可以包含明確評分（👍thumbs-up/👎down、⭐1-5 顆星）或文字評論。持續的負面回饋應該會警示您，因為這代表代理人未如預期運作。

**隱含使用者回饋（Implicit User Feedback）：** 即使沒有明確評分，使用者行為也會提供間接回饋。這可以包含立即重新措辭問題、重複查詢或點擊重試按鈕。例如，如果您看到使用者不斷重複問同一個問題，這表示代理人未如預期運作。

**準確性（Accuracy）：** 代理人產生正確或理想輸出的頻率如何？準確性的定義會有所不同（例如問題解決的正確性、資訊擷取的準確性、使用者滿意度）。第一步是定義代理人的成功標準。您可以透過自動檢查、評分或任務完成標記來追蹤準確性。例如，將追蹤標記為「成功」或「失敗」。

**自動化評估指標（Automated Evaluation Metrics）：** 您也可以設定自動評估。例如，使用 LLM 為代理人的輸出打分（如是否有幫助、是否準確等）。也有數個開源函式庫可幫助您為代理人的不同面向打分，例如用於 RAG 代理人的 [RAGAS](https://docs.ragas.io/) 或用於偵測有害語言或 prompt 注入的 [LLM Guard](https://llm-guard.com/)。

在實務上，這些指標的組合能為 AI 代理人的健康狀況提供最佳覆蓋。在本章的 [範例 notebook](./code_samples/10-expense_claim-demo.ipynb) 中，我們會示範這些指標在實際範例中的呈現，但首先，我們會了解典型的評估工作流程。

## 為代理人加入儀表化（Instrument your Agent）

要收集追蹤資料，您需要為程式碼加入儀表化。目標是讓代理人程式碼發出能被可觀察性平台捕獲、處理與視覺化的追蹤與指標。

**OpenTelemetry (OTel)：** [OpenTelemetry](https://opentelemetry.io/) 已成為 LLM 可觀察性的業界標準。它提供一組 API、SDK 與工具，用於產生、收集與匯出遙測資料。

有許多儀表化程式庫會包裝現有的代理人框架，並讓將 OpenTelemetry spans 匯出到可觀察性工具變得簡單。Microsoft Agent Framework 原生整合了 OpenTelemetry。以下是為 MAF 代理人加入儀表化的範例：

```python
from agent_framework.observability import get_tracer, get_meter

tracer = get_tracer()
meter = get_meter()

with tracer.start_as_current_span("agent_run"):
    # 代理執行會自動被追蹤
    pass
```

本章的 [範例 notebook](./code_samples/10-expense_claim-demo.ipynb) 將示範如何為您的 MAF 代理人加入儀表化。

**手動建立 Span（Manual Span Creation）：** 雖然儀表化程式庫提供了良好的基線，但常有需要更詳細或自訂資訊的情況。您可以手動建立 spans 以新增自訂應用邏輯。更重要的是，您可以用自訂屬性（也稱為標籤或元資料）來豐富自動或手動建立的 spans。這些屬性可以包含業務專屬資料、中間計算或任何對除錯或分析有用的上下文，例如 `user_id`、`session_id` 或 `model_version`。

以下為使用 [Langfuse Python SDK](https://langfuse.com/docs/sdk/python/sdk-v3) 手動建立追蹤與區段的範例：

```python
from langfuse import get_client
 
langfuse = get_client()
 
span = langfuse.start_span(name="my-span")
 
span.end()
```

## 代理人評估

可觀察性提供我們指標，但評估是分析那些資料（並執行測試）以判斷 AI 代理人表現如何及如何改進的過程。換句話說，當您擁有那些追蹤與指標時，如何利用它們來評估代理人並做出決策？

定期評估很重要，因為 AI 代理人通常具有非決定性，並可能隨著更新或模型行為漂移而演變——若沒有評估，您無法知道您的「智慧代理人」是否真正在做好它的工作或是否已經退化。

針對 AI 代理人有兩類評估：**線上評估（online evaluation）**與**離線評估（offline evaluation）**。兩者都很有價值，且互補。我們通常從離線評估開始，因為這是在部署任何代理人前的最低必要步驟。

### 離線評估

![Langfuse 的資料集項目](https://langfuse.com/images/cookbook/example-autogen-evaluation/example-dataset.png)

這涉及在受控環境中評估代理人，通常使用測試資料集，而非即時使用者查詢。您使用已策劃的資料集，知道期望輸出或正確行為，然後在這些資料上執行代理人。

例如，如果您建立了一個解數學文字題的代理人，您可能會有一個 100 題且答案已知的 [測試資料集](https://huggingface.co/datasets/gsm8k)。離線評估通常在開發期間進行（也可以成為 CI/CD 管線的一部分），以檢查改進或防止回歸。其優點是 **可重複，且因為有真實標準答案可以得到清晰的準確性指標**。您也可以模擬使用者查詢，並將代理人的回應與理想答案比較，或使用先前描述的自動化指標。

離線評估的主要挑戰是確保您的測試資料集具有全面性且保持相關性——代理人可能在固定的測試集上表現良好，但在生產中遇到非常不同的查詢。因此，您應該持續更新測試集，加入新的邊緣案例與反映真實世界場景的範例。混合小型「煙霧測試」案例與較大型的評估集會很有用：小型集合用於快速檢查，較大型則提供更廣泛的效能指標。

### 線上評估

![可觀察性指標總覽](https://langfuse.com/images/cookbook/example-autogen-evaluation/dashboard.png)

這指的是在真實環境中、也就是生產使用期間對代理人進行評估。線上評估包含監控代理人在真實使用者互動中的表現並持續分析結果。

例如，您可能會追蹤成功率、使用者滿意度分數或其他在實際流量上的指標。線上評估的優點是它 **能捕捉在實驗室中可能無法預見的情況**——您可以觀察到模型隨時間的漂移（若代理人的效能隨輸入模式改變而下降），並抓到測試資料中不存在的意外查詢或情境。它提供了一個代理人在實際運作中的真實圖景。

線上評估通常涉及收集隱含與明確的使用者回饋（如前述），並可能執行陰影測試或 A/B 測試（在平行環境中執行新版本代理人以與舊版比較）。挑戰是為即時互動取得可靠標籤或分數可能很困難——您可能得依賴使用者回饋或下游指標（例如使用者是否點擊結果）。

### 結合兩者

線上與離線評估並非互斥；它們高度互補。線上監控的洞見（例如代理人在某些新型查詢上的低表現）可用來增強並改進離線測試資料集。相反地，在離線測試中表現良好的代理人，可以更有信心地部署並在線上監控。

事實上，許多團隊採用一個循環：

_離線評估 -> 部署 -> 線上監控 -> 收集新失敗案例 -> 加入離線資料集 -> 精煉代理人 -> 重複_。

## 常見問題

當您將 AI 代理人部署到生產環境時，可能會遇到各種挑戰。以下是一些常見問題及其可能的解決方案：

| **問題**    | **可能的解決方案**   |
| ------------- | ------------------ |
| AI 代理人無法穩定地執行任務 | - 精煉給予 AI 代理人的 prompt；明確目標。<br>- 找出將任務拆分為子任務並由多個代理人處理是否有幫助。 |
| AI 代理人陷入連續迴圈  | - 確保有明確的終止條款與條件，讓代理人知道何時停止流程。<br>- 對於需要複雜推理與規劃的任務，使用專門用於推理任務的較大模型。 |
| AI 代理人呼叫工具表現不佳   | - 在代理人系統外部測試並驗證工具的輸出。<br>- 精煉所定義的參數、prompts 與工具命名。  |
| 多代理系統無法穩定運作 | - 精煉給予每個代理人的 prompt，確保它們具體且彼此區別。<br>- 建立層級式系統，使用「路由」或控制者代理人來決定哪個代理人最合適。 |

許多這類問題在部署可觀察性後可以更有效地被識別。我們先前討論的追蹤與指標有助於精確找出在代理人工作流程中哪個環節出現問題，讓除錯與優化變得更有效率。

## 管理成本
以下是一些管理將 AI 代理部署到生產環境成本的策略：

**使用較小的模型：** 小型語言模型 (SLMs) 在某些代理式使用情境中表現良好，並能顯著降低成本。如前所述，建立評估系統以決定並比較與較大型模型的效能，是了解 SLM 在你的使用案例中表現如何的最佳方式。考慮在較簡單的任務（如意圖分類或參數擷取）使用 SLMs，同時將較大型模型保留給複雜推理。

**使用路由模型：** 類似的策略是使用多樣化的模型與尺寸。你可以使用 LLM/SLM 或無伺服器函數，根據複雜度將請求路由到最適合的模型。這也有助於降低成本，同時確保在正確的任務上達到效能。例如，將簡單查詢路由到較小、較快速的模型，只有在複雜推理任務時才使用昂貴的大型模型。

**快取回應：** 識別常見請求與任務，並在它們通過你的代理式系統之前提供回應，是減少類似請求量的好方法。你甚至可以實作一個流程，使用較基礎的 AI 模型來判定一個請求與你的快取請求的相似程度。此策略能顯著降低常見問題或常見工作流程的成本。
