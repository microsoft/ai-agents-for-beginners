# Образцы квитанций

Три заранее сгенерированных файла квитанций для проверки без запуска ноутбука.

| File | Что это такое |
|---|---|
| `01_valid_receipt.json` | Действительная подписанная квитанция для вызова инструмента `lookup_flights`. Проверка возвращает True. |
| `02_tampered_receipt.json` | Та же квитанция с изменённым после подписания одним полем. Проверка возвращает False. |
| `03_chain_three_receipts.json` | Цепочка из трёх действительных квитанций (поиск, бронь, оформление) с `previous_receipt_hash`, связывающим каждую с предыдущей. |

## Проверка образцов

Ноутбук пошагово рассматривает проверку в четырёх разделах. Чтобы проверить эти образцы напрямую, не проходя через текст ноутбука:

```python
import json
from pathlib import Path

# Предполагается, что вы завершили импорт и вспомогательные функции
# из разделов 1 и 2 ноутбука 18-signed-receipts.ipynb.

valid = json.loads(Path("01_valid_receipt.json").read_text())
print(f"Valid receipt: {verify_receipt(valid)}")        # Истина

tampered = json.loads(Path("02_tampered_receipt.json").read_text())
print(f"Tampered receipt: {verify_receipt(tampered)}")  # Ложь

chain = json.loads(Path("03_chain_three_receipts.json").read_text())
for r in verify_chain(chain):
    print(f"  Receipt {r['index']} ({r['tool']}): {'VALID' if r['overall_valid'] else 'INVALID'}")
```

## Как они были сгенерированы

Образцы используют тот же код, что и ноутбук, с одним фиксированным ключом подписи
и фиксированными временными метками для воспроизводимости на уровне байт. Чтобы сгенерировать заново:

```bash
python3 generate_fixtures.py
```

(Скрипт находится в `generate_fixtures.py` в этом каталоге.)

## Что студенты узнают, изучая сырой JSON

Изучение формата сырой квитанции развивает интуицию, которой не всегда хватает в ячейках ноутбука. Студенты, которые просматривают JSON, часто замечают:

1. Подпись — это непрозрачная строка в формате base64url, но все остальные поля — обычный читаемый JSON. Подпись не шифрует содержимое; она его подтверждает.
2. `public_key` встроен в квитанцию. Аудитору ничего больше не требуется для проверки (при условии доверия к тому, что ключ действительно принадлежит заявленному издателю; см. README урока по инфраструктуре идентификации).
3. Изменение одного символа в любом поле и последующее сравнение этого файла с `02_tampered_receipt.json` делает механизм на уровне байтов более наглядным.
