# സാമ്പിൾ റിസീറ്റ് ഫിക്‌സ്ചേർസ്

നോട്ട്‌ബുക്ക് ഓടിക്കാതെ പരിശോധനയ്‌ക്കായി മൂന്ന് മുൻകൂട്ടി സൃഷ്ടിച്ച റിസീറ്റ് ഫയലുകൾ.

| ഫയൽ | എന്താണ് ഇത് |
|---|---|
| `01_valid_receipt.json` | `lookup_flights` ടൂൾ കോൾക്കുള്ള വാലിഡ് സൈൻ ചെയ്‌ത റിസീറ്റ്. വെരിഫിക്കേഷൻ True ആണ്. |
| `02_tampered_receipt.json` | ഒരെണ്ണം ഫീൽഡ് സൈൻ ചെയ്തതിന് ശേഷം മാറ്റിയ അതേ റിസീറ്റ്. വെരിഫിക്കേഷൻ False ആണ്. |
| `03_chain_three_receipts.json` | മൂന്ന് വാലിഡ് റിസീറ്റ്‌കളുടെ (search, hold, book) ഒരു ചെയിൻ, ഓരോന്നും മുൻവരവേറെ സെലക്ഷൻ ചെയ്യാൻ `previous_receipt_hash` ഉപയോഗിക്കുന്നു. |

## സാമ്പിളുകൾ സ്ഥിരീകരിക്കുന്നത്

നോട്ട്‌ബുക്ക് നാലു വിഭാഗങ്ങളായി വെരിഫിക്കേഷൻ വിശദീകരിക്കുന്നു. 
നോട്ട്‌ബുക്ക് നറുകഥ വഴി ഓടിക്കാതെ നേരിട്ട് ഈ ഫിക്‌സ്ച്ചറുകൾ\
നിർവഹിക്കാൻ:

```python
import json
from pathlib import Path

# നിങ്ങൾ ഇറക്കുമതികളും സഹായക ഫംഗ്ഷനുകളും പൂർത്തിയാക്കിയത് പരിഗണിക്കുന്നു
# 18-signed-receipts.ipynb ന്റെ സെക്ഷനുകൾ 1, 2 ൽ നിന്നുള്ള.

valid = json.loads(Path("01_valid_receipt.json").read_text())
print(f"Valid receipt: {verify_receipt(valid)}")        # സത്യം

tampered = json.loads(Path("02_tampered_receipt.json").read_text())
print(f"Tampered receipt: {verify_receipt(tampered)}")  # തെറ്റ്

chain = json.loads(Path("03_chain_three_receipts.json").read_text())
for r in verify_chain(chain):
    print(f"  Receipt {r['index']} ({r['tool']}): {'VALID' if r['overall_valid'] else 'INVALID'}")
```

## ഇവ എങ്ങനെ സൃഷ്ടിച്ചത്

ഫിക്‌സ്ച്ചറുകൾക്ക് കുറിച്ച് ഉപയോഗിക്കുന്നത് നോട്ട്‌ബുക്കിന്റെ അതേ കോഡ് പാതയാണ്, 
ഒരു നിശ്ചിത സൈൻ ചെയ്‌ത കീയും ബൈറ്റ് പുനരുപയോഗസാധ്യതാവകാശത്തോടെ നിശ്ചിത
ടൈംസ്റ്റാമ്പുകളും ഉപയോഗിച്ച്. വീണ്ടും സൃഷ്ടിക്കാൻ:

```bash
python3 generate_fixtures.py
```

(സ്ക്രിപ്റ്റ് ഈ ഡയറക്ടറിയിലെ `generate_fixtures.py` ന് കീഴിലാണ്.)

## വിദ്യാർത്ഥികൾ റോ JSON പരിശോധിക്കുന്നതിൽ നിന്ന് പഠിക്കുന്നതെന്ത്

റോ റിസീറ്റ് ഫോർമാറ്റ് വായിക്കുന്നത് നോട്ട്‌ബുക്കിലെ സെല്ലുകൾ എല്ലായ്പ്പോഴും നൽകാത്ത
അന്തരീക്ഷം വളർത്തുന്നു. JSON സ്ഖിമു് നോക്കുന്ന വിദ്യാർത്ഥികൾ പലപ്പോഴും കണ്ടു:

1. സിഗ്നേച്ചർ ഒരു അOpaque base64url സ്ട്രിംഗ് ആണ്, എന്നാൽ ബാക്കി എല്ലാ ഫീൽഡുകളും
   സാദ്ധ്യമായ JSON ആണ്. സിഗ്നേച്ചർ ഉള്ളടക്കം എൻക്രിപ്റ്റ് ചെയ്യുന്നില്ല; അതിനെക്കുറിച്ചു
   ഉറപ്പിക്കുന്നു.
2. `public_key` റിസീറ്റിൽ കുത്തിപ്പിടിച്ചിരിക്കുന്നു. ഒരു ഓഡിറ്റർമാർക്കു വേണ്ടിയത് മറ്റൊന്നുമില്ല,
   വെരിഫൈ ചെയ്യാൻ (കീ യഥാർത്ഥത്തിൽ കുറിച്ച വ്യക്തിക്ക് ചേർന്നതാണ് എന്നത് വിശ്വസിക്കുന്നതിനു
   പോരായ്മയിൽ; ഐഡന്റിറ്റി ഇന്‍ഫ്രാസ്ട്രക്ചർ-ലുള്ള പാഠ സമൃദ്ധി നോക്കുക).
3. ഒരു ഫീൽഡിലെ ഒരു ഒറ്റ അക്ഷരം മാറ്റി, ഈ ഫയൽ `02_tampered_receipt.json` ന്
   വേണ്ടി വീണ്ടും താരതമ്യം ചെയ്താൽ ബൈറ്റ് നിലവാര സംവിധാനത്തെ വെളിപ്പെടുത്തുന്നു.
