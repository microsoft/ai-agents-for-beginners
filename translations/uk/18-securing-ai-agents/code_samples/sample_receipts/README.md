# Приклади квитанцій

Три заздалегідь згенеровані файли квитанцій для ознайомлення без запуску блокнота.

| Файл | Що це |
|---|---|
| `01_valid_receipt.json` | Дійсна підписана квитанція для виклику інструменту `lookup_flights`. Перевірка повертає True. |
| `02_tampered_receipt.json` | Та сама квитанція з одним полем, зміненим після підпису. Перевірка повертає False. |
| `03_chain_three_receipts.json` | Ланцюжок з трьох дійсних квитанцій (пошук, утримання, бронювання) з `previous_receipt_hash`, що поєднує кожну з попередньою. |

## Перевірка прикладів

У блокноті перевірка розібрана на чотири розділи. Щоб перевірити ці приклади
безпосередньо, не проходячи наратив блокнота:

```python
import json
from pathlib import Path

# Припускає, що ви завершили імпорти та допоміжні функції
# з розділів 1 і 2 файлу 18-signed-receipts.ipynb.

valid = json.loads(Path("01_valid_receipt.json").read_text())
print(f"Valid receipt: {verify_receipt(valid)}")        # Правда

tampered = json.loads(Path("02_tampered_receipt.json").read_text())
print(f"Tampered receipt: {verify_receipt(tampered)}")  # Хиба

chain = json.loads(Path("03_chain_three_receipts.json").read_text())
for r in verify_chain(chain):
    print(f"  Receipt {r['index']} ({r['tool']}): {'VALID' if r['overall_valid'] else 'INVALID'}")
```

## Як це було згенеровано

Приклади використовують той самий код, що й блокнот, з одним фіксованим ключем підпису
та фіксованими часовими позначками для відтворюваності на рівні байтів. Щоб згенерувати знову:

```bash
python3 generate_fixtures.py
```

(Сценарій знаходиться у файлі `generate_fixtures.py` у цій директорії.)

## Чого навчаються студенти, переглядаючи сирий JSON

Читання сирого формату квитанції формує інтуїцію, якої часто бракує у комірках блокнота.
Студенти, які переглядають JSON, часто помічають:

1. Підпис — це непрозорий рядок у base64url, але всі інші поля — це звичайний
   читабельний JSON. Підпис не шифрує вміст; він його підтверджує.
2. `public_key` вбудований у квитанцію. Аудитору нічого більше не потрібно,
   щоб перевірити підпис (за умови довіри до того, що ключ дійсно належить заявленому
   видавцю; див. README уроку про інфраструктуру ідентичності).
3. Зміна навіть одного символу в будь-якому полі, а потім порівняння цього файлу з
   `02_tampered_receipt.json` робить роботу механізму на рівні байтів конкретною.
