## AIエージェントの定義とAIエージェントのタイプ

### AIエージェントとは？

簡単に言うと：

> **AIエージェントは、大規模言語モデル（LLM）が単にプロンプトに答えるだけでなく、ツールや知識を与えられて実際に「行動」できるシステムです。**

もう少し詳しく見てみましょう：

- <strong>システム</strong> — AIエージェントは単一のものではなく、複数のパーツが連携する仕組みです。基本的に、すべてのエージェントには3つの要素があります：
  - <strong>環境</strong> — エージェントが動作する空間です。旅行予約エージェントであれば、それは予約プラットフォームそのものです。
  - <strong>センサー</strong> — エージェントが環境の現在の状態を読み取る方法。旅行代理店は、ホテルの空室状況やフライトの料金をチェックします。
  - <strong>アクチュエーター</strong> — エージェントが行動をとる方法。旅行代理店は部屋を予約したり、確認メッセージを送信したり、予約をキャンセルしたりします。

![What Are AI Agents?](../../../translated_images/ja/what-are-ai-agents.1ec8c4d548af601a.webp)

- <strong>大規模言語モデル</strong> — エージェントはLLM以前から存在しましたが、現代のエージェントを強力にしているのはLLMです。自然言語の理解、コンテキストの推論、あいまいなユーザー要求を具体的な行動計画に変えられます。

- <strong>行動の実行</strong> — エージェントシステムがなければ、LLMはただテキストを生成するだけです。エージェントシステム内では、LLMが実際にステップを実行できます — データベース検索、API呼び出し、メッセージ送信など。

- <strong>ツールへのアクセス</strong> — 使えるツールは(1)エージェントが動作する環境と(2)開発者が与えた内容によります。旅行代理店はフライト検索はできても顧客記録の編集はできないことがあります — すべて配線次第です。

- **メモリ＋知識** — エージェントは短期記憶（現在の会話）と長期記憶（顧客データベースや過去のやりとり）を持つことができます。旅行代理店はあなたが窓側の席を好むことを「覚えている」かもしれません。

### AIエージェントの種類

すべてのエージェントが同じ構成ではありません。旅行予約のエージェント例を使って主なタイプを説明します：

| <strong>エージェントタイプ</strong> | <strong>機能</strong> | <strong>旅行代理店の例</strong> |
|---|---|---|
| <strong>単純反射エージェント</strong> | ハードコードされたルールに従う — メモリなし、計画なし。 | 苦情メールを見たら → カスタマーサポートに転送。それだけ。 |
| <strong>モデルベース反射エージェント</strong> | 世界の内部モデルを持ち、変化に応じて更新する。 | 過去のフライト料金を追跡し、急に高くなった経路を警告。 |
| <strong>目的志向エージェント</strong> | 目的を持ち、段階的に達成方法を決める。 | 現在地から目的地までの往復旅行（フライト、車、ホテル）を予約。 |
| <strong>効用志向エージェント</strong> | 1つの解決策ではなく、トレードオフを考慮して最善の解を見つける。 | 費用対便宜をバランスして、好みに合った最高の旅程を見つける。 |
| <strong>学習エージェント</strong> | フィードバックから学び、時間とともに改善する。 | 旅行後のアンケート結果に基づいて今後の予約推奨を調整。 |
| <strong>階層型エージェント</strong> | 上位エージェントが作業を小タスクに分割し、下位エージェントに委任。 | 「旅行キャンセル」依頼をフライトキャンセル、ホテルキャンセル、レンタカーキャンセルに分割。 |
| **マルチエージェントシステム（MAS）** | 複数の独立エージェントが協力/競争。 | 協力型：別々のエージェントがホテル、フライト、娯楽を担当。競争型：複数エージェントが最良価格のホテル予約で競う。 |

## いつAIエージェントを使うべきか

使えるからといって常にAIエージェントを使うべきとは限りません。エージェントが本当に効果的なのは次のような状況です：

![When to use AI Agents?](../../../translated_images/ja/when-to-use-ai-agents.54becb3bed74a479.webp)

- <strong>オープンエンドの問題</strong> — 問題解決のステップを事前にプログラムできない場合。LLMが動的に道筋を見つける必要があります。
- <strong>複数ステップのプロセス</strong> — 何度もツールを使いながら処理を進める必要があるタスク。単純な検索や生成ではない。
- <strong>時間経過による改善</strong> — ユーザーのフィードバックや環境の信号に応じてシステムを賢くしたい場合。

このコースの後半の<strong>信頼できるAIエージェントの構築</strong>レッスンで、いつ使うべきか（および使うべきでないか）をさらに詳しく扱います。

## エージェントソリューションの基本

### エージェント開発

エージェント構築でまず行うのは、「何ができるか」すなわちツール、行動、振る舞いを定義することです。

このコースでは主に<strong>Azure AI Agent Service</strong>をプラットフォームとして使います。以下をサポートしています：

- OpenAI、Mistral、Meta（Llama）などのプロバイダーのモデル
- Tripadvisorなどのプロバイダーのライセンス済みデータ
- 標準化されたOpenAPI 3.0ツール定義

### エージェントパターン

LLMとはプロンプトを使ってやりとりします。エージェントでは、多くの場合すべてのプロンプトを手作業で作るのは難しく、多段階のアクションを自動化する必要があります。そこで<strong>エージェントパターン</strong>です。これは、LLMをよりスケーラブルかつ信頼性高く動かすための再利用可能な戦略です。

本コースは最も一般的かつ有用なエージェントパターンに基づいて構成されています。

### エージェントフレームワーク

エージェントフレームワークは、開発者にエージェント構築用のテンプレート、ツール、インフラを提供します。これにより、

- ツールや機能の接続が簡単に
- エージェントの動作観察（問題発生時のデバッグ）が可能に
- 複数のエージェント間での協力が容易に

本コースでは、プロダクションレベルのエージェント構築に向けて<strong>Microsoft Agent Framework (MAF)</strong>に焦点を当てています。
