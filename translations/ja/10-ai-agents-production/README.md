# AIエージェントの本番運用：可観測性と評価

[![AIエージェントの本番運用](../../../translated_images/ja/lesson-10-thumbnail.2b79a30773db093e.webp)](https://youtu.be/l4TP6IyJxmQ?si=reGOyeqjxFevyDq9)

実験的なプロトタイプから実際のアプリケーションへと移行するにつれて、エージェントの振る舞いを理解し、パフォーマンスを監視し、出力を体系的に評価する能力が重要になります。

## 学習目標

このレッスンを修了すると、以下を理解／実行できるようになります：
- エージェントの可観測性と評価のコア概念
- エージェントのパフォーマンス、コスト、効果を改善する手法
- AIエージェントを体系的に何を、どのように評価するか
- AIエージェントを本番にデプロイする際のコスト管理方法
- Microsoft Agent Frameworkで構築したエージェントの計測方法

目標は、あなたの「ブラックボックス」的なエージェントを透明で管理可能、信頼できるシステムへと変えるための知識を提供することです。

_**注：** 安全で信頼できるAIエージェントをデプロイすることが重要です。こちらのレッスンも参照してください：[信頼できるAIエージェントの構築](./06-building-trustworthy-agents/README.md)。_

## トレースとスパン

[Langfuse](https://langfuse.com/) や [Microsoft Foundry](https://learn.microsoft.com/en-us/azure/ai-foundry/what-is-azure-ai-foundry) のような可観測性ツールは、エージェントの実行を通常トレースとスパンとして表現します。

- **トレース** は、ユーザーの質問処理のような開始から終了までのエージェントの完全なタスクを表します。
- **スパン** は、トレース内の個々のステップ（言語モデルの呼び出しやデータ取得など）を表します。

![Langfuse のトレースツリー](https://langfuse.com/images/cookbook/example-autogen-evaluation/trace-tree.png)
<!-- Image URL retained for illustration purposes -->

可観測性がないと、AIエージェントは「ブラックボックス」のように感じられ、内部状態や推論が不透明で問題の診断や性能最適化が困難になります。可観測性があれば、エージェントは「ガラスの箱」のように透明になり、信頼構築や期待通りに動作していることを確認するために不可欠です。

## 本番環境で可観測性が重要な理由

AIエージェントを本番環境に移行すると、新たな課題と要件が生じます。可観測性はもはや「あると良い」機能ではなく、重要な能力です：

*   **デバッグと根本原因分析**：エージェントが失敗したり予期しない出力を生成した場合、可観測性ツールはエラーの原因を特定するためのトレースを提供します。複数のLLM呼び出し、ツールの相互作用、条件付きロジックを含む複雑なエージェントでは特に重要です。
*   **待ち時間とコスト管理**：AIエージェントはトークンや呼び出しごとに課金されるLLMや外部APIに依存することが多いです。可観測性によりこれらの呼び出しを正確に追跡でき、過度に遅いまたは高コストな操作を特定できます。これによりプロンプトの最適化、より効率的なモデルの選定、ワークフローの再設計などで運用コストを管理し、良好なユーザー体験を確保できます。
*   **信頼性・安全性・準拠**：多くのアプリケーションでは、エージェントが安全かつ倫理的に振る舞うことを保証する必要があります。可観測性はエージェントの行動と決定の監査トレイルを提供します。これによりプロンプトインジェクション、有害なコンテンツの生成、個人を特定できる情報（PII）の不適切な取り扱いなどの問題を検出・軽減できます。例えば、ある応答や特定のツール使用の理由を理解するためにトレースを確認できます。
*   **継続的改善ループ**：可観測性データは反復的な開発プロセスの基盤です。本番でのエージェントの挙動を監視することで、改善点を特定し、モデルのファインチューニングや変更の影響を検証するためのデータを収集できます。これにより、本番から得られたインサイトをオフライン実験に反映させるフィードバックループが形成され、エージェントの性能が段階的に向上します。

## 追跡すべき主要な指標

エージェントの振る舞いを監視・理解するには、さまざまな指標やシグナルを追跡する必要があります。具体的な指標はエージェントの目的によって異なりますが、普遍的に重要なものがあります。

ここでは、可観測性ツールが監視する代表的な指標をいくつか紹介します：

**レイテンシ（遅延）:** エージェントはどれくらい速く応答するか？長い待機時間はユーザー体験に悪影響を与えます。トレースによりタスクや個々のステップのレイテンシを測定すべきです。例えば、全てのモデル呼び出しに20秒かかるエージェントは、より速いモデルの使用やモデル呼び出しの並列実行で高速化できるかもしれません。

**コスト:** エージェント1回あたりの費用は？AIエージェントはトークン単位や外部APIの呼び出しごとに課金されるLLM呼び出しに依存します。頻繁なツール利用や複数のプロンプトはコストを急速に増加させる可能性があります。例えば、エージェントが品質向上のためにLLMを5回呼び出す場合、そのコストが正当化されるか、呼び出し回数を減らすか、より安価なモデルを使えるかを評価する必要があります。リアルタイム監視は、予期しないスパイク（例：過剰なAPIループを引き起こすバグ）を検出するのにも役立ちます。

**リクエストエラー:** エージェントはどれくらいリクエストに失敗したか？これにはAPIエラーやツール呼び出しの失敗が含まれます。本番でこれらに対してエージェントを堅牢にするために、フォールバックやリトライを設定できます。例：LLMプロバイダAがダウンした場合、バックアップとしてLLMプロバイダBに切り替える。

**ユーザーフィードバック:** 直接的なユーザー評価を実装すると貴重なインサイトが得られます。これには明示的な評価（👍thumbs-up/👎down、⭐1-5 stars）やテキストコメントが含まれます。一貫したネガティブなフィードバックは、エージェントが期待どおりに動作していないサインです。

**暗黙のユーザーフィードバック:** ユーザー行動は明示的な評価がなくても間接的なフィードバックを提供します。これには即時の質問の言い換え、繰り返しのクエリ、再試行ボタンのクリックなどが含まれます。例えば、ユーザーが同じ質問を繰り返しているのを観察したら、それはエージェントが期待どおりに機能していない兆候です。

**正確性（Accuracy）:** エージェントはどのくらい頻繁に正しいまたは望ましい出力を生成するか？正確性の定義は用途により異なります（問題解決の正確性、情報検索の精度、ユーザー満足度など）。まずはエージェントにとっての成功が何かを定義することが重要です。自動チェック、評価スコア、タスク完了ラベルなどで正確性を追跡できます。例えば、トレースに「succeeded」や「failed」を付けるなどです。

**自動評価指標:** 自動評価を設定することもできます。例えば、LLMを使ってエージェントの出力が有用か、正確かをスコアリングすることが可能です。また、エージェントのさまざまな側面をスコアリングするのに役立つオープンソースライブラリもいくつかあります。例：RAGエージェント向けの [RAGAS](https://docs.ragas.io/) や、有害な言語やプロンプトインジェクションを検出する [LLM Guard](https://llm-guard.com/)。

実務では、これらの指標を組み合わせることでエージェントの健全性を最も包括的に把握できます。この章の[例のノートブック](./code_samples/10-expense_claim-demo.ipynb)では、これらの指標が実際の例でどのように見えるかを示しますが、まず典型的な評価ワークフローを学びましょう。

## エージェントの計測（Instrument your Agent）

トレースデータを収集するには、コードに計測を組み込む必要があります。目的は、可観測性プラットフォームがキャプチャ、処理、可視化できるトレースとメトリクスをエージェントコードが出力するように計測することです。

**OpenTelemetry (OTel):** [OpenTelemetry](https://opentelemetry.io/) はLLM可観測性の業界標準として台頭しています。テレメトリデータの生成、収集、エクスポートのためのAPI、SDK、ツール群を提供します。

多くの計測ライブラリが既存のエージェントフレームワークをラップして、OpenTelemetryのスパンを可観測性ツールへ簡単にエクスポートできるようにします。Microsoft Agent FrameworkはOpenTelemetryとネイティブに統合されています。以下はMAFエージェントを計測する例です：

```python
from agent_framework.observability import get_tracer, get_meter

tracer = get_tracer()
meter = get_meter()

with tracer.start_as_current_span("agent_run"):
    # エージェントの実行は自動的に追跡されます
    pass
```

この章の[例のノートブック](./code_samples/10-expense_claim-demo.ipynb)では、MAFエージェントの計測方法を実演します。

**手動スパン作成（Manual Span Creation）:** 計測ライブラリは良いベースラインを提供しますが、より詳細またはカスタム情報が必要なケースがよくあります。カスタムアプリケーションロジックを追加するためにスパンを手動で作成できます。さらに重要なのは、自動または手動で作成したスパンにカスタム属性（タグやメタデータとも呼ばれる）を付与して拡充できる点です。これらの属性には、`user_id`、`session_id`、`model_version` のようなビジネス固有のデータ、中間計算、デバッグや解析に有用なコンテキストを含めることができます。

[Langfuse Python SDK](https://langfuse.com/docs/sdk/python/sdk-v3) を使ってトレースとスパンを手動で作成する例：

```python
from langfuse import get_client
 
langfuse = get_client()
 
span = langfuse.start_span(name="my-span")
 
span.end()
```

## エージェント評価

可観測性は指標を提供しますが、評価はそのデータ（およびテストの実行）を分析して、AIエージェントがどれだけうまく機能しているか、どのように改善できるかを判断するプロセスです。つまり、トレースと指標を取得した後、それらをどのように使ってエージェントを評価し意思決定を行うか、ということです。

AIエージェントはしばしば非決定論的で更新やモデル挙動のドリフトによって変化するため、定期的な評価は重要です。評価がなければ、あなたの「スマートなエージェント」が本当に期待通りに機能しているか、あるいは性能が低下していないかを知ることができません。

AIエージェントの評価には主に2つのカテゴリがあります：**オンライン評価** と **オフライン評価**。どちらも価値があり、互いを補完します。通常はオフライン評価から始めます。これはエージェントをデプロイする前に最低限必要なステップです。

### オフライン評価

![Langfuse のデータセット項目](https://langfuse.com/images/cookbook/example-autogen-evaluation/example-dataset.png)

これは、通常テストデータセットを用いた制御された環境でエージェントを評価することを指し、ライブのユーザー問い合わせは含みません。期待される出力や正しい振る舞いが分かっているキュレートされたデータセットを使ってエージェントを実行します。

例えば、数学の文章題エージェントを作った場合、既知の解答がある100問の[test dataset](https://huggingface.co/datasets/gsm8k) を持っているかもしれません。オフライン評価は開発中に行われることが多く（CI/CDパイプラインの一部にもなり得ます）、改善の確認やリグレッションの防止に使われます。利点は、**再現可能であり、グラウンドトゥルースがあるため明確な精度指標を得られる**点です。シミュレートしたユーザー問い合わせに対してエージェントの応答を理想的な回答と比較したり、前述の自動指標を用いることもできます。

オフライン評価の主な課題は、テストデータセットが包括的であり続けることを保証することです——固定のテストセットでうまく動作していても、本番では非常に異なるクエリに遭遇する可能性があります。したがって、テストセットを新たなエッジケースや実世界のシナリオを反映する例で更新し続けるべきです。小さな「スモークテスト」ケースと大規模な評価セットを混ぜるのが有用です：小さなセットは迅速なチェック用、大きなセットはより広範な性能指標用です。

### オンライン評価

![可観測性指標の概要](https://langfuse.com/images/cookbook/example-autogen-evaluation/dashboard.png)

これは実際の使用、本番環境でエージェントを評価することを指します。オンライン評価は実際のユーザーインタラクションでのエージェントの性能を継続的に監視・分析することを含みます。

例えば、成功率、ユーザー満足度スコア、その他のメトリクスをライブトラフィックで追跡するかもしれません。オンライン評価の利点は、**ラボ設定では予期できない事象を捉えられること**です——入力パターンの変化によるモデルドリフト（時間とともにエージェントの有効性が低下する場合）を観察したり、テストデータにはなかった予期しないクエリや状況を検出できます。本番での挙動の実像を得られます。

オンライン評価では、前述のように暗黙的および明示的なユーザーフィードバックを収集したり、シャドウテストやA/Bテスト（新バージョンのエージェントを並列で実行して旧バージョンと比較する）を実行することもあります。課題は、ライブインタラクションに対して信頼できるラベルやスコアを得るのが難しい場合があることです——ユーザーフィードバックや下流メトリクス（例：ユーザーが結果をクリックしたかどうか）に依存することが多いです。

### 両者の組み合わせ

オンライン評価とオフライン評価は排他的ではなく、非常に相互補完的です。オンライン監視から得られたインサイト（例：エージェントが苦手とする新しいタイプのユーザー問い合わせ）は、オフラインのテストデータセットを拡充・改善するために使えます。逆に、オフラインテストで良好な性能を示したエージェントは、より自信を持って本番にデプロイし、オンラインで監視できます。

多くのチームは次のようなループを採用します：

_オフラインで評価 -> デプロイ -> オンラインで監視 -> 新しい失敗ケースを収集 -> オフラインデータセットに追加 -> エージェントを改良 -> 繰り返し_。

## よくある問題

AIエージェントを本番にデプロイすると、さまざまな課題に直面することがあります。以下は一般的な問題とその可能な解決策です：

| **Issue**    | **Potential Solution**   |
| ------------- | ------------------ |
| AI Agent not performing tasks consistently | - プロンプトを洗練し、目的を明確にする。<br>- タスクをサブタスクに分割し、複数のエージェントで処理することで改善できる箇所を特定する。 |
| AI Agent running into continuous loops  | - エージェントがいつプロセスを終了すべきかの明確な終了条件を設ける。<br>- 推論や計画が必要な複雑なタスクには、推論向けに特化したより大きなモデルを使用する。 |
| AI Agent tool calls are not performing well   | - エージェントシステムの外でツールの出力をテスト・検証する。<br>- 定義されたパラメータ、プロンプト、ツール名を洗練する。  |
| Multi-Agent system not performing consistently | - 各エージェントに与えるプロンプトを具体的かつ互いに区別されるように改善する。<br>- どのエージェントが適切かを決定する「ルーティング」またはコントローラーエージェントを使って階層的なシステムを構築する。 |

これらの多くの問題は、可観測性を導入することでより効果的に特定できます。先に述べたトレースと指標は、エージェントのワークフローのどの部分で問題が発生しているかを正確に特定するのに役立ち、デバッグや最適化を大幅に効率化します。

## コスト管理
Here are some strategies to manage the costs of deploying AI agents to production:

**Using Smaller Models:** 小規模言語モデル（Small Language Models、SLMs）は特定のエージェント的なユースケースで良好な性能を発揮し、コストを大幅に削減できます。前述のように、性能を評価して大型モデルと比較する評価システムを構築することが、SLMがあなたのユースケースでどれだけ有効かを理解する最良の方法です。意図分類やパラメータ抽出のようなより単純なタスクにはSLMを検討し、複雑な推論には大型モデルを使うことを検討してください。

**Using a Router Model:** 類似の戦略として、様々なモデルサイズを使い分ける方法があります。複雑さに応じてリクエストを最適なモデルにルーティングするために、LLM/SLMやサーバーレス関数を使用できます。これによりコストを削減しつつ、適切なタスクでの性能を確保できます。例えば、単純なクエリはより小さく高速なモデルに振り分け、複雑な推論タスクには高価な大型モデルのみを使用するようにします。

**Caching Responses:** よくあるリクエストやタスクを特定して、エージェントシステムを通す前にレスポンスを返すことは、類似リクエストの量を減らす良い方法です。より基本的なAIモデルを使ってリクエストがキャッシュ済みのリクエストとどれほど類似しているかを判定するフローを実装することもできます。この戦略は、よくある質問や一般的なワークフローに対してコストを大幅に削減できます。
