# AI 代理在生产环境：可观测性与评估

[![AI 代理在生产环境](../../../translated_images/zh-CN/lesson-10-thumbnail.2b79a30773db093e.webp)](https://youtu.be/l4TP6IyJxmQ?si=reGOyeqjxFevyDq9)

随着 AI 代理从实验原型走向真实世界的应用，理解其行为、监控其性能并系统性地评估其输出变得非常重要。

## 学习目标

完成本课后，您将知道/理解：
- 代理可观测性与评估的核心概念
- 提高代理性能、成本和有效性的技术
- 如何系统性地评估您的 AI 代理以及评估什么
- 在将 AI 代理部署到生产时如何控制成本
- 如何为使用 Microsoft Agent Framework 构建的代理添加监测埋点

目标是让您具备将“黑盒”代理转变为透明、易管理且可靠系统的知识。

_**注意：** 部署安全且值得信赖的 AI 代理非常重要。也请查看[构建值得信赖的 AI 代理](./06-building-trustworthy-agents/README.md)这一课。_

## 跟踪与跨度

可观测性工具（例如 [Langfuse](https://langfuse.com/) 或 [Microsoft Foundry](https://learn.microsoft.com/en-us/azure/ai-foundry/what-is-azure-ai-foundry)）通常将代理运行表示为跟踪（traces）和跨度（spans）。

- **Trace（跟踪）** 表示从开始到结束的完整代理任务（例如处理一个用户查询）。
- **Spans（跨度）** 是跟踪中的各个步骤（例如调用语言模型或检索数据）。

![Langfuse 中的跟踪树](https://langfuse.com/images/cookbook/example-autogen-evaluation/trace-tree.png)
<!-- Image URL retained for illustration purposes -->

没有可观测性，AI 代理会像一个“黑盒”——其内部状态和推理不透明，难以诊断问题或优化性能。有了可观测性，代理就变成了“透明盒子”，提供了构建信任并确保其按预期运行所需的透明度。

## 为什么可观测性在生产环境中很重要

将 AI 代理迁移到生产环境会带来一系列新的挑战和需求。可观测性不再是“可有可无”，而是一项关键能力：

*   **调试与根因分析：** 当代理失败或产生意外输出时，可观测性工具提供了定位错误来源所需的跟踪。这在可能涉及多个 LLM 调用、工具交互和条件逻辑的复杂代理中尤为重要。
*   **延迟与成本管理：** AI 代理通常依赖按 token 或按调用计费的 LLM 和其他外部 API。可观测性允许精确跟踪这些调用，帮助识别异常慢或过于昂贵的操作。这使团队能够优化提示、选择更高效的模型，或重新设计工作流以管理运营成本并确保良好的用户体验。
*   **信任、安全与合规：** 在许多应用中，确保代理行为安全且合乎伦理非常重要。可观测性提供了代理行为和决策的审计轨迹。这可用于检测和缓解诸如提示注入（prompt injection）、有害内容生成或对个人身份信息（PII）的错误处理等问题。例如，您可以审查跟踪以了解代理为何提供某个响应或使用特定工具。
*   **持续改进循环：** 可观测性数据是迭代开发过程的基础。通过监控代理在真实世界中的表现，团队可以识别改进点、收集用于微调模型的数据并验证变更的影响。这创造了一个反馈环：在线评估中的生产洞察反过来指导离线实验和改进，从而逐步提升代理性能。

## 需要跟踪的关键指标

为了监控和理解代理行为，应跟踪一系列指标和信号。具体指标可能根据代理用途有所不同，但有些指标是普遍重要的。

以下是可观测性工具通常监控的一些常见指标：

**延迟：** 代理响应有多快？长时间等待会对用户体验产生负面影响。您应通过跟踪代理运行来测量任务和各个步骤的延迟。例如，如果一个代理的所有模型调用总共需要 20 秒，则可以通过使用更快速的模型或并行运行模型调用来加速。

**成本：** 每次代理运行的费用是多少？AI 代理依赖按 token 或外部 API 调用计费的 LLM。频繁使用工具或多次提示会迅速增加成本。例如，如果一个代理为了边际质量提升而调用 LLM 五次，您必须评估这是否值得，或者是否可以减少调用次数或使用更便宜的模型。实时监控还可以帮助识别意外峰值（例如导致过多 API 循环的 bug）。

**请求错误：** 代理失败了多少请求？这可能包括 API 错误或工具调用失败。为了让您的代理在生产环境中更稳健，您可以设置回退或重试机制。例如，如果 LLM 提供商 A 宕机，您可以切换到 LLM 提供商 B 作为备用。

**用户反馈：** 实施直接的用户评估可提供有价值的见解。这可以包括显式评分（👍thumbs-up/👎down、⭐1-5 星）或文本评论。持续的负面反馈应引起警觉，因为这表明代理未按预期工作。

**隐式用户反馈：** 即使没有显式评分，用户行为也会提供间接反馈。这可以包括立即重述问题、重复查询或点击重试按钮。例如，如果您看到用户反复问同一个问题，这就是代理未按预期工作的信号。

**准确性：** 代理产生正确或期望输出的频率如何？准确性的定义会因场景而异（例如问题解决的正确性、信息检索的准确性、用户满意度）。第一步是定义代理成功的标准。您可以通过自动检查、评估分数或任务完成标签来跟踪准确性。例如，将跟踪标记为“成功”或“失败”。

**自动化评估指标：** 您还可以设置自动评估。例如，可以使用 LLM 对代理的输出进行评分，例如判断其是否有帮助、是否准确等。也有若干开源库可帮助对代理的不同方面进行评分，例如用于 RAG 代理的 [RAGAS](https://docs.ragas.io/) 或用于检测有害语言或提示注入的 [LLM Guard](https://llm-guard.com/)。

在实践中，这些指标的组合能最好地覆盖 AI 代理的健康状况。在本章的 [示例笔记本](./code_samples/10-expense_claim-demo.ipynb) 中，我们将展示这些指标在真实示例中的样子，但首先，我们将了解典型的评估工作流是怎样的。

## 为你的代理添加监测埋点

要收集跟踪数据，您需要为代码添加埋点。目标是对代理代码进行埋点，使其发出可被可观测性平台捕获、处理和可视化的跟踪和指标。

**OpenTelemetry (OTel)：** [OpenTelemetry](https://opentelemetry.io/) 已成为 LLM 可观测性的行业标准。它提供了一组用于生成、收集和导出遥测数据的 API、SDK 和工具。

有许多封装现有代理框架并便于将 OpenTelemetry 跨度导出到可观测性工具的埋点库。Microsoft Agent Framework 原生集成了 OpenTelemetry。下面是为 MAF 代理添加埋点的示例：

```python
from agent_framework.observability import get_tracer, get_meter

tracer = get_tracer()
meter = get_meter()

with tracer.start_as_current_span("agent_run"):
    # 代理执行会被自动跟踪
    pass
```

本章的 [示例笔记本](./code_samples/10-expense_claim-demo.ipynb) 将演示如何为您的 MAF 代理添加埋点。

**手动创建跨度：** 虽然埋点库提供了良好的基线，但通常会遇到需要更详细或自定义信息的情况。您可以手动创建跨度以添加自定义的应用逻辑。更重要的是，您可以为自动或手动创建的跨度添加自定义属性（也称为标签或元数据）。这些属性可以包含业务特定的数据、中间计算结果或任何可能对调试或分析有用的上下文，例如 `user_id`、`session_id` 或 `model_version`。

使用 [Langfuse Python SDK](https://langfuse.com/docs/sdk/python/sdk-v3) 手动创建跟踪和跨度的示例：

```python
from langfuse import get_client
 
langfuse = get_client()
 
span = langfuse.start_span(name="my-span")
 
span.end()
```

## 代理评估

可观测性为我们提供了指标，而评估则是分析这些数据（并执行测试）以确定 AI 代理的表现如何以及如何改进的过程。换句话说，一旦您拥有这些跟踪和指标，如何使用它们来判断代理并做出决策？

定期评估很重要，因为 AI 代理通常是非确定性的并且会演变（通过更新或模型行为漂移）——如果没有评估，您不会知道您的“智能代理”是否真正履行其职责或是否出现回归。

对 AI 代理的评估分为两类：**在线评估**和**离线评估**。两者都很有价值，彼此互补。我们通常从离线评估开始，因为这是在部署任何代理之前的最低必要步骤。

### 离线评估

![Langfuse 中的数据集项](https://langfuse.com/images/cookbook/example-autogen-evaluation/example-dataset.png)

这涉及在受控环境中评估代理，通常使用测试数据集，而不是实时用户查询。您使用经过策划的数据集，其中已知预期输出或正确行为，然后在这些数据集上运行代理。

例如，如果您构建了一个数学文字题代理，您可能有一个包含 100 道已知答案的问题的[测试数据集](https://huggingface.co/datasets/gsm8k)。离线评估通常在开发过程中进行（并且可以作为 CI/CD 管道的一部分）以检查改进或防止回归。其好处是**可重复且因为存在真实答案而可以获得清晰的准确性指标**。您还可以模拟用户查询并将代理的响应与理想答案进行比较，或使用上文描述的自动化指标。

离线评估的关键挑战是确保您的测试数据集是全面且保持相关的——代理可能在固定测试集上表现良好，但在生产中遇到截然不同的查询。因此，您应及时更新测试集，加入反映真实世界场景的新边缘案例和示例。混合使用小型“冒烟测试”用例与更大规模的评估集是有益的：小集用于快速检查，大集用于更广泛的性能指标。

### 在线评估

![可观测性指标概览](https://langfuse.com/images/cookbook/example-autogen-evaluation/dashboard.png)

这指的是在实时、真实的环境中评估代理，即在生产中的实际使用期间。在线评估涉及监控代理在真实用户交互中的表现并持续分析结果。

例如，您可能会跟踪成功率、用户满意度评分或实时流量的其他指标。在线评估的优点是它**捕捉到您在实验室环境中可能无法预见的情况**——您可以观察随时间发生的模型漂移（如果输入模式变化导致代理效果下降）并捕获测试数据中未出现的意外查询或情形。它提供了代理在真实环境中行为的真实图景。

在线评估通常涉及收集隐式和显式用户反馈（如前所述），并可能运行影子测试或 A/B 测试（在并行运行新版本代理以与旧版本比较）。挑战在于为实时交互获取可靠标签或评分可能很困难——您可能需要依赖用户反馈或下游指标（例如用户是否点击结果）。

### 两者结合

在线和离线评估并不相互排斥；它们高度互补。来自在线监控的洞察（例如代理在某类新查询上表现不佳）可以用于扩充和改进离线测试数据集。反之，在离线测试中表现良好的代理可以更有把握地部署并在上线后进行监控。

事实上，许多团队采用如下循环：

_离线评估 -> 部署 -> 在线监控 -> 收集新的失败案例 -> 添加到离线数据集 -> 优化代理 -> 重复_。

## 常见问题

在将 AI 代理部署到生产时，您可能会遇到各种挑战。以下是一些常见问题及其潜在解决方案：

| **问题**    | **潜在解决方案**   |
| ------------- | ------------------ |
| AI Agent not performing tasks consistently | - 精炼给 AI 代理的提示；明确目标。<br>- 识别将任务拆分为子任务并由多个代理处理的场景。 |
| AI Agent running into continuous loops  | - 确保有明确的终止条件，让代理知道何时停止流程。<br>- 对于需要推理与规划的复杂任务，使用在推理方面更擅长的大型模型。 |
| AI Agent tool calls are not performing well   | - 在代理系统之外测试并验证工具的输出。<br>- 精炼工具的参数、提示和命名。  |
| Multi-Agent system not performing consistently | - 优化分配给每个代理的提示，确保它们具体且彼此不同。<br>- 构建一个分层系统，使用 "routing" 或控制器代理来确定哪个代理是正确的。 |

通过部署可观测性，许多这些问题可以更有效地被识别。我们之前讨论的跟踪和指标有助于精确定位代理工作流中出现问题的位置，使调试和优化更加高效。

## 成本管理
以下是一些在将 AI 代理部署到生产环境时管理成本的策略：

**使用更小的模型：** 小型语言模型 (SLMs) 在某些代理化用例上表现良好，并能显著降低成本。正如之前提到的，构建一个评估系统来确定并比较其与更大型模型的性能，是了解 SLM 在你的用例中表现如何的最佳方式。考虑在更简单的任务（如意图分类或参数提取）中使用 SLM，而将更大的模型保留用于复杂推理。

**使用路由模型：** 类似的策略是使用不同种类和规模的模型。你可以使用 LLM/SLM 或无服务器函数，根据请求的复杂度将其路由到最合适的模型。这不仅有助于降低成本，还能确保在合适的任务上获得良好性能。例如，将简单查询路由到更小更快的模型，只有在复杂推理任务时才使用昂贵的大模型。

**缓存响应：** 识别常见请求和任务，并在它们进入你的代理系统之前提供响应，是减少类似请求数量的好方法。你甚至可以实现一个流程，使用更基础的 AI 模型来判断某个请求与已缓存请求的相似程度。对于常见问题或常用工作流，这种策略可以显著降低成本。
