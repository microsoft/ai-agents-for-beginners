# ماڈل رسید کے نمونے

نوٹ بک چلائے بغیر معائنہ کے لئے تین پہلے سے تیار شدہ رسید فائلیں۔

| فائل | یہ کیا ہے |
|---|---|
| `01_valid_receipt.json` | `lookup_flights` ٹول کال کے لئے ایک درست دستخط شدہ رسید۔ توثیق نتیجہ True آتا ہے۔ |
| `02_tampered_receipt.json` | دستخط کے بعد ایک فیلڈ میں ترمیم کی گئی وہی رسید۔ توثیق نتیجہ False آتا ہے۔ |
| `03_chain_three_receipts.json` | تین درست رسیدوں کا سلسلہ (search, hold, book) جس میں `previous_receipt_hash` ہر رسید کو پچھلی سے جوڑتا ہے۔ |

## نمونوں کی توثیق

نوٹ بک چار حصوں میں توثیق کی وضاحت کرتی ہے۔ ان فکسچرز کو براہ راست نوٹ بک کی وضاحت پڑھے بغیر ویریفائی کرنے کے لیے:

```python
import json
from pathlib import Path

# فرض کرتا ہے کہ آپ نے درآمدات اور مددگار افعال مکمل کر لئے ہیں
# سیکشن 1 اور 2 سے 18-signed-receipts.ipynb کی

valid = json.loads(Path("01_valid_receipt.json").read_text())
print(f"Valid receipt: {verify_receipt(valid)}")        # درست

tampered = json.loads(Path("02_tampered_receipt.json").read_text())
print(f"Tampered receipt: {verify_receipt(tampered)}")  # غلط

chain = json.loads(Path("03_chain_three_receipts.json").read_text())
for r in verify_chain(chain):
    print(f"  Receipt {r['index']} ({r['tool']}): {'VALID' if r['overall_valid'] else 'INVALID'}")
```

## یہ کیسے بنائے گئے

فکسچرز اسی کوڈ راستے کو استعمال کرتے ہیں جو نوٹ بک استعمال کرتی ہے، ایک مقررہ دستخطی کلید اور بایٹ دوبارہ پیدا کرنے کے لئے مقررہ وقت کے ساتھ۔ دوبارہ بنانے کے لئے:

```bash
python3 generate_fixtures.py
```

(اسکرپٹ اس ڈائریکٹری میں `generate_fixtures.py` پر ہے۔)

## طلباء کو خام JSON کا معائنہ کر کے کیا سیکھنے کو ملتا ہے

خام رسید فارمیٹ پڑھنا وہ بصیرت فراہم کرتا ہے جو نوٹ بک کی سیلز ہمیشہ فراہم نہیں کرتیں۔ وہ طلباء جو JSON کو جلدی دیکھتے ہیں اکثر محسوس کرتے ہیں:

1. دستخط ایک مبہم base64url سٹرنگ ہے، لیکن باقی ہر فیلڈ صاف پڑھنے والا JSON ہے۔ دستخط مواد کو انکرپٹ نہیں کرتا؛ یہ اس کی تصدیق کرتا ہے۔
2. `public_key` رسید میں شامل ہے۔ ایک آڈیٹر کو توثیق کے لئے اور کچھ نہیں چاہیے (بشرطیکہ یقین ہو کہ یہ کلید واقعی دعویٰ کنندہ جاری کنندہ کی ہے؛ شناختی انفراسٹرکچر کے لیے اسباق کا README دیکھیں)۔
3. کسی بھی فیلڈ کے ایک حرف میں ترمیم کریں، پھر اس فائل کو `02_tampered_receipt.json` سے دوبارہ موازنہ کریں، تو یہ بائٹ سطح کا میکانزم واضح ہوجاتا ہے۔
