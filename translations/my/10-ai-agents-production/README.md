# ထုတ်လုပ်မှုရှိ AI အေဂျင့်များ: ကြည့်ရှုနိုင်ခြင်းနှင့် အကဲဖြတ်ခြင်း

[![ထုတ်လုပ်မှုရှိ AI အေဂျင့်များ](../../../translated_images/my/lesson-10-thumbnail.2b79a30773db093e.webp)](https://youtu.be/l4TP6IyJxmQ?si=reGOyeqjxFevyDq9)

AI အေဂျင့်များကို စမ်းသပ်မှု ပုံစံများမှ အမှန်တကယ်အသုံးချနိုင်သော အသုံးချမှုများသို့ ရွှေ့ပြောင်းလာမည်ဖြစ်လျှင်၊ ၎င်းတို့၏ အပြုအမူကို နားလည်နိုင်ခြင်း၊ စွမ်းဆောင်ရည်ကို မော်နီတာလုပ်နိုင်ခြင်းနှင့် ထုတ်လွှတ်ချက်များကို စနစ်တကျ အကဲဖြတ်နိုင်ခြင်းတို့သည် အရေးကြီးとなります။

## သင်ယူရမည့် ရည်မှန်းချက်များ

ဒီသင်ခန်းစာကို ပြီးဆုံးပါက သင်သည် အောက်ပါအချက်များကို သိရှိ/နားလည်နိုင်ပါမည်။  
- အေဂျင့်ကြည့်ရှုနိုင်ခြင်းနှင့် အကဲဖြတ်ခြင်း၏ မူလအယူအဆများ  
- အေဂျင့်များ၏ စွမ်းဆောင်ရည်၊ ကုန်ကျစရိတ်နှင့် ထိရောက်မှု တိုးတက်အောင်လုပ်နိုင်သည့်နည်းများ  
- သင့် AI အေဂျင့်များကို အရာများအရ စနစ်တကျ ဘယ်လို အကဲဖြတ်ရမည်နည်း  
- ထုတ်လုပ်ရေးတွင် AI အေဂျင့်များ တပ်ဆင်စဉ် ကုန်ကျစရိတ်ကို ဘယ်လို ထိန်းချုပ်မည်နည်း  
- Microsoft Agent Framework ဖြင့် တည်ဆောက်ထားသော အေဂျင့်များကို ဘယ်လို ပစ္စည်းစားမကထည့်မည်နည်း

ရည်ရွယ်ချက်မှာ သင့် "ဖြုတ်မပြည့် ဘောက်စ်" အဖြစ်ရှိသည့် အေဂျင့်များကို သက်သာမြင်သာပြီး စီမံထိန်းချုပ်နိုင်ပြီး ယုံကြည်စိတ်ချရသော စနစ်များအဖြစ် ပြောင်းလဲနိုင်စေဖို့ ဖြစ်ပါတယ်။

_**မှတ်ချက်:** AI အေဂျင့်များကို ဘေးမဲ့၊ ယုံကြည်စိတ်ချရအောင် ထုတ်လုပ်ရေးတွင် တပ်ဆင်ပေးခြင်းသည် အရေးကြီးပါသည်။ [Building Trustworthy AI Agents](./06-building-trustworthy-agents/README.md) သင်ခန်းစာကိုလည်း ကြည့်ပါ။_

## Traces and Spans

ကြည့်ရှုနိုင်မှု စက်ကိရိယာများ (ဥပမာ၊ [Langfuse](https://langfuse.com/) သို့မဟုတ် [Microsoft Foundry](https://learn.microsoft.com/en-us/azure/ai-foundry/what-is-azure-ai-foundry)) တွင်၊ အေဂျင့် run များကို အထူးသဖြင့် traces နှင့် spans အဖြစ် ကိုယ်စားပြုထားသည်။

- **Trace** သည် စေ့စပ်ပြီး အစမှ အဆုံးထိ တစ်ခုတည်းသော အေဂျင့်လုပ်ငန်းတာဝန်တစ်ရပ်ကို ကိုယ်စားပြုသည် (ဥပမာ၊ ဧည့်သည်မေးခွန်းတစ်ခုကို ကိုင်တွယ်ခြင်း)။  
- **Spans** သည် trace အတွင်းရှိ တစ်ဆင့်ချင်းစီဖြစ်ပြီး (ဥပမာ၊ ဘာသာစကားမော်ဒယ်ကို ခေါ်ဆိုခြင်း သို့မဟုတ် ဒေတာ ယူဆောင်လာခြင်း)။

ကြည့်ရှုနိုင်မှု မရှိပါက၊ AI အေဂျင့်သည် "မော်ဒယ်စက်ပုလင်း" အဖြစ် ခံစားရနိုင်သည် — ၎င်း၏ အတွင်းပိုင်း အခြေအနေနှင့် သဒ္ဒါကလည်း မမြင်သာ၍ ပြဿနာများကို ရှာဖွေရန် သို့မဟုတ် စွမ်းဆောင်ရည်အား အ tốiိမ့်ပြုပြင်ရန် ဒုက္ခရောက်စေသည်။ ကြည့်ရှုနိုင်မှုရှိပါက၊ အေဂျင့်များသည် "ရောင်ကြည့်မှန်ဖန်ချပ်" တို့လို ဖြစ်ပြီး ယုံကြည်မှု တည်ဆောက်ရန်နှင့် ရည်ရွယ်ချက်အတိုင်း လည်ပတ်မှုကို အာမခံရန် အလွန်အရေးကြီးသော ထင်ရှားမှုကို ပေးစွမ်းသည်။

## ထုတ်လုပ်ရေးပတ်ဝန်းကျင်များတွင် ကြည့်ရှုနိုင်မှု အရေးကြီးခြင်း

AI အေဂျင့်များကို ထုတ်လုပ်ရေးပတ်ဝန်းကျင်သို့ ရွှေ့ပြောင်းသည်နှင့်အမျှ အသစ်သော စိန်ခေါ်မှုများနှင့် လိုအပ်ချက်များ ပေါ်ပေါက်လာသည်။ ကြည့်ရှုနိုင်မှုသည် "ရှိသင့်တယ်" မဟုတ်ပဲ အဓိကလိုအပ်ချက်တစ်ခုဖြစ်လာပါသည်။

*   **Debugging နှင့် Root-Cause Analysis**: အေဂျင့်မအောင်မြင် သို့မဟုတ် မမျှော်လင့်ထားသော output ထုတ်ပေးသောအခါ၊ ကြည့်ရှုနိုင်မှုကိရိယာများသည် အမှားပျဉ်းစီးမှု၏ မူလအကြောင်းရင်းကို ဖော်ထုတ်ရန် လိုအပ်သည့် traces များကို ပေးစွမ်းသည်။ ၎င်းသည် LLM ခေါ်ဆိုမှုများ အများအပြား၊ ကိရိယာများနှင့် အပြန်အလှန် ဆက်ဆံမှုများ၊ အခြေအနေဆိုင်ရာ နည်းကိုင်သော Logic များပါဝင်နိုင်သော ရုပ်သံရှုပ်ထွေးသော အေဂျင့်များတွင် အထူးအရေးကြီးသည်။  
*   **Latency နှင့် ကုန်ကျစရိတ် စီမံခန့်ခွဲခြင်း**: AI အေဂျင့်များသည် မကြာခဏ LLM များနှင့် အခြားအပြင် APIs များတင့်ကို ထားရှိပြီး token နဲ့ သို့မဟုတ် ခေါ်ဆိုမှု တစ်ခုစီအလိုက် ငွေကြေးတွက်ချက်ခံရနိုင်သည်။ ကြည့်ရှုနိုင်မှုကိရိယာများက ထိုခေါ်ဆိုမှုများကို တိကျစွာ လိုက်လံမှတ်တမ်းတင်နိုင်စေပြီး နှေးကွေးသော်လည်း ဈေးကြီးသော လုပ်ငန်းစဉ်များကို ဖော်ထုတ်ကူညီပေးပါသည်။ ၎င်းက အဖွဲ့များအား prompt များကို တိုးတက်အောင်လုပ်ခြင်း၊ ပိုထိရောက်သော မော်ဒယ်များရွေးချယ်ခြင်း သို့မဟုတ် လုပ်ငန်းစဉ်များကို ပြန်ဒီဇိုင်းဆွဲခြင်းဖြင့် စက်မှုကုန်ကျစရိတ်များကို ထိန်းချုပ်စေပါသည်။  
*   **ယုံကြည်မှု၊ ဘေးကင်းမှုနှင့် လိုက်နာမှု**: အလွန်အများအသုံးချသော အသုံးချမှုများတွင် အေဂျင့်များသည် ဘေးကင်းစိတ်ချရပြီး ကျင့်ဝတ်ရှိစေရန်အရေးကြီးသည်။ ကြည့်ရှုနိုင်မှုကိရိယာများက အေဂျင့်လုပ်ဆောင်ချက်များနှင့် ဆုံးဖြတ်ချက်များ၏ audit trail ကို ပေးသည်။ ၎င်းကို prompt injection,ထိခိုက်နိုင်သော အကြောင်းအရာများ ထုတ်လုပ်ခြင်း သို့မဟုတ် ကိုယ်ရေးရာဇဝင် (PII) များကို မှားယွင်းစွာ ကိုင်တွယ်ခြင်း ကဲ့သို့သော ပြဿနာများကို ဖော်ထုတ်၍ တားဆီးရန် အသုံးပြုနိုင်သည်။ ဥပမာအားဖြင့်၊ trace များကို ပြန်လည်သုံးသပ်၍ အေဂျင့်သည် မည်သည့်အကြောင်းဖြင့် တုံ့ပြန်ချက်တစ်ခု ပေးခဲ့သည် သို့မဟုတ် သတ်မှတ် tool တစ်ခုကို မည်သို့ အသုံးပြုခဲ့သည်ကို နားလည်နိုင်သည်။  
*   **ဆက်လက်တိုးတက်မှု Loop များ**: ကြည့်ရှုနိုင်မှုဒေတာသည် iterative ဖွံ့ဖြိုးရေးလုပ်ငန်းစဉ်၏ အခြေခံဖြစ်သည်။ အေဂျင့်များ၏ ပတ်ဝန်းကျင်တွင် ဘယ်လို စွမ်းဆောင်နေသည်ကို မော်နီတာလုပ်ခြင်းအားဖြင့် တိုးတက်ရန် နေရာများကို ဖော်ထုတ်နိုင်ပြီး မော်ဒယ်များအတွက် fine-tuning အချက်အလက်စုဆောင်းခြင်းနှင့် အပြောင်းအလဲများ၏ ထိရောက်မှုကို အတည်ပြုနိုင်သည်။ ၎င်းသည် online အကဲဖြတ်မှ offline စမ်းသပ်မှုများသို့ ထုတ်လုပ်ရေးအမြင်များကို ပြန်လည်အသုံးချသည့် feedback loop တစ်ခုကို ဖန်တီးပေးသည်။

## လိုက်လံစောင့်ကြည့်ရန် အဓိက မီထရစ်များ

အေဂျင့်၏ အပြုအမူကို မော်နီတာလုပ်ရန်နှင့် နားလည်ရန်၊ မီထရစ်များနှင့် အချက်အလက်အမျိုးမျိုးကို လိုက်လံမှတ်တမ်းတင်သင့်သည်။ အေဂျင့်ရည်ရွယ်ချက်ပေါ် မူတည်၍ အချို့ metrics များကတော့ အမျိုးမျိုးဖက်ကွဲနိုင်ပေမယ့် အောက်ပါအချို့သည် အများအားဖြင့် အရေးကြီးသည်။

ဒီမှာ ကြည့်ရှုနိုင်မှုကိရိယာများ ပုံမှန်စောင့်ကြည့်သည့် ထိပ်တန်း metrics အချို့ကို ဖော်ပြထားသည်။

**Latency:** အေဂျင့်သည် မျှဘယ့်အမြန်ဖြင့် တုံ့ပြန်သနည်း။ ကြာရှည်စောင့်ရခြင်းသည် အသုံးပြုသူ အတွေ့အကြုံကို မကောင်းစေပါ။ သင်သည် လုပ်ဆောင်ချက်များနှင့် တစ်ဆင့်ချင်းစီအတွက် latency ကို trace များနှင့် တိုင်းတာသင့်သည်။ ဥပမာအားဖြင့်၊ မော်ဒယ်ခေါ်ဆိုမှုများ အားလုံးအတွက် 20 စက္ကန့် ကြာမောင်းမဆို သင်သည် မြန်ဆန်သော မော်ဒယ် အသုံးပြုခြင်း သို့မဟုတ် မော်ဒယ်ခေါ်ဆိုမှုများကို 병렬 (parallel) ဖြင့် ဆောင်ရွက်ခြင်းဖြင့် မြန်ဆန်အောင်လုပ်နိုင်သည်။

**Costs:** တစ်ခုချင်းစီ အေဂျင့် run အတွက် ကုန်ကျစရိတ် ဘယ်လောက်လဲ။ AI အေဂျင့်များသည် LLM ခေါ်ဆိုမှုများ သို့မဟုတ် အပြင် API များကို ရင်းနှီးစွာ အသုံးပြုကြပြီး token အလိုက် သို့မဟုတ် ခေါ်ဆိုမှုအလိုက် ငွေပေးချေကြသည်။ ကိရိယာများကို မကြာခဏ သုံးခြင်း သို့မဟုတ် prompts အများအပြားရှိခြင်းက ကုန်ကျစရိတ်လျင်မြန်စွာ တိုးပွားနိုင်သည်။ ဥပမာအားဖြင့်၊ အေဂျင့်သည် အရည်အသွေးတိုးတက်မှုနည်းနည်းအတွက် LLM ကို ငါးကြိမ်ခေါ်ဆိုလျင်၊ ဤကုန်ကျစရိတ်သည် တန်ဖိုးရှိမရှိကို သတ်မှတ်ရမည်၊ သို့မဟုတ် ခေါ်ဆိုမှုအရေအတွက်ကို လျော့ချပေးခြင်း သို့မဟုတ် ဈေးသက်သာသော မော်ဒယ် အသုံးပြုခြင်းကို သုံးစွဲနိုင်မည်မဟုတ်ဆိုတာ ဆန်းစစ်ရမည်။ အချိန်နှင့်တပြေးညီ မော်နီတာလုပ်ခြင်းက မမျှော်လင့်ထားသော တက်လှမ်းမှုများ (ဥပမာ၊ အမှားများကြောင့် API loop များ မကုန်ဆုံးခြင်း) ကို ဖော်ထုတ်ရာတွင် အထောက်အကူပြုသည်။

**Request Errors:** အေဂျင့်သည် မည်မျှသော တောင်းဆိုချက်များကို မအောင်မြင်ခဲ့သနည်း။ ၎င်းတွင် API error များ သို့မဟုတ် tool ခေါ်ဆိုမှု မအောင်မြင်မှုများ ပါဝင်နိုင်သည်။ ထုတ်လုပ်ရေးတွင် ဤနှင့်အမျိုးမျိုးသော အမှားများကြောင့် အေဂျင့်ကို ပိုမိုစောင့်ရှောက်နိုင်ရန်၊ fallback များ သို့မဟုတ် retry မက်ကနစ်များကို ဆက်ဆံပြီး set up လုပ်နိုင်သည်။ ဥပမာ။ LLM provider A ချ်င်းချုပ်နေပါက၊ provider B သို့ ဗက်ခ်အပ်အဖြစ် ပြောင်းနိုင်သည်။

**User Feedback:** တိုက်ရိုက် အသုံးပြုသူ အကဲဖြတ်ချက်များကို တပ်ဆင်ခြင်းက တန်ဖိုးရှိသော အမြင်များကို ပေးသည်။ ၎င်းတွင် တိုက်ရိုက် အဆင့်သတ်မှတ်ခြင်းများ (👍thumbs-up/👎down, ⭐1-5 ကြာ) သို့မဟုတ် စာသားပြောကြားချက်များ ပါဝင်နိုင်သည်။ ဆက်တိုက် ဖျက်ဖျက်စရာ မကျမနေပေးသည့် အမူအရာများသည် အေဂျင့် သင့်တော်စွာ လည်ပတ်မနေကြောင်း သတိပေးမှု ဖြစ်စေသည်။

**Implicit User Feedback:** အသုံးပြုသူ အပြုအမူများသည် တိုက်ရိုက်အဆင့်သတ်မှတ်ချက် မရှိပေမယ့် အကြောင်းပြချက် အသုံးပြုသူ တုံ့ပြန်ချက် အမျိုးမျိုးကို ပေးသည်။ ၎င်းတွင် မေးခွန်းကို ချက်ချင်း ပြန်လည်ဖွဲ့စည်းခြင်း၊ မေးခွန်းများကို ချက်ချင်း ထပ်မံမေးခြင်း သို့မဟုတ် retry ခလုတ်ကို နှိပ်ျခင်း ပြုမူမှုများ ပါဝင်နိုင်သည်။ ဉပမာ။ အသုံးပြုသူများသည် တပြန်တလှန် အတူတကွ တူညီသော မေးခွန်းကို ဆက်တိုက်မေးနေပါက၊ ၎င်းသည် အေဂျင့်သည် မမျှော်လင့်သည့် နည်းဖြင့် လည်ပတ်နေသည်ဟု အချက်ပြစေသည်။

**Accuracy:** အေဂျင့်သည် မှန်ကန် သို့မဟုတ် စိတ်တိုင်းမကျလှည့်လည်မှုများကို မကြာခဏ ထုတ်ပေးသနည်း။ Accuracy အဓိပ္ပာယ်များသည် မတူကွဲပြားနိုင်သည် (ဥပမာ၊ ပြဿနာဖြေရှင်းမှုမှန်ကန်မှု၊ သတင်းအချက်အလက် ရှာဖွေမှန်ကန်မှု၊ အသုံးပြုသူ စိတ်ကျေနပ်မှု)။ ပထမဆုံး ချမှတ်ရမည့် အချက်မှာ သင့် အေဂျင့်အတွက် အောင်မြင်မှုသည် ဘာလဲဆိုတာ သတ်မှတ်ခြင်းဖြစ်သည်။ သင်သည် automated 검사 များ၊ အကဲဖြတ်အမှတ်များ သို့မဟုတ် တာဝန်ပြီးစီးခြင်း အမှတ်အသားများမှတဆင့် accuracy ကို လိုက်လံတိုင်ကြားနိုင်သည်။ ဥပမာ၊ trace များကို "succeeded" သို့မဟုတ် "failed" ဟု အမှတ်ပေးခြင်း။

**Automated Evaluation Metrics:** automated evals များကိုလည်း တပ်ဆင်နိုင်သည်။ ဥပမာအားဖြင့်၊ ထုတ်လွှတ်ထားသည့် output ကို LLM တစ်ခု အသုံးပြု၍ အကဲဖြတ်ပေးစေ၍ ၎င်းသည် ဒီအချက်များအတွက် အထောက်အကူဖြစ်သည်၊ မှန်ကန်မှုရှိသလား၊ ကူညီဖြစ်စေသလားဆိုတာ သတ်မှတ်နိုင်သည်။ agent ၏ မတူညီသော အစိတ်အပိုင်းများကို အမှတ်ပေးရာတွင် ကူညီပေးသည့် အများအပြား ဖွင့်လှစ် အသံလက်ခံ ဗဟိုစာကြည့်တိုက် (open source) ไลဘရေးရီများလည်း ရှိကြသည်။ ဥပမာ။ RAG အေဂျင့်များအတွက် [RAGAS](https://docs.ragas.io/) သို့မဟုတ် ထိခိုက်နိုင်သော စကားလုံးများ သို့မဟုတ် prompt injection ကို မှတ်သားစစ်ဆေးရန် [LLM Guard](https://llm-guard.com/) စတဲ့ tool များရှိသည်။ 

ကိုက်ညီစွာကျင့်သုံးပါက၊ ဤ metrics များ စုစုပေါင်းက AI အေဂျင့်၏ ကျန်းမာရေးကို အကောင်းဆုံး ဖုံးလွှမ်းပေးနိုင်သည်။ ဤအခန်းတွင်ရှိသော [example notebook](./code_samples/10-expense_claim-demo.ipynb) တွင် သင့်အား ဤ metrics များကို အမှန်တကယ် ဥပမာများဖြင့် ပြသပေးမည်၊ သို့သော် ပထမဦးဆုံး သာမန်အကဲဖြတ် လုပ်ငန်းစဉ်သည် ဘယ်လို ဖြစ်ကြောင်း သိရှိလိုပါသည်။

## သင့် အေဂျင့်ကို ပစ္စည်းစားပါ (Instrument your Agent)

tracing data များစုဆောင်းရန်၊ သင်သည် ကိုဒ်ကို ပစ္စည်းစားရန် လိုအပ်မည်။ ရည်ရွယ်ချက်မှာ အေဂျင့်ကို trace များနှင့် metrics များ ထုတ်ပေးရန် instrument လုပ်ခြင်းဖြစ်ပြီး ၎င်းတို့ကို ကြည့်ရှုနိုင်မှု ပလက်ဖောင်းတစ်ခုဖြင့် ဖမ်းဆီး၊ ကိုင်တွယ်နှင့် ပြသနိုင်ရမည်။

**OpenTelemetry (OTel):** [OpenTelemetry](https://opentelemetry.io/) သည် LLM ကြည့်ရှုနိုင်ရေးအတွက် စက်မှုစံချိန်စနစ်တစ်ခုအဖြစ် ပေါ်ထွန်းလာပြီး telemetry ဒေတာများ ဖန်တီး၊ စုဆောင်းနှင့် export လုပ်ရန် API များ၊ SDK များနှင့် ကိရိယာများကို ပေးသည်။ 

အမျိုးမျိုးသော instrumentation ไลဘရေးရီများသည် ရှိပြီး ယင်းတို့က ရှိပြီးသား အေဂျင့် framework များကို ဝက်/ထုပ်ပိုးပေး၍ OpenTelemetry span များကို ကြည့်ရှုနိုင်မှုကိရိယာသို့ အလွယ်တကူ export ပြုလုပ်နိုင်စေသည်။ Microsoft Agent Framework သည် OpenTelemetry နှင့် ပုံမှန်ချိတ်ဆက်ထားသည်။ အောက်တွင် MAF agent တစ်ခုကို instrument လုပ်နည်း၏ ဥပမာတစ်ခုကို ပြထားသည်။

```python
from agent_framework.observability import get_tracer, get_meter

tracer = get_tracer()
meter = get_meter()

with tracer.start_as_current_span("agent_run"):
    # အေဂျင့် ဆောင်ရွက်မှုကို အလိုအလျောက် လိုက်လံမှတ်တမ်းတင်သည်
    pass
```

ဤ အခန်းရှိ [example notebook](./code_samples/10-expense_claim-demo.ipynb) သည် သင့် MAF agent ကို ဘယ်လို ပစ္စည်းစားရမလဲကို ပြသပေးမည်။

**Manual Span Creation:** instrumentation ไลဘရေးရီများသည် ကောင်းမွန်သည့် စံဖြစ်သော်လည်း၊ ပိုမိုအသေးစိတ် သို့မဟုတ် အထူးလိုအပ်ချက်များ ရှိလာနိုင်သည်။ ထိုအခါ သင်သည် custom application logic ထည့်သွင်းရန် မျက်နှာတင် span များကို ကိုယ်တိုင် ဖန်တီးနိုင်သည်။ အဓိကအားဖြင့်၊ ၎င်းတို့သည် အလိုအလျောက် သို့မဟုတ် ကိုယ်တိုင် ဖန်တီးထားသော span များအား custom attributes (tag များ သို့မဟုတ် metadata အဖြစ်လည်း သိကြသည်) ဖြင့် အထောက်အပံ့ဖြည့်စွက်နိုင်သည်။ ၎င်း attributes များတွင် စီးပွားရေးအထူးဆိုင်ရာ ဒေတာများ၊ အလယ်အလတ်တွက်ချက်ချက်များ သို့မဟုတ် debugging သို့မဟုတ် ခွဲခြမ်းစိတ်ဖြာရန် အသုံးဝင်နိုင်သည့် အရာများ (ဥပမာ `user_id`, `session_id`, သို့မဟုတ် `model_version`) ပါဝင်နိုင်သည်။

[Langfuse Python SDK](https://langfuse.com/docs/sdk/python/sdk-v3) ဖြင့် traces နှင့် spans များကို ကိုယ်တိုင် ဖန်တီးခြင်း၏ ဥပမာ:

```python
from langfuse import get_client
 
langfuse = get_client()
 
span = langfuse.start_span(name="my-span")
 
span.end()
```

## အေဂျင့် အကဲဖြတ်ခြင်း

ကြည့်ရှုနိုင်မှုက မီထရစ်များ ပေးသည်၊ သို့သော် အကဲဖြတ်ခြင်းသည် ထိုဒေတာကို ခွဲခြမ်းစိတ်ဖြာခြင်း (နှင့် စမ်းသပ်ချက်များ အသုံးပြုခြင်း) ဖြစ်ပြီး AI အေဂျင့်၏ စွမ်းဆောင်ရည် ကောင်း/မကောင်းနှင့် ကောင်းမွန်လာစေရန် ဘယ်လိုပြုပြင်နိုင်သည်ကို သတ်မှတ်ပေးသည်။ အခြားသော ဝေါဟာရဖြင့် ဆိုရမည်ဆိုရင်၊ သင်မှာ ထို traces နှင့် metrics များ ရှိသွားလျှင်၊ ၎င်းတို့ကို ဘယ်လို အသုံးချ၍ အေဂျင့်ကို အကဲဖြတ်၍ ဆုံးဖြတ်ချက်များချမည်နည်း။

အကဲဖြတ်မှုကို ပုံမှန်ပြုလုပ်ခြင်းသည် အရေးကြီးသည်၊ အကြောင်းမှာ AI အေဂျင့်များသည် မကြာခဏ နမူနာ မဟုတ်သော နည်းဖြင့် အလုပ်လုပ်နိုင်ပြီး (နမူနာပြောင်းလဲမှုများ သို့မဟုတ် မော်ဒယ် ဝမ်းနှိမ်မှုများကြောင့်) တိုးတက်လာနိုင်သည် — အကဲဖြတ်မရှိလျှင် သင့် "စမတ် အေဂျင့်" သည် တကယ်ထိရောက်စွာ လုပ်ဆောင်နေမနေ မှတ်မိနိုင်ဘူး။

AI အေဂျင့်များအတွက် အကဲဖြတ်မှုကို နှစ်မျိုးခွဲနိုင်သည်။ **online evaluation** နှင့် **offline evaluation** ။ နှစ်ခုစလုံး အသုံးဝင်ပြီး၊ ကွဲပြားပြီး ဆက်လျက်အကူဖြစ်သည်။ ကျွန်တော်တို့သည် ပုံမှန်အားဖြင့် offline evaluation ဖြင့် စတင်ပြီး၊ ၎င်းသည် အေဂျင့်တစ်ခုတည်းကို ထုတ်လုပ်ရေးသို့ တင်မပြုလုပ်ခင် အနည်းဆုံး လုပ်ဆောင်သင့်သည့် အဆင့်ဖြစ်သည်။

### Offline Evaluation

![Langfuse တွင် Dataset items](https://langfuse.com/images/cookbook/example-autogen-evaluation/example-dataset.png)

ဤသည်မှာ ထိန်းချုပ်ထားသော ပတ်ဝန်းကျင်တွင် (ယေဘုယျအားဖြင့် စမ်းသပ် dataset များကို အသုံးပြု၍) အေဂျင့်ကို အကဲဖြတ်ခြင်းကို ဆိုလိုသည်၊ အသက်သက်သော အသုံးပြုသူ မေးခွန်းများ မဟုတ်ပဲ။ သင်သည် မျှော်မှန်းထားသော output သို့မဟုတ် မှန်ကန်သော အပြုအမူကို သိရှိထားသည့် curated dataset များကို အသုံးပြု၍ agent ကို တိုက်တွန်းRun လုပ်ကြသည်။

ဥပမာအားဖြင့်၊ သင်သည် သင်တန်းဒီဇိုင်းထားသည့် သင်ခန်းစာ math word-problem အေဂျင့်တစ်ခု တည်ဆောက်ထားပါက၊ သင်သည် နိဒါန်းဖြေကြောင်းသတ်မှတ်ထားသည့် ပြဿနာ 100 ခုပါသော [test dataset](https://huggingface.co/datasets/gsm8k) တစ်ခုရှိနိုင်သည်။ Offline evaluation ကို မူလအားဖြင့် ဖွံ့ဖြိုးရေးအတွင်း (နှင့် CI/CD pipeline အားဖြင့် အစိတ်အပိုင်းတစ်ခုအနေနှင့်) ပုံမှန်ပြုလုပ်ကြသည်၊ လမ်းကြောင်းပြုပြင်မှုများ သို့မဟုတ် regression များကို ကာကွယ်ရန်။ အကျိုးချမ်းသာချက်မှာ ၎င်းသည် **ထပ်ခါထပ်ခါဆောင်ရွက်နိုင်ပြီး မှန်ကန်မှန်မှုကို မြင်သာစေသည်**၊ သင်တွင် ground truth ရှိသဖြင့် ဖြစ်သည်။ သင်သည် အသုံးပြုသူ မေးခွန်းများကို จำลองပုံစံဖြင့် စမ်းသပ်၍ agent ၏ တုံ့ပြန်ချက်များကို ကောင်းမွန်သော ဖြေရှင်းချက်များနှင့် နှိုင်းယှဉ်နိုင်သည် သို့မဟုတ် အထက်တွင်ဖော်ပြထားသည့် automated metrics များကို အသုံးပြုနိုင်သည်။

Offline eval ၏ အဓိက စိန်ခေါ်မှုမှာ သင်၏ စမ်းသပ် dataset သည် အကျုံကပ်ပြီး သက်ဆိုင်လျက် ရှိနေစေရန် ဖြစ်သည် — အေဂျင့်သည် ပိတ်ထားသော စမ်းသပ်စာစုကျသည့် အားဖြင့် ကောင်းစွာ အလုပ်လုပ်နိုင်ပေမယ့် ထုတ်လုပ်ရေးတွင် ကွဲပြားသော မေးခွန်းများနှင့် ရင်ဆိုင်နိုင်သည်။ ထို့ကြောင့် စမ်းသပ်အစုများကို အဆုံးမပျက်သလို နောက်ထပ် edge case များနှင့် လက်တွေ့လည်းကောင်းသော ဥပမာများဖြင့် update လုပ်သင့်သည်။ အရွယ်ချင်းသေးသည့် "smoke test" ကိစ္စများနှင့် ပိုမိုကြီးမားသော အကဲဖြတ်စာစုများ အားလုံးပေါင်းစပ်အသုံးပြုခြင်းက အသုံးဝင်သည်။ သေးသော စစ်ဆေးမှုများသည် အမြန်စစ်ဆေးမှုများအတွက်၊ ကြီးမားသော အစုများသည် ကျယ်ပြန့်သော စွမ်းဆောင်ရည် metrics များအတွက် အသုံးဝင်သည်။

### Online Evaluation 

![Observability metrics overview](https://langfuse.com/images/cookbook/example-autogen-evaluation/dashboard.png)

ဤသည်မှာ ထုတ်လုပ်ရေးတွင် တိုက်ရိုက် အသုံးပြုနေစဉ်၊ လက်တွေ့ ပတ်ဝန်းကျင်တွင် အေဂျင့်ကို အကဲဖြတ်ခြင်းကို ဆိုလိုသည်။ Online evaluation သည် လက်တွေ့အသုံးပြုမှုများပေါ်တွင် အေဂျင့်၏ စွမ်းဆောင်ရည်ကို မော်နီတာလုပ်ခြင်းနှင့် ရလဒ်များကို အဆက်မပြတ် ခွဲခြမ်းစိတ်ဖြာခြင်းကို ပါဝင်သည်။

ဥပမာအားဖြင့်၊ သင်သည် live traffic ပေါ်တွင် အောင်မြင်မှုနှုန်းများ၊ အသုံးပြုသူ ကျေနပ်မှု အဆင့်များ သို့မဟုတ် အခြား metrics များကို လိုက်လံကြည့်ရှုနိုင်သည်။ Online evaluation ၏ အားသာချက်မှာ သင်ျမင်မထင်ခဲ့ဖြစ်နိုင်သည့် အရာများကို လက်တွေ့ကနေ သိရှိနိုင်ခြင်းဖြစ်သည် — သင်သည် ၎င်းကို lab ပတ်ဝန်းကျင်တွင် မမျှော်လင့်ရသည့် အရာများကို ဖော်ထုတ်နိုင်သည်၊ မော်ဒယ်၏ drift ကို အချိန်နှင့်အမျှ မြင်တွေ့နိုင်ပါသည် (input ပုံစံများ ပြောင်းလဲသွားသည်နှင့်အမျှ agent ၏ ထိရောက်မှုလျော့ကျသွားနိုင်သည်)။ ၎င်းက သင့်အား အေဂျင့်သည် လက်တွေ့ဘယ်လို လုပ်ဆောင်သနည်းကို တိကျစွာ ပြသပေးနိုင်သည်။

Online evaluation တွင် implicit နှင့် explicit အသုံးပြုသူ တုံ့ပြန်ချက်များစုဆောင်းခြင်းပါဝင်နိုင်ပြီး ကာလတစ်ခုပြီးမှ shadow tests သို့မဟုတ် A/B tests (agent ၏ အသစ်သော ဗားရှင်းကို ရှေ့ဟောင်းနှင့် နှိုင်းယှဉ်ရန် 병렬 run လုပ်ခြင်း) များ ပြုလုပ်နိုင်သည်။ စိန်ခေါ်မှုမှာ live အပြုအမူများအတွက် ယုံကြည်စိတ်ချရသော အမှတ်ချထားချက်များကို ရယူရခက်သည် — သင်သည် အသုံးပြုသူ တုံ့ပြန်ချက်များ သို့မဟုတ် အောက်ခြေ metrics များ (ဥပမာ၊ အသုံးပြုသူက ရလဒ်ကို နှိပ်ကြည့်ခဲ့လား) ကို အားမလိုက်နိုင်လိမ့်မည်။

### နှစ်မျိုးကို ပေါင်းစပ်ခြင်း

Online နှင့် offline အကဲဖြတ်မှုများသည် ဆန့်ကျင်ဘက် မဟုတ်ပေ။ ၎င်းတို့သည် အချင်းချင်း အပြည့်အ၀ မျှောက်ပေးကြသည်။ Online မော်နီတာမှ ရရှိသည့် အမြင်များ (ဥပမာ၊ အေဂျင့်သည် အားနည်းသည့် အသုံးပြုသူ မေးခွန်း အသစ်များ) ကို offline စမ်းသပ် dataset များကို တိုးမြှင့်ရန် အသုံးပြုနိုင်သည်။ အပြန်အလှန်အားဖြင့် offline စမ်းသပ်တွင် ကောင်းစွာ လုပ်ဆောင်သော အေဂျင့်များကို ထုတ်လုပ်ရေးသို့ ယုံကြည်စိတ်ချစွာ တင်ပြီး online တွင် ပိုမိုကောင်းမွန်စွာ မော်နီတာလုပ်နိုင်သည်။

အမှန်တကယ် အဖွဲ့များ စနစ်တကျ loop တစ်ခုကို လက်ခံလိုက်ကြသည်။

_evaluate offline -> deploy -> monitor online -> collect new failure cases -> add to offline dataset -> refine agent -> repeat_.

## ပုံမှန်တွေ့ကြုံရသော ပြဿနာများ

AI အေဂျင့်များကို ထုတ်လုပ်ရေးသို့ တပ်ဆင်သောအခါ အမျိုးမျိုးသော စိန်ခေါ်မှုများနှင့် ရင်ဆိုင်ရနိုင်သည်။ အောက်တွင် ပုံမှန်တွေ့ကြုံရသော ပြဿနာများနှင့် ၎င်းတို့အတွက် အလားအလာဖြေရှင်းနည်းများကို ဖော်ပြထားသည်။

| **ပြဿနာများ**    | **ဖြေရှင်းနည်းများ**   |
| ------------- | ------------------ |
| AI Agent not performing tasks consistently | - Prompt ကို ပြန်လည်သိမ်းပိုက်ပေးပါ; ရည်ရွယ်ချက်များကို ထပ်မံရှင်းလင်းစေပါ။<br>- တာဝန်များကို သေးငယ်သည့် subtasks များသို့ ခွဲ၍ agent များစွာဖြင့် ကိုင်တွယ်ရန်ဖြစ်နိုင်သည်။ |
| AI Agent running into continuous loops  | - Agent သည် လုပ်ငန်းစဉ်ကို ရပ်တန့်ရန် နိယာမများကို ရှင်းလင်းစွာ သိရှိစေရန် termination သတ်မှတ်ချက်များထည့်ပါ။<br>- ဆင်ခြေခက်သော သဒ္ဒါနှင့် ပြင်ဆင်မှုလိုအပ်သော အလုပ်များအတွက် အတွေးခေါ်နိုင်သည့် အထူးသင့်တော်သည့် ကြီးမားသော မော်ဒယ်တစ်မျိုးကို အသုံးပြုပါ။ |
| AI Agent tool calls are not performing well   | - tool ၏ output ကို အေဂျင့်စနစ်အပြင်၌ စမ်းသပ်၍ အတည်ပြုပါ။<br>- သတ်မှတ်ထားသော ပါရာမီတာများ၊ prompts နှင့် tool များ၏ နာမည္ပုံစံများကို ပြန်လည်တိုးတက်စေပါ။  |
| Multi-Agent system not performing consistently | - အသင်းတစ်ခုချင်းစီကို ပေးသည့် prompts များကို ပိုမိုသီးသန့်နှင့် ကွဲပြားစေရန် ပြုပြင်ပါ။<br>- ဘယ် agent သည် မှန်ကန်မှုရှိသည်ကို ဆုံးဖြတ်ရန် "routing" သို့မဟုတ် controller agent တစ်ယောက်ကို အသုံးပြု၍ အဆင့်လိုက် ပုံစံတက်ဆောက်ပါ။ |

ဤပြဿနာအများစုကို ကြည့်ရှုနိုင်မှုကိုတပ်ဆင်ထားလျှင် ပိုမိုထိရောက်စွာ ဖော်ထုတ်နိုင်သည်။ အထက်တွင် ဆွေးနွေးထားသည့် trace များနှင့် metrics များက အက်ဂျင့်လုပ်ငန်းစဉ်၏ ဘယ်နေရာတွင် ပြဿနာများ ဖြစ်ပေါက်နေသည်ကို တိတိကျကျ ဖော်ထုတ်ပေးကာ debugging နှင့် အကောင်းစီမံခြင်းကို အလွန်ထိရောက်စေပါသည်။

## ကုန်ကျစရိတ် စီမံခြင်း
Here are some strategies to manage the costs of deploying AI agents to production:

**သေးငယ်သည့် မော်ဒယ်များကို အသုံးပြုခြင်း:** Small Language Models (SLMs) သည် agent များ အသုံးပြုသည့် အချို့သော use-case များတွင် စွမ်းဆောင်ချက်ကောင်းစွာ ပြသနိုင်ပြီး ကုန်ကျစရိတ်ကို ထင်ရှားစွာ လျော့နည်းစေပါသည်။ ယခင်တွင် ဖော်ပြခဲ့သလို၊ SLM တစ်ခုက သင့် use-case တွင် မည်မျှ ကောင်းမွန်စွာ လုပ်ဆောင်နိုင်မည်ကို သိရှိရန်အတွက် ကြီးမားသော မော်ဒယ်များနှင့် ယှဉ်ပြိုင်၍ စွမ်းဆောင်ချက်များကို သတ်မှတ်နှိုင်းယှဉ်နိုင်သော အကဲဖြတ်စနစ်တစ်ခု တည်ဆောက်ခြင်းဖြင့် ပိုမိုမှန်ကန်စွာ နားလည်နိုင်မည်ဖြစ်သည်။ intent classification သို့မဟုတ် parameter extraction ကဲ့သို့ ရိုးရှင်းသော အလုပ်တာဝန်များအတွက် SLM များကို အသုံးပြုရန် စဉ်းစားပြီး ရှုပ်ထွေးသော အတွေးအခေါ်များအတွက် ကြီးမားသော မော်ဒယ်များကို သီးသန့် ထားသုံးပါ။

**Router မော်ဒယ်ကို အသုံးပြုခြင်း:** ဆင်တူသော နည်းလမ်းတစ်ခုမှာ မော်ဒယ်အမျိုးအစားနှင့် အရွယ်အစား မျိုးစုံကို အလှည့်မပြတ် အသုံးပြုခြင်း ဖြစ်သည်။ LLM/SLM သို့မဟုတ် serverless function တစ်ခုကို အသုံးပြုပြီး မေးခွန်းများ၏ ရှုပ်ထွေးမှုအခြေအနေအပေါ်မူတည်၍ အလွန် သင့်လျော်သော မော်ဒယ်များထံ route ပေးနိုင်သည်။ ၎င်းသည် ကုန်ကျစရိတ်ကိုလည်း လျော့နည်းစေပြီး သင့်လုပ်ငန်းအတွက် သင့်လျော်သော တာဝန်များတွင် စွမ်းဆောင်နိုင်စွမ်းကို သေချာစေပါမည်။ ဥပမာအားဖြင့် ရိုးရှင်းသော မေးခွန်းများကို သေးငယ်၊ ပိုမိုမြန်နှုန်းရှိသော မော်ဒယ်များသို့ လမ်းကြောင်းပေးကာ ရှုပ်ထွေးသော သဘောတရားနှင့် သက်ဆိုင်သော အလုပ်များအတွက်သာ တန်ဖိုးအမြင့်မော်ဒယ်များကို သုံးပါ။

**ဖြေကြားချက်များကို ကက်ရှ်ထားခြင်း:** မကြာခဏတောင်းဆိုမှုများနှင့် ပုံမှန် အလုပ်စဉ်များကို ဖော်ထုတ်ကာ agent စနစ်ထဲသို့ ကောက်ခံခံရမှီ မဟုတ်ပဲ ကြိုတင်ဖြေကြားပေးခြင်းသည် ဆင်တူသော တောင်းဆိုမှုများ၏ ပမာဏကို လျော့နည်းစေရန် ကောင်းသော နည်းလမ်းဖြစ်သည်။ ပိုမိုမူလကျသော AI မော်ဒယ်များကို အသုံးပြု၍ တောင်းဆိုမှုတစ်ခုအား သင့် cache ရှိ တောင်းဆိုမှုများနှင့် မည်မျှ ဆင်တူသည်ကို သတ်မှတ်ပေးနိုင်သော flow တစ်ခုကိုလည်း ထည့်သွင်းတည်ဆောက်နိုင်သည်။ ဤနည်းလမ်းသည် မကြာခဏ မေးလေ့ရှိသော မေးခွန်းများ သို့မဟုတ် ပေါင်းစည်းလေ့ရှိသည့် workflow များအတွက် ကုန်ကျစရိတ်ကို အလွန်သက်သာစေပါသည်။
