# AI Agents in Production: Observability & Evaluation

AI အေးဂျင့်များကို စမ်းသပ်မှုအဆင့်မှ အမှန်တကယ်အသုံးချနိုင်သော အက်ပလီကေးရှင်းများသို့ ရွှေ့ပြောင်းသည့်အခါ၊ ၎င်းတို့၏ အပြုအမူကို နားလည်ခြင်း၊ စွမ်းဆောင်ရည်ကို စောင့်ကြည့်ခြင်းနှင့် အထွက်များကို စနစ်တကျ အကဲဖြတ်ခြင်းတို့သည် အရေးပါလာသည်။

## သင်ယူရမည့်အရာများ

ဒီသင်ခန်းစာပြီးဆုံးပြီးနောက်၊ သင်သည် အောက်ပါအရာများကို သိရှိနားလည်မည်ဖြစ်သည်-
- အေးဂျင့်၏ စောင့်ကြည့်နိုင်မှုနှင့် အကဲဖြတ်မှုဆိုင်ရာ အဓိကအယူအဆများ
- အေးဂျင့်များ၏ စွမ်းဆောင်ရည်၊ ကုန်ကျစရိတ်နှင့် ထိရောက်မှုကို တိုးတက်စေရန် နည်းလမ်းများ
- သင့် AI အေးဂျင့်များကို စနစ်တကျ အကဲဖြတ်ရမည့်အရာများနှင့် နည်းလမ်းများ
- AI အေးဂျင့်များကို ထုတ်လုပ်မှုအဆင့်တွင် အသုံးချသည့်အခါ ကုန်ကျစရိတ်ကို ထိန်းချုပ်နည်း
- AutoGen ဖြင့် တည်ဆောက်ထားသော အေးဂျင့်များကို စောင့်ကြည့်ခြင်းနည်းလမ်း

ဤသင်ခန်းစာ၏ ရည်ရွယ်ချက်မှာ "အနက်ရောင်သေတ္တာ" အေးဂျင့်များကို ပွင့်လင်းမြင်သာသော၊ စီမံခန့်ခွဲနိုင်သော၊ ယုံကြည်စိတ်ချရသော စနစ်များအဖြစ် ပြောင်းလဲနိုင်ရန် သင့်အား အသိပညာပေးခြင်းဖြစ်သည်။

_**မှတ်ချက်:** ယုံကြည်စိတ်ချရသော AI အေးဂျင့်များကို ထုတ်လုပ်ရန် အရေးကြီးသည်။ [Building Trustworthy AI Agents](./06-building-trustworthy-agents/README.md) သင်ခန်းစာကို ကြည့်ရှုပါ။_

## Traces နှင့် Spans

[Langfuse](https://langfuse.com/) သို့မဟုတ် [Azure AI Foundry](https://learn.microsoft.com/en-us/azure/ai-foundry/what-is-azure-ai-foundry) ကဲ့သို့သော စောင့်ကြည့်မှုကိရိယာများသည် အေးဂျင့်၏ လုပ်ဆောင်မှုများကို trace နှင့် span အဖြစ် ဖော်ပြလေ့ရှိသည်။

- **Trace** သည် အေးဂျင့်၏ တစ်ခုလုံးသော တာဝန်ကို အစမှ အဆုံးအထိ ကိုယ်စားပြုသည် (ဥပမာ- အသုံးပြုသူမေးခွန်းကို ကိုင်တွယ်ခြင်း)။
- **Spans** သည် trace အတွင်းရှိ တစ်ခုချင်းစီသော လုပ်ဆောင်မှုအဆင့်များကို ကိုယ်စားပြုသည် (ဥပမာ- ဘာသာစကားမော်ဒယ်ကို ခေါ်ယူခြင်း သို့မဟုတ် ဒေတာကို ရယူခြင်း)။

![Trace tree in Langfuse](https://langfuse.com/images/cookbook/example-autogen-evaluation/trace-tree.png)

စောင့်ကြည့်မှုမရှိပါက AI အေးဂျင့်သည် "အနက်ရောင်သေတ္တာ" ကဲ့သို့ ခံစားရနိုင်သည် - ၎င်း၏ အတွင်းအခြေအနေနှင့် အကြောင်းပြချက်များသည် မမြင်သာသောကြောင့် ပြဿနာများကို ရှာဖွေခြင်း သို့မဟုတ် စွမ်းဆောင်ရည်ကို အကောင်းဆုံးဖြစ်စေရန် ခက်ခဲနိုင်သည်။ စောင့်ကြည့်မှုရှိပါက အေးဂျင့်များသည် "ဖန်သားသေတ္တာ" ကဲ့သို့ ဖြစ်လာပြီး ယုံကြည်မှုတည်ဆောက်ရန်နှင့် ၎င်းတို့ရည်ရွယ်သည့်အတိုင်း လုပ်ဆောင်ရန် အရေးပါသော ပွင့်လင်းမြင်သာမှုကို ပေးစွမ်းသည်။

## ထုတ်လုပ်မှုပတ်ဝန်းကျင်တွင် စောင့်ကြည့်မှုအရေးပါမှု

AI အေးဂျင့်များကို ထုတ်လုပ်မှုပတ်ဝန်းကျင်သို့ ရွှေ့ပြောင်းခြင်းသည် စိန်ခေါ်မှုအသစ်များနှင့် လိုအပ်ချက်များကို ဖန်တီးပေးသည်။ စောင့်ကြည့်မှုသည် "လိုအပ်လို့ရှိရမည့်အရာ" မဟုတ်တော့ဘဲ အရေးပါသော စွမ်းရည်တစ်ခုဖြစ်လာသည်-

*   **ပြဿနာရှာဖွေခြင်းနှင့် အကြောင်းရင်းခွဲခြမ်းစိတ်ဖြာခြင်း**: အေးဂျင့်သည် မအောင်မြင်ခြင်း သို့မဟုတ် မမျှော်လင့်ထားသော အထွက်ကို ဖန်တီးသောအခါ၊ စောင့်ကြည့်မှုကိရိယာများသည် ပြဿနာရင်းမြစ်ကို ရှာဖွေရန် လိုအပ်သော trace များကို ပေးစွမ်းသည်။ ၎င်းသည် LLM ခေါ်ယူမှုများ၊ ကိရိယာများနှင့် အခြေအနေအခြေခံသော logic များပါဝင်နိုင်သော ရှုပ်ထွေးသော အေးဂျင့်များတွင် အထူးအရေးပါသည်။
*   **နောက်ကျမှုနှင့် ကုန်ကျစရိတ်စီမံခန့်ခွဲမှု**: AI အေးဂျင့်များသည် token သို့မဟုတ် call တစ်ခုချင်းစီအတွက် ကုန်ကျစရိတ်ကို တိုက်ရိုက်ပေးဆောင်ရသော LLM များနှင့် အခြား API များကို အားကိုးလေ့ရှိသည်။ စောင့်ကြည့်မှုသည် ၎င်းတို့၏ call များကို တိကျစွာ ခြေရာခံရန် ခွင့်ပြုသည်။ ၎င်းသည် prompt များကို အကောင်းဆုံးဖြစ်စေရန်၊ ပိုထိရောက်သော မော်ဒယ်များကို ရွေးချယ်ရန် သို့မဟုတ် workflow များကို ပြန်လည်ဒီဇိုင်းဆွဲရန် ကူညီသည်။
*   **ယုံကြည်မှု၊ လုံခြုံမှုနှင့် အညီအနေ**: အက်ပလီကေးရှင်းများစွာတွင် အေးဂျင့်များသည် လုံခြုံစွာနှင့် ကျင့်ဝတ်အတိုင်း လုပ်ဆောင်ရန် အရေးပါသည်။ စောင့်ကြည့်မှုသည် အေးဂျင့်၏ လုပ်ဆောင်မှုများနှင့် ဆုံးဖြတ်ချက်များ၏ စစ်ဆေးမှုလမ်းကြောင်းကို ပေးသည်။ ၎င်းကို prompt injection၊ အန္တရာယ်ရှိသော အကြောင်းအရာဖန်တီးခြင်း သို့မဟုတ် PII ကို မမှန်ကန်စွာ ကိုင်တွယ်ခြင်းကဲ့သို့သော ပြဿနာများကို ရှာဖွေပြီး ဖြေရှင်းရန် အသုံးပြုနိုင်သည်။
*   **ဆက်လက်တိုးတက်မှုလှိုင်းများ**: စောင့်ကြည့်မှုဒေတာသည် iterative development လုပ်ငန်းစဉ်၏ အခြေခံအုတ်မြစ်ဖြစ်သည်။ အေးဂျင့်များသည် အမှန်တကယ်ကမ္ဘာတွင် လုပ်ဆောင်ပုံကို စောင့်ကြည့်ခြင်းအားဖြင့် တိုးတက်မှုအပိုင်းများကို ရှာဖွေခြင်း၊ မော်ဒယ်များကို fine-tuning ပြုလုပ်ရန် ဒေတာစုဆောင်းခြင်းနှင့် ပြောင်းလဲမှုများ၏ သက်ရောက်မှုကို အတည်ပြုခြင်းတို့ကို ပြုလုပ်နိုင်သည်။

## စောင့်ကြည့်ရန် အဓိက Metrics

အေးဂျင့်၏ အပြုအမူကို စောင့်ကြည့်ရန်နှင့် နားလည်ရန်၊ metrics များနှင့် အချက်အလက်များကို စောင့်ကြည့်ရမည်။ အေးဂျင့်၏ ရည်ရွယ်ချက်ပေါ်မူတည်၍ metrics များကွဲပြားနိုင်သော်လည်း အချို့သည် အားလုံးအတွက် အရေးပါသည်။

အေးဂျင့်၏ ကျန်းမာရေးကို စောင့်ကြည့်ရန် metrics များ၏ ပေါင်းစပ်မှုသည် coverage အကောင်းဆုံးပေးသည်။ ဤအခန်း၏ [example notebook](../../../10-ai-agents-production/code_samples/10_autogen_evaluation.ipynb) တွင် ဤ metrics များကို အမှန်တကယ်သော ဥပမာများတွင် ပြသမည်ဖြစ်သော်လည်း အရင်ဆုံး typical evaluation workflow များကို လေ့လာမည်။

## သင့်အေးဂျင့်ကို Instrument လုပ်ခြင်း

trace ဒေတာကို စုဆောင်းရန်၊ သင့် code ကို instrument လုပ်ရန် လိုအပ်သည်။ ရည်ရွယ်ချက်မှာ အေးဂျင့် code ကို instrument လုပ်ပြီး trace နှင့် metrics များကို စောင့်ကြည့်မှု platform မှာ ဖမ်းယူ၊ အလုပ်လုပ်စေ၊ visualization ပြုလုပ်နိုင်ရန် ဖြစ်သည်။

**OpenTelemetry (OTel):** [OpenTelemetry](https://opentelemetry.io/) သည် LLM စောင့်ကြည့်မှုအတွက် စက်မှုလုပ်ငန်းစံအဖြစ် ပေါ်ထွက်လာသည်။ ၎င်းသည် telemetry ဒေတာကို ဖန်တီးခြင်း၊ စုဆောင်းခြင်းနှင့် export ပြုလုပ်ရန် API များ၊ SDK များနှင့် ကိရိယာများကို ပေးသည်။

## အေးဂျင့်အကဲဖြတ်ခြင်း

စောင့်ကြည့်မှုသည် metrics များကို ပေးသည်၊ သို့သော် အကဲဖြတ်ခြင်းသည် ၎င်းဒေတာကို ခွဲခြမ်းစိတ်ဖြာခြင်း (နှင့် စမ်းသပ်မှုများပြုလုပ်ခြင်း) ဖြစ်ပြီး AI အေးဂျင့်၏ စွမ်းဆောင်ရည်ကောင်းမွန်မှုကို သတ်မှတ်ရန်နှင့် တိုးတက်စေရန် ဖြစ်သည်။

AI အေးဂျင့်များသည် non-deterministic ဖြစ်ပြီး update များ သို့မဟုတ် မော်ဒယ်အပြောင်းအလဲများကြောင့် တိုးတက်နိုင်သည့်အတွက် အကဲဖြတ်မှုမရှိပါက "smart agent" သည် ၎င်း၏ တာဝန်ကို ကောင်းစွာလုပ်ဆောင်နေသည်ဟု သေချာမရနိုင်ပါ။

AI အေးဂျင့်များအတွက် အကဲဖြတ်မှုအမျိုးအစားနှစ်မျိုးရှိသည်- **online evaluation** နှင့် **offline evaluation**။ နှစ်မျိုးစလုံး အရေးပါပြီး ၎င်းတို့သည် အချင်းချင်းဖြည့်စွမ်းပေးသည်။

## ရင်ဆိုင်ရနိုင်သော ပြဿနာများ

AI အေးဂျင့်များကို ထုတ်လုပ်မှုတွင် အသုံးချသည့်အခါ၊ အမျိုးမျိုးသော စိန်ခေါ်မှုများနှင့် ရင်ဆိုင်ရနိုင်သည်။ အောက်တွင် အများဆုံးတွေ့ရသော ပြဿနာများနှင့် ၎င်းတို့၏ ဖြေရှင်းနည်းများကို ဖော်ပြထားသည်-

| **ပြဿနာ**    | **ဖြေရှင်းနည်း**   |
| ------------- | ------------------ |
| AI အေးဂျင့်သည် တာဝန်များကို တိကျစွာ မဆောင်ရွက်နိုင်ခြင်း | - AI အေးဂျင့်ကို ပေးထားသော prompt ကို ပြန်လည်တိုးတိရစေပါ။<br>- တာဝန်များကို subtask များအဖြစ် ခွဲခြားပြီး အေးဂျင့်များစွာဖြင့် ကိုင်တွယ်ပါ။ |
| AI အေးဂျင့်သည် အဆက်မပြတ် loop များတွင် ရောက်နေခြင်း  | - အေးဂျင့်သည် လုပ်ငန်းစဉ်ကို ရပ်တန့်ရန် သိရှိစေရန် termination terms များကို ရှင်းလင်းစွာ သတ်မှတ်ပါ။<br>- ရှုပ်ထွေးသော reasoning နှင့် planning လိုအပ်သော တာဝန်များအတွက် reasoning task များအတွက် အထူးပြုထားသော မော်ဒယ်ကို အသုံးပြုပါ။ |
| AI အေးဂျင့် tool call များကောင်းစွာ မလုပ်ဆောင်နိုင်ခြင်း   | - အေးဂျင့်စနစ်အပြင်တွင် tool output ကို စမ်းသပ်ပြီး အတည်ပြုပါ။ |

- သတ်မှတ်ထားသော parameters, prompts, နှင့် tools များ၏အမည်များကို ပြန်လည်တိုးတက်စေပါ။  
| Multi-Agent system သည် တိကျမှုမရှိစွာ လုပ်ဆောင်နေသည် | - တစ်ဦးချင်း agent များအား ပေးထားသော prompts များကို ပြန်လည်တိုးတက်စေပြီး တစ်ခုနှင့်တစ်ခု မတူညီစေရန် သေချာစေပါ။<br>- "routing" သို့မဟုတ် controller agent ကို အသုံးပြု၍ hierarchical system တစ်ခု တည်ဆောက်ပြီး မည်သည့် agent သည် မှန်ကန်သည်ကို ဆုံးဖြတ်ပါ။ |

ဤပြဿနာများအများစုကို observability ရှိနေပါက ပိုမိုထိရောက်စွာ ရှာဖွေနိုင်ပါသည်။ ယခင်တွင် ဆွေးနွေးခဲ့သော traces နှင့် metrics များက agent workflow ၏ မည်သည့်နေရာတွင် ပြဿနာများ ဖြစ်ပေါ်နေသည်ကို တိကျစွာ ဖော်ထုတ်ပေးပြီး debugging နှင့် optimization ကို ပိုမိုထိရောက်စေပါသည်။

## ကုန်ကျစရိတ်များကို စီမံခန့်ခွဲခြင်း

AI agents များကို production တွင် အသုံးပြုရာတွင် ကုန်ကျစရိတ်များကို စီမံခန့်ခွဲရန် အောက်ပါနည်းလမ်းများကို အသုံးပြုနိုင်ပါသည်-

**အသေးစား Models အသုံးပြုခြင်း:** Small Language Models (SLMs) သည် အချို့သော agentic use-cases များတွင် ကောင်းစွာ လုပ်ဆောင်နိုင်ပြီး ကုန်ကျစရိတ်များကို အလွန်လျှော့ချနိုင်ပါသည်။ ယခင်တွင် ပြောခဲ့သည့်အတိုင်း၊ performance ကို ကြည့်ရှု၍ SLM များ၏ လုပ်ဆောင်နိုင်စွမ်းကို နားလည်ရန်အတွက် အကဲဖြတ်စနစ်တစ်ခု တည်ဆောက်ခြင်းသည် အကောင်းဆုံးနည်းလမ်းဖြစ်ပါသည်။ SLM များကို intent classification သို့မဟုတ် parameter extraction ကဲ့သို့သော ရိုးရှင်းသော အလုပ်များအတွက် အသုံးပြုရန် စဉ်းစားပါ၊ အကြံဉာဏ်လိုအပ်သော အလုပ်များအတွက်သာ ကြီးမားသော models များကို အသုံးပြုပါ။

**Router Model အသုံးပြုခြင်း:** ထိုနည်းတူပင်၊ မတူညီသော models နှင့် အရွယ်အစားများကို အသုံးပြုခြင်းသည်လည်း ကောင်းမွန်သော နည်းလမ်းတစ်ခုဖြစ်ပါသည်။ LLM/SLM သို့မဟုတ် serverless function တစ်ခုကို အသုံးပြု၍ တောင်းဆိုမှုများကို အလွယ်အကူဆုံး model များသို့ လမ်းညွှန်ပေးပါ။ ဤနည်းလမ်းသည် ကုန်ကျစရိတ်များကို လျှော့ချနိုင်သလို၊ တိကျသော အလုပ်များတွင် စွမ်းဆောင်ရည်ကိုလည်း သေချာစေပါသည်။ ဥပမာအားဖြင့်၊ ရိုးရှင်းသော မေးခွန်းများကို အသေးစား၊ မြန်ဆန်သော models များသို့ လမ်းညွှန်ပေးပြီး၊ အကြံဉာဏ်လိုအပ်သော အလုပ်များအတွက်သာ ကြီးမားသော models များကို အသုံးပြုပါ။

**Caching Responses:** ရိုးရှင်းသော တောင်းဆိုမှုများနှင့် အလုပ်များကို ရှာဖွေပြီး၊ agentic system မှတဆင့် မသွားမီ အဖြေများကို ပေးခြင်းသည် တူညီသော တောင်းဆိုမှုများ၏ ပမာဏကို လျှော့ချရန် ကောင်းမွန်သော နည်းလမ်းတစ်ခုဖြစ်ပါသည်။ ထို့အပြင်၊ တောင်းဆိုမှုတစ်ခုသည် cached requests များနှင့် ဘယ်လောက်တူညီသည်ကို ရှာဖွေရန် အခြေခံ AI models များကို အသုံးပြု၍ flow တစ်ခုကိုလည်း တည်ဆောက်နိုင်ပါသည်။ ဤနည်းလမ်းသည် မကြာခဏ မေးလေ့ရှိသော မေးခွန်းများ သို့မဟုတ် ရိုးရှင်းသော workflow များအတွက် ကုန်ကျစရိတ်များကို အလွန်လျှော့ချနိုင်ပါသည်။
