# Примерни фикстури на разписки

Три предварително създадени файлове с разписки за преглед без стартиране на бележника.

| Файл | Какво представлява |
|---|---|
| `01_valid_receipt.json` | Валидна подписана разписка за извикване на инструмента `lookup_flights`. Проверката връща True. |
| `02_tampered_receipt.json` | Същата разписка с едно поле променено след подписването. Проверката връща False. |
| `03_chain_three_receipts.json` | Вериги от три валидни разписки (търсене, задържане, резервация) с `previous_receipt_hash`, свързващ всяка с предходната. |

## Проверка на примерите

Бележникът преминава през проверката в четири секции. За да проверите тези фикстури
директно, без да следвате разказа в бележника:

```python
import json
from pathlib import Path

# Предполага се, че сте завършили импортирането и помощните функции
# от секциите 1 и 2 на 18-signed-receipts.ipynb.

valid = json.loads(Path("01_valid_receipt.json").read_text())
print(f"Valid receipt: {verify_receipt(valid)}")        # Вярно

tampered = json.loads(Path("02_tampered_receipt.json").read_text())
print(f"Tampered receipt: {verify_receipt(tampered)}")  # Невярно

chain = json.loads(Path("03_chain_three_receipts.json").read_text())
for r in verify_chain(chain):
    print(f"  Receipt {r['index']} ({r['tool']}): {'VALID' if r['overall_valid'] else 'INVALID'}")
```

## Как са създадени

Фикстурите използват същия кодов път като бележника, с един фиксиран подписващ ключ
и фиксирани времеви печати за възпроизводимост на байтово ниво. За да ги създадете наново:

```bash
python3 generate_fixtures.py
```

(Скриптът се намира в `generate_fixtures.py` в тази директория.)

## Какво научават студентите при разглеждане на суровия JSON

Четенето на суровия формат на разписката изгражда интуиция, която клетките в бележника
не винаги предоставят. Студентите, които преглеждат JSON-а, често забелязват:

1. Подписът е непрозрачен низ в base64url, но всяко друго поле е обикновен
   четим JSON. Подписът не криптира съдържанието; той го удостоверява.
2. `public_key` е вграден в разписката. Одиторът не се нуждае от нищо друго
   за проверка (с изключение на това да се вярва, че ключът наистина принадлежи на заявения
   издател; вижте урока README за инфраструктурата на идентичността).
3. Промяна на един символ в който и да е поле и след това сравнение на този файл с
   `02_tampered_receipt.json` прави байтово-нивото механизъм конкретен.
