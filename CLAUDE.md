# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Setup

```bash
python -m venv venv && source venv/bin/activate
pip install -r requirements.txt
cp .env.example .env   # then fill in values
az login               # notebooks authenticate via AzureCliCredential — no API keys needed
```

## Running Code

```bash
# Start Jupyter for interactive notebooks
jupyter notebook

# Convert and run a notebook as a script (smoke-test imports/flow)
jupyter nbconvert --to script <lesson-folder>/code_samples/<notebook>.ipynb --stdout | python

# Run the hotel booking workflow sample directly
python 14-microsoft-agent-framework/code-samples/hotel_booking_workflow_sample.py

# Run the MCP resumable server
python -m 11-agentic-protocols.code_samples.mcp-agents.server.server --port 8006

# Run the Chainlit-based multi-agent app (Lesson 11)
chainlit run 11-agentic-protocols/code_samples/github-mcp/app.py

# Run tests (Lesson 04 plugin tests only)
cd 04-tool-use/code_samples && python test_demo_plugins.py
```

## Architecture

### Two Provider Patterns

Most notebooks use one of two client patterns that are not interchangeable:

1. **`AzureAIProjectAgentProvider`** (primary) — connects to Azure AI Foundry Agent Service V2. Agents are registered server-side and visible in the Foundry portal. Requires `az login` + `AZURE_AI_PROJECT_ENDPOINT`.

2. **`OpenAIChatClient`** (secondary) — works with any OpenAI-compatible endpoint: OpenAI, GitHub Models (`https://models.inference.ai.azure.com`), or MiniMax. Provider is auto-selected at runtime based on which env var is set (`MINIMAX_API_KEY` → `GITHUB_TOKEN` → default OpenAI).

### Agent Construction Flow

Agents in this repo are built in three layers:

```
Instructions (system prompt string)
  → provider.create_agent() / chat_client.create_agent()
      → AgentExecutor (wraps agent, gives it an id)
          → WorkflowBuilder (wires executors into a graph)
```

`WorkflowBuilder` supports **sequential** (`.add_edge(a, b)`), **conditional** (`.add_edge(a, b, condition=fn)`), concurrent, handoff, and group-chat patterns. Conditions receive the upstream `AgentExecutorResponse` and return a bool.

### Tool Registration

- `@tool` decorator (MAF) — for tools passed to `workflow.run(..., tools=[fn])`
- `@ai_function` decorator (MAF) — for tools passed directly into `chat_client.create_agent(tools=[fn])`
- Both use `Annotated[type, "description"]` for parameter docs; no separate schema needed.
- Structured outputs use Pydantic models passed as `response_format=MyModel` to `create_agent`.

### MCP Protocol Layer (Lesson 11)

The resumable MCP server (`11-agentic-protocols/code_samples/mcp-agents/`) runs as a Starlette/uvicorn ASGI app on port 8006. Key design:

- `ResumableServer(Server)` — registers tools via `@self.list_tools()` / `@self.call_tool()` decorators inside `__init__`
- `SimpleEventStore` — persists SSE events for session resumption after client disconnect
- `StreamableHTTPSessionManager` — bridges MCP server to ASGI, mounts at `/mcp`
- Tools can call `ctx.session.elicit()` (human-in-the-loop confirmation) and `ctx.session.create_message()` (LLM sampling mid-tool) as async calls

The Chainlit app (`github-mcp/app.py`) connects to MCP tools via `@cl.on_mcp_connect`, builds a sequential 3-agent workflow (GitHub → Hackathon → Events), and routes single vs. multi-agent calls via regex on the user message.

### Lesson Structure

Each lesson directory follows: `README.md` + `code_samples/*-python-agent-framework.ipynb`. Lesson 14 is the most advanced — it contains standalone `.py` files demonstrating all MAF workflow patterns (sequential, concurrent, conditional, handoff, human-loop, middleware).

### Translation System

50+ language translations live under `translations/<lang-code>/`. These are auto-generated by the `co-op-translator` GitHub Action and should not be hand-edited. Source of truth is always the English files at the repo root.
